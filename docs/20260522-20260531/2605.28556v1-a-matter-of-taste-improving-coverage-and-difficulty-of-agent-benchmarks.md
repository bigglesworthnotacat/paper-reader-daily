## 摘要
随着智能体能力的提升，现有的基准测试（如$τ^2$-Bench）正逐渐饱和。然而，构建新的基准测试任务仍然复杂、昂贵且劳动密集。此外，标准方法（先以自然语言编写场景，再映射到工具序列）仅能捕捉智能体所使用的工具模式的狭窄子集。在本文中，我们通过逆转任务构建过程来解决这些问题。我们提出TASTE：基于工具序列演化的任务合成，这是一种自动方法，能够生成具有更广工具使用覆盖度的挑战性任务。TASTE利用在LLM判断的有效性信号上训练的自适应对比$n$-gram模型，从而能够采样覆盖大量工具组合的有效工具序列。然后，TASTE通过聚类从池中选择代表性序列，将其实例化为完整的基准测试任务，并通过迭代难度演化进行优化。使用TASTE，我们构建了$τ^c$-Bench，这是$τ^2$-Bench三个领域的具有挑战性的扩展。我们评估了11个智能体/用户LLM对，发现几乎在$τ^2$-Bench上饱和的模型在我们的任务上性能大幅下降（例如，Gemini-3-Flash从$0.82\!-\!0.94$降至$0.28\!-\!0.61$）。除了增加难度外，我们生成的任务使智能体必须执行的独特工具组合数量翻了一番以上。我们的结果表明，现有基准测试的高分往往反映的是饱和而非稳健的问题解决能力。通过自动化生成高难度、高覆盖度的基准测试，TASTE使得对未来智能体的持续、可扩展评估成为可能。

## Abstract
As agent capabilities advance, existing benchmarks, such as $τ^2$-Bench, are becoming increasingly saturated. Yet constructing new benchmark tasks remains complex, costly, and labor-intensive. Moreover, the standard approach, in which scenarios are first written in natural language and then mapped to tool sequences, captures only a narrow subset of the tool-use patterns agents exercise. In this paper, we address these problems by reversing the task construction process. We propose TASTE: Task Synthesis from Tool Sequence Evolution, an automatic method that generates challenging tasks with broader tool-use coverage. TASTE utilizes an Adaptive Contrastive $n$-gram model trained on LLM-judged validity signals. This enables sampling valid tool sequences that cover a vast range of tool combinations. TASTE then selects representative sequences from the pool via clustering, instantiates them into complete benchmark tasks, and refines them through iterative difficulty evolution. Using TASTE, we construct $τ^c$-Bench, a challenging extension of the three domains of $τ^2$-Bench. We evaluate $11$ agent/user LLM pairs and find that models nearly saturating $τ^2$-Bench suffer severe performance drops on our tasks (e.g., Gemini-3-Flash falls from $0.82\!-\!0.94$ to $0.28\!-\!0.61$). Beyond increasing difficulty, our generated tasks more than double the number of unique tool combinations agents must execute. Our results suggest high scores on existing benchmarks often reflect saturation rather than robust task-solving ability. By automating the generation of difficult, high-coverage benchmarks, TASTE enables continuous, scalable evaluation of future agents.

- **本文目标**：设计一种**自动化、可扩展**的基准任务生成方法，同时满足 **有效性（Validity）、难度（Difficulty）和覆盖度（Coverage）** 三个期望性质，从而支持对不断进步的智能体进行持续、有区分度的评估。

## 2. 方法论：核心思想与关键技术

- **核心思路**：**反转传统任务构建流程**——不从场景出发，而是直接从**工具序列空间**采样、聚类，再围绕序列合成完整任务（用户指令、数据库状态、工具参数）。
- **技术细节（三阶段流水线 TASTE）**：
  1. **阶段一：工具序列采样（覆盖度 + 有效性）**
     - 训练一个**自适应对比 n‑gram 模型**（Adaptive Contrastive n‑gram Model）。
     - 维护两个计数表 C⁺（有效序列）和 C⁻（无效序列），通过**对比比率**计算下一个工具的条件概率。
     - 模型通过“生成‑LLM 判断”循环在线更新，并引入温度退火和负样本权重。
  2. **阶段二：聚类与选择（覆盖度）**
     - 从训练后的模型采样 N ≫ K 条序列（K 为目标任务数）。
     - 用**加权 Levenshtein 距离**（依据工具语义/功能相似性设置替代成本）进行 K‑medoids 聚类，选每个簇的 medoid 为代表。
     - 对 medoid 做有效性验证，失效时从簇内替换或重新聚类。
  3. **阶段三：任务生成与难度演化（有效性 + 难度）**
     - 为每个 medoid 序列生成**基础任务**（用户场景 + 数据库 + 完整的工具调用参数），并通过**规则检查 + 提示辅助验证器智能体（Verifier Agent）** 确保可解性。
     - **难度演化**：通过三个子步骤——**策略分析**（识别每个写操作可被用户干扰的点）、**环境扰动**（添加诱饵数据库记录）、**场景重写**（使指令模糊、用户不合作）——生成更具挑战性的变体，并重新验证。
- **公式要点**：条件概率 \(P(t_i \mid \text{ctx}_i) \propto \frac{S^+(t_i \mid \text{ctx}_i)}{[S^-(t_i \mid \text{ctx}_i)]^{\lambda_{\text{neg}}}}\)，其中 S⁺、S⁻ 是经 Dirichlet 平滑后的 n‑gram 分数。

## 6. 主要结论与发现

1. **τᶜ‑Bench 显著更难**：几乎所有模型在 τᶜ‑Bench 上性能出现大幅下降（**相对降幅 5%–80%**），尤其对在 τᵇⱽ 上几乎饱和的模型（如 Gemini-3-Flash 从 0.82–0.94 降至 0.28–0.61）。

3. **高基准分数≠稳健能力**：在 τᵇⱽ 上的饱和高分掩盖了智能体在更真实、更复杂场景下的缺陷，τᶜ‑Bench 能更好区分 SOTA 模型。
4. **TASTE 流水线有效**：各组件消融证实了自适应在线训练、对比负证据、加权编辑距离聚类、难度演化等关键设计均有显著贡献。

## 7. 优点

- **方法创新**：首次从工具序列空间出发自动化生成基准任务，直接优化覆盖度和难度，而非依赖人工构思场景。
- **明确且可操作的定义**：将覆盖度操作化为工具序列的多样性（n‑gram TTR、编辑距离等），使评估可量化。
- **自动化与可扩展**：整个流程无人干预，成本可控（仅为 API 调用费用），可快速适配新领域。
- **难度可控**：通过调节序列长度、读写比例、演化强度等可以生成不同难度等级的任务，支持持续更新。
- **验证体系严密**：规则检查 + 提示辅助验证器 + 多级退化回退，确保生成任务几乎全有效（精度近 1.0）。

## 8. 不足与局限

- **场景限制**：仅针对**对话式工具使用智能体**，未验证非对话或单轮场景。作者声称可迁移但未实验。
- **覆盖度定义局限**：仅基于工具序列（名称）的多样性，忽略了参数分布、交互复杂度等其他维度。
- **n‑gram 模型假设**：其局部依赖假设在长程约束场景（如多轮状态依赖）下可能不足，虽实验有效但理论上有限。
- **验证器召回率不高**：精度接近 1.0 但召回仅 0.75–0.83，可能拒绝部分有效任务，导致生成效率降低。
- **演化可能引入模式固化**：对抗性用户行为模式来自人工编目，可能使智能体学到针对特定技巧的应对而非通用能力。
- **计算成本随领域扩展**：虽然单次成本低，但若要持续生成大量领域任务，总成本可能显著增加。
- **与现有基准关系**：目前只基于 τ²-Bench 扩展，未探索完全新的领域或工具集，通用性需进一步验证。
