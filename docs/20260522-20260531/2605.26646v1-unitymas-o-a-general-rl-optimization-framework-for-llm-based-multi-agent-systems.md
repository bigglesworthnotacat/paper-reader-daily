## 摘要
基于大语言模型的多智能体系统将复杂任务分解为交互角色，但大多数系统仍通过提示、工具和控制规则进行手动编排，且智能体很少通过统一的强化学习接口进行优化。现有的强化学习后训练框架主要针对单策略优化，缺乏对用户定义的多智能体工作流、结构化交互、角色特定信用分配和可配置参数共享的抽象。本文提出UnityMAS-O，一个用于基于大语言模型的多智能体系统的通用强化学习优化框架。UnityMAS-O将完整工作流作为优化单元，而非单个响应或策略轨迹。它通过四个第一类对象表示工作流：逻辑智能体角色、图轨迹、用户定义奖励和智能体-模型映射。这解耦了逻辑智能体与物理模型参数，支持完全共享、完全分离和部分共享，奖励在角色、轮次和轨迹级别分配。UnityMAS-O在verl基础上扩展了基于Ray的星形拓扑运行时。中央控制器执行工作流、调用工具、记录结构化轨迹并组装奖励；模型本地工作器组负责数据收集、缓冲、优势计算和分布式PPO风格更新。用户可以定义智能体、工作流、模型映射和奖励，而无需重写优化基础设施。我们在检索增强问答、迭代智能搜索和反思性代码生成上实例化了UnityMAS-O。在Natural Questions、HotpotQA和保留代码任务上，多智能体强化学习在优化后改进了手动指定的工作流，尤其在小模型和严格的代码全通过指标上取得了较大提升。这些结果表明，UnityMAS-O可作为可复用基础，将多样化的基于大语言模型的多智能体工作流转化为可训练的多智能体强化学习系统。

## Abstract
LLM-based multi-agent systems decompose complex tasks into interacting roles, but most remain manually orchestrated by prompts, tools, and control rules, while agents are rarely optimized through a unified reinforcement learning interface. Existing RL post-training frameworks mainly target single-policy optimization and lack abstractions for user-defined multi-agent workflows, structured interaction, role-specific credit assignment, and configurable parameter sharing.   We present UnityMAS-O, a general RL optimization framework for LLM-based multi-agent systems. UnityMAS-O treats the complete workflow as the optimization unit, rather than a single response or policy trajectory. It represents workflows through four first-class objects: logical agent roles, graph trajectories, user-defined rewards, and agent--model mappings. This decouples logical agents from physical model parameters, supporting full sharing, full separation, and partial sharing, with rewards assigned at role, turn, and trajectory levels.   UnityMAS-O extends verl with a Ray-based star-topology runtime. A central controller executes workflows, invokes tools, records structured trajectories, and assembles rewards; model-local worker groups handle rollout, buffering, advantage computation, and distributed PPO-style updates. Users can define agents, workflows, model mappings, and rewards without rewriting the optimization infrastructure.   We instantiate UnityMAS-O on retrieval-augmented QA, iterative agentic search, and reflective code generation. Across Natural Questions, HotpotQA, and held-out code tasks, multi-agent RL improves manually specified workflows after optimization, with especially large gains for smaller models and strict code all-passed metrics. These results show that UnityMAS-O can serve as a reusable substrate for converting diverse LLM-based multi-agent workflows into trainable multi-agent RL systems.

## 2. 方法论
### 核心思想
- 以**完整的多智能体工作流**（而非单个响应或策略）为优化单位。工作流由有向图 \( G=(V,E) \) 定义，节点表示逻辑智能体角色，边表示信息/控制流。  
- 引入 **智能体-模型映射** \( \phi: V \to M \)：将逻辑角色显式映射到物理 LLM 实例，支持三种参数共享模式：  
  - 全共享：所有角色共用同一模型。  
  - 全分离：每个角色对应独立模型。  
  - 部分共享：角色分组内共享、组间独立。  
- 提供 **三级奖励接口**：节点级（局部）+ 轮次级（中间上下文）+ 轨迹级（最终结果），支持延迟奖励和边际改进奖励。  
- 基于 verl（Sheng et al., 2024）和 Ray 构建**星形拓扑分布式运行时**：中心控制器负责工作流执行、工具调用、轨迹记录和奖励组装；模型本地工作器组负责生成、缓冲、优势计算和 PPO 更新。

### 关键技术细节
- **工作流调度**：中心控制器异步执行多个任务实例的图，根据当前状态和活跃角色，通过映射 \( \phi \) 将生成请求路由到对应工作器组。工作器组返回轻量级动作（如生成文本），控制器更新状态并调度下一角色。  
- **模型本地缓冲**：生成时产生的张量（log 概率、值函数等）保留在工作器组本地，控制器只传递角色 ID、路由信息和奖励元数据，减少通信开销。  
- **PPO 风格训练循环**：  
  - 异步 rollout → 奖励组装与提交 → 构建模型特定批次 → 优势计算（GAE） → 每个工作器组独立更新参数 → 同步权重到 rollout 后端（vLLM）。  
- **信用分配**：例如在代码工作流中，初始规划-编码对获得原始验证分数；后续反思和再编码只获得其带来的分数增量，鼓励逐步改进。

### 算法流程（文字说明）
1. 用户定义逻辑角色、工作流图、智能体-模型映射和每角色的奖励函数。  
2. 中心控制器在任务实例上执行工作流：  
   - 对于每个活跃角色，将状态+角色 ID 发送给 \( \phi(v) \) 对应的工作器组。  
   - 工作器组生成输出，返回控制器，控制器记录结构化轨迹并应用环境/工具转换。  
3. 轨迹完成后，根据奖励函数为每个角色调用计算奖励（节点级/轮次级/轨迹级）。  
4. 每个工作器组将自身负责的角色轨迹片段（含奖励）存入本地缓冲区，构建训练批次。  
5. 工作器组计算优势（GAE），使用 PPO 进行策略更新。  
6. 更新后的权重同步至推理引擎，开始下一迭代。

## 6. 主要结论与发现
- **MARL 训练显著提升性能**：在所有 QA 和代码任务上，训练后的最佳 F1 或全通过率均超过未训练版本，尤其对小模型（0.5B、1.5B）改善巨大（如 NQ 上 0.5B 的 QD-Answer F1 从 0.022 升至 0.445）。

- **跨工作流一致性**：不同类型工作流均可通过 UnityMAS-O 统一优化，验证了框架的任务无关性。

## 7. 优点
- **统一抽象**：将用户定义的工作流图、角色、奖励、模型映射提升为框架级第一类对象，避免了每次重新实现训练管道。  
- **灵活的信用分配**：支持节点级、轮次级、轨迹级奖励组合，可表达增量改进（如迭代搜索中的边际 F1），更适合多智能体信用归属。  
- **参数共享普适性**：通过单一的映射 \( \phi \) 即可切换全共享/部分共享/全分离，便于研究资源-性能折中。  
- **系统设计干净**：控制器负责轻量级元数据，模型本地缓冲区保留重张量，减少通信；基于 Ray 和 verl 构建，可利用现有分布式基础设施。  
- **实验范围全面**：覆盖多种模型大小（0.5B-14B）、两种任务领域、多种工作流拓扑，结果具有普遍性。
