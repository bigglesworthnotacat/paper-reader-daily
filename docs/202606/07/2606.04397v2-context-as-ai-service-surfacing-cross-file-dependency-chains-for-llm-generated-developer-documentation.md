## 摘要
大型语言模型（LLM）智能体日益参与编写和维护开发者文档，但其有用性与准确性常依赖于难以直观跟踪的依赖链。即便上下文中包含更多文件，智能体仍需决定追溯哪些跨文件依赖。我们提出Context-as-AI-Service（CAIS），这是一个检索层，LLM智能体可在审查或生成文档时查询该层，以在代码库中寻找证据。CAIS对源代码、API参考和上游文档建立索引，然后使智能体能够通过结合关键词与语义搜索的工具调用来查询该索引。我们使用Claude Sonnet 4.6在生产级SDK上进行两项案例研究来评估CAIS：改进核心源文件中的API参考注释，以及验证一份由LLM生成的教程。在这两项研究中，基线条件已配备常规仓库工具，如文件读取、关键词搜索和符号导航。CAIS在此基础上添加了检索层，因此对比隔离了新增的检索能力，而非基本的仓库访问能力。在API参考审查中，增强CAIS的智能体产生了与基线相同的5处缺失文档修复，并额外发现了基线遗漏的4项发现：2处跨文件事实性错误和2处说明不充分的API注释。在教程验证中，它发现了基线流程未捕获的1个可执行错误、1个API使用改进点和2个缺失的先决条件。这些发现要求追踪跨越工具文件、框架内部实现、使用示例、测试以及组件创建逻辑的非直观依赖链。在每种条件下运行五次，添加CAIS使两项任务的墙钟时间减少了22%至34%，并降低了输入令牌使用量。

## Abstract
LLM agents increasingly write and maintain developer documentation, but usefulness and accuracy often rely on dependency chains that are not obvious to follow. Even with more files in context, the agent must still decide which cross-file dependencies to trace. We present Context-as-AI-Service (CAIS), a retrieval layer that LLM agents query to find evidence across the codebase as they review or generate documentation. CAIS indexes source code, API references, and upstream documentation, then enables agents to query the index through tool calls that combine keyword and semantic search. We evaluate CAIS in two case studies using Claude Sonnet 4.6 on a production SDK: improving API reference comments in a core source file and validating an LLM-generated tutorial. In both studies, the baseline already had ordinary repository tools such as file reads, keyword search, and symbol navigation. CAIS adds a retrieval layer on top, so the comparison isolates added retrieval rather than basic repository access. In the API-reference review, the CAIS-augmented agent produced the same 5 missing-documentation fixes as the baseline and surfaced 4 findings the baseline missed: 2 cross-file factual errors and 2 underspecified API comments. In the tutorial validation, it surfaced 1 executable bug, 1 API-usage improvement, and 2 missing prerequisites that the baseline pipeline did not catch. These findings required tracing non-obvious dependency chains across utility files, framework internals, usage examples, tests, and component-creation logic. Over five runs per condition, adding CAIS reduced wall-clock time by 22% to 34% across the two tasks and lowered input-token usage.

## 1. 论文的核心问题与整体含义

- **研究动机**：LLM 智能体越来越多地被用于自动生成和维护开发者文档（如 API 注释、教程），但文档的准确性与完整性常取决于跨越多个文件的非直观依赖链。例如，一个方法的描述可能在当前文件中看似合理，但其真实行为却由其他文件中的状态管理层、注册组件或框架生命周期规则决定。
- **问题本质**：作者称之为“跨文件文档问题”——文档在单一文件中可能语法流畅且看似正确，但当其依赖的符号、实现、测试或框架规则分布在代码库的其他位置时，错误便容易隐藏。即使智能体能访问大量文件，也难以自行判断该追踪哪些依赖链。
- **整体含义**：本文提出一种可组合的检索层 **Context-as-AI-Service (CAIS)**，其目标不是替代智能体的文件审查与推理，而是为智能体提供一种可主动查询的代码库范围证据检索手段，从而在生成或审查文档时更容易发现非明显的跨文件依赖，提升文档可靠性。

## 2. 论文提出的方法论

- **核心思想**：在常规仓库工具（文件读取、关键词搜索、符号导航）之上，叠加一个预索引的代码库检索服务，使 LLM 智能体可以在文档工作流中按需调用，以获取与当前文档声明相关的跨文件证据。
- **系统架构（四阶段流水线）**：
  - **数据摄入（Source Ingestion）**：收集三类来源——源代码（实现、工具、测试）、API 参考（从注释和参考页面提取的结构化文档）、上游文档（第三方平台或引擎文档）。每个文档被标准化为带有源类型、文件路径和时间戳的记录。
  - **存储与索引（Storage and Indexing）**：保留原始格式与富化格式；使用 BM25 进行词汇索引，使用 DRAMA 模型生成稠密嵌入，支持按需重索引。
  - **检索接口（Retrieval Interface）**：为 LLM 智能体暴露可工具调用的查询接口。查询同时执行 BM25（精确词汇匹配）和 DRAMA 密集检索（语义匹配），采用倒数秩融合（reciprocal rank fusion）合并结果，返回带文档 ID、分数和片段的排序列表。
  - **审查层（Review Layer）**：检索结果可作为证据直接由智能体使用，或通过轻量审查提示（标记为“一致”、“矛盾”或“不完整”）辅助判断。最终仅当证据支持对文档或教程的具体修正时才保留为有效发现。
- **关键技术细节**：
  - 混合检索：BM25 处理准确的方法名、枚举值；DRAMA 捕捉语义相关的描述、行为解释和示例。
  - 查询由智能体自行发起（如 `query(“deferred removal semantics”, corpus=“codebase”, top_k=5)`），并非固定脚本。
  - CAIS 与智能体解耦（agent-agnostic），任何可调用工具的 LLM 智能体均可使用。

## 7. 优点

- **问题定义清晰**：首次明确刻画了“跨文件文档问题”，将其与普通的本地文档缺陷区分开，并围绕此构建解决方案。
- **增量式设计**：CAIS 作为检索层叠加在现有仓库工具之上，不替代智能体的决策流程，实现模块化组合，易于集成到现有 LLM 工作流。
- **检索策略合理**：混合 BM25 与 DRAMA 的稠密检索，既保证精确符号匹配，又捕捉语义相关性；借助倒数秩融合统一排序，覆盖面更广。
- **实验对比严谨**：通过完全相同的智能体和任务，仅改变是否添加 CAIS，较好隔离了检索层的贡献，排除了仓库基本访问能力差异的干扰。
- **证据链透明**：对每一项额外发现提供了清晰的追溯路径（证据类型、基线遗漏原因、影响），增强了案例研究的可解释性和可信度。
- **效率实证**：不仅关注效果，还测量了时间和令牌成本，证明在提升质量的同时不会增加系统开销，反而更高效。

## 8. 不足与局限

- **实验范围有限**：基于单个生产级 SDK 和两种工作流，无法保证结论在不同语言、代码库规模、文档类型或智能体设计下的泛化性。作者亦将此列为主要限制。
- **无法公开复现**：采用专有 SDK 与日志，不提供原始语料，外部无法独立复现结果。
- **基线单一**：仅对比了 Claude Sonnet 4.6 配合其原生仓库工具这一种基线配置，未将 CAIS 与其他先进检索策略或大上下文模型（如显式指示进行跨文件依赖跟踪的提示策略）相比较。
- **评估未涉及人类研究**：文档质量改进仅通过自动发现的可追溯缺陷来衡量，缺少受控的人类开发者研究来评估实际开发体验或生产率的影响。
- **检索粒度较粗**：当前主要按文档或文件级别返回片段，作者指出未来可引入符号级检索，以减少上下文窗口的浪费，提高证据定位精度。
- **效率和结果可能波动**：虽然每种条件运行了 5 次并报告了均值和标准差，但样本量仍然较小，且所发现的额外问题高度依赖具体查询和智能体策略，可能在不同运行间存在一定随机性。
