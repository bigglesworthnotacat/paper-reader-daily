## 摘要
人工智能驱动的智能体正越来越多地融入持续集成与持续交付/部署（CI/CD）流水线，以自主审查拉取请求（PR）、分类问题并维护代码库。这些智能体在拥有较高仓库权限的同时摄入不受信的内容，使其自然成为具有供应链后果的提示注入攻击目标。我们提出GitInject，一个用于评估真实、活跃的GitHub工作流（一种广泛部署的CI/CD流水线实例）中提示注入漏洞的开源框架。与以往模拟工具调用的智能体安全基准不同，GitInject会配置临时仓库并触发实际工作流运行，以便沙箱约束、凭证处理及权限边界完全按照生产环境进行。我们利用GitInject研究四种AI服务提供商的工作流配置，并记录了十一种命名攻击，涵盖配置文件注入、凭证窃取、判断操纵和可用性攻击。我们发现，所有受测提供商在其默认配置下都至少易受一种攻击类别的影响，且最关键的漏洞是结构性的：它们源于CI/CD基础设施处理凭证和配置文件的方式，而非特定模型的行为。针对每一种确认的攻击类别，我们确定了最小成本的工作流级应对措施，并分析了其覆盖范围与局限性。GitInject已公开发布，以推动该方向的进一步研究。

## Abstract
AI-powered agents are increasingly embedded in continuous integration and continuous delivery/deployment (CI/CD) pipelines to autonomously review pull requests (PRs), triage issues, and maintain codebases. These agents ingest untrusted content while operating with elevated repository permissions, making them a natural target for prompt injection attacks with supply chain consequences. We present GitInject, an open-source framework for evaluating prompt injection vulnerabilities in real, live GitHub workflows, a widely deployed instance of CI/CD pipelines. Unlike prior agent security benchmarks that simulate tool calls, GitInject provisions ephemeral repositories and triggers actual workflow runs, so that sandbox constraints, credential handling, and permission boundaries behave exactly as in production. Using GitInject, we study workflow configurations across four AI providers and document eleven named attacks spanning config-file injection, credential exfiltration, judgment manipulation, and availability. We find that all tested providers are susceptible to at least one attack class in their default configuration, and that the most critical vulnerabilities are structural: they arise from how CI/CD infrastructure handles credentials and configuration files, not from any specific model's behavior. For each confirmed attack class, we identify the minimum-cost workflow-level countermeasure and analyze its coverage and limitations. GitInject is released publicly to facilitate further research in this direction.

### 1. 论文的核心问题与整体含义

- **研究动机**：随着AI智能体（如Anthropic Claude、OpenAI Codex、Google Gemini、Cline等）被集成到CI/CD流水线中（如GitHub Actions），它们能够自动审查PR、分类问题、修复测试等，但它们同时拥有高权限（如`GITHUB_TOKEN`）且必须摄入不可信内容（PR描述、问题正文、代码等），这构成了“致命三要素”——私密数据访问、不受信内容、外部通信能力，极易遭受提示注入攻击。
- **核心问题**：这类攻击不仅可窃取凭据、审批恶意代码，还可能影响下游供应链安全。然而，现有AI安全基准（如AgentDojo）通过模拟工具调用来评估安全性，忽略了CI/CD基础设施的真实语义（如沙箱约束、凭据存储、网络策略），导致评估结果与生产环境严重不一致。因此，**真实CI/CD工作流中的提示注入漏洞缺乏系统化、基于真实执行的研究与评估框架**。
- **整体含义**：论文提出并验证了一个核心论点——最危险的漏洞并非来自模型的“轻信”，而是**结构性缺陷**：CI/CD基础设施处理凭据和配置文件的方式，特别是`actions/checkout`对`GITHUB_TOKEN`的存储以及AI CLI工具对配置文件的自动加载，使得攻击者可以通过配置文件注入将恶意指令从低信任度提升至操作员信任级别。

### 6. 论文的主要结论与发现

- **结构性漏洞是根本**：所有受测AI提供商在默认配置下均易受至少一种攻击，最严重漏洞源于CI/CD基础设施（尤其是`actions/checkout`将`GITHUB_TOKEN`写入`.git/config`的行为）和AI CLI工具的配置文件自动加载机制，而非具体模型的行为。
- **配置文件注入绕过现有信任边界**：通过`pull_request`（而非`pull_request_target`）工作流，攻击者添加`CLAUDE.md`等文件可将载荷从用户信任级提升至操作员信任级，使前沿模型也无法可靠区分合法项目偏好与恶意覆盖。
- **模拟评估严重不可靠**：`AgentDojo`等模拟基准遗漏了71.2%的真实攻击（假阴性），并错误地将5.0%被基础设施阻止的通道标记为成功（假阳性）。真实执行揭示了平台特定的调度规则（如分叉仓库不填充`workflow_run.pull_requests`）对漏洞可利用性的决定性影响。
- **最有效的防御是工作流级措施**：针对每类攻击给出最低成本修复方案，如`persist-credentials: false`、工具限制（移除`Bash(grep:*)`等）、作者关联过滤等，但配置文件注入的判断操纵无法通过纯工作流控制完全消除，需人工审阅或仅将代理设为顾问权限。

### 7. 优点

- **真实性**：首次在真实GitHub运行器中执行AI工作流并评估安全性，覆盖沙箱、凭据、网络策略等生产环境的所有层次。
- **框架设计与可扩展性**：通过工作流、场景、评估器的正交抽象，以及离线优化模式、对抗性模型替换、攻击载荷可移植性，使框架易于扩展至新提供商、新平台（如GitLab）。
- **攻击分类清晰**：按MITRE ATT&CK分类组织案例，并明确定义了注入路径（PR正文 vs. 配置文件）的信任级别差异，突出了配置文件注入这一新颖且高危的攻击面。
- **对策分析务实**：针对每个确认攻击类，不仅给出最小成本修复，还分析其覆盖范围和残余风险，对实践者有直接指导意义。
- **伦理与负责任披露**：所有实验均在作者自有的临时仓库上进行，已向相关厂商（GitHub、Anthropic、OpenAI、Google）披露漏洞。

### 8. 不足与局限

- **攻击覆盖率有限**：仅枚举了11种具名攻击，虽覆盖了主要类别，但未声称穷尽所有注入可能性，因此“未成功”不代表工作流完全安全。
- **时效性与模型版本依赖**：结果反映的是评估时的模型版本与工作流配置，厂商可能通过补丁消除已确认漏洞，需在发布前重新验证。
- **威胁模型限定**：假设外部低权限攻击者仅能通过公开接口提交内容（如PR、问题），未涉及拥有推送权限的内部威胁或更复杂的供应链攻击。
- **防御评估局限性**：对某些攻击（如可用性攻击）缺少精确的工作流级修复（如GitHub事件载荷大小上限无法由YAML更改），且判断操纵的最终防御（人工审核）会降低自动化价值，难以平衡。
- **平台泛化性**：实证部分集中在GitHub Actions，虽然理论上分析了GitLab和Bitbucket的类推攻击，并实证移植了GitLab上的一个攻击，但Bitbucket的分析未经验证。
