## 摘要
工具增强的大语言模型智能体通常依赖于逐步的原子工具调用，每次调用、观察和值传递都在主推理轨迹中暴露。这造成了执行粒度不匹配：局部确定性的工具工作流被展开为重复的模型可见决策，消耗上下文并迫使模型在轨迹中管理低级数据流。我们引入了 HyperTool，一个统一的、可执行的 MCP 风格工具接口，它改变了工具执行的模型可见单元。模型通过一个代码块调用 HyperTool，该代码块可以通过原有模式调用现有工具、操作返回值并在本地传递中间结果，将确定性的工具子程序折叠为单个外部调用。为了训练模型使用此接口，我们从跨工具组合任务合成 HyperTool 格式的轨迹，并在真实 MCP 环境中验证它们。在 MCP-Universe 上，HyperTool 将 Qwen3-32B 的平均准确率从 15.69% 提升到 35.29%，将 Qwen3-8B 的平均准确率从 9.93% 提升到 33.33%，并在平均准确率上超越了 GPT-OSS 和 Kimi-k2.5，表明我们的 HyperTool 可以显著改善多步工具使用。

## Abstract
Tool-augmented LLM agents commonly rely on step-wise atomic tool calls, where each invocation, observation, and value transfer is exposed in the main reasoning trace. This creates an \emph{execution-granularity mismatch}: locally deterministic tool workflows are unfolded into repeated model-visible decisions, consuming context and forcing the model to manage low-level dataflow in the trace. We introduce \textbf{HyperTool}, a unified executable MCP-style tool interface that changes the model-visible unit of tool execution. A model invokes HyperTool with a code block that can call existing tools through their original schemas, manipulate returned values, and pass intermediate results locally, folding deterministic tool subroutines into a single outer call. To train models to use this interface, we synthesize HyperTool-format trajectories from cross-tool compositional tasks and verify them in real MCP environments. On MCP-Universe, HyperTool improves average accuracy from 15.69\% to 35.29\% on Qwen3-32B and from 9.93\% to 33.33\% on Qwen3-8B, and surpass GPT-OSS and Kimi-k2.5 on average accuracy, showing that our HyperTool can substantially improve multi-step tool use.

## 2. 方法论

### 2.1 核心思想
- 设计一个统一的、可执行的 MCP 风格工具 **HyperTool**。它将工具调用的基本单位从单次原子调用提升为一个可包含多步工具调用、局部计算与值传递的 **代码块**。
- 模型不再以逐步方式反复进入决策循环，而是直接生成一个 Python 代码块，通过 `call_tool(server, tool, args)` 的模式调用已有 MCP 工具，并在块内完成数据解析、过滤、聚合等确定性操作，最后将结果赋值给内置变量 `result` 返回给主轨迹。

### 2.2 关键技术与算法流程

- **问题形式化**：标准 MCP 接口下，每个动作 `a_t` 均为原子调用 `Call(τ_t, x_t)`；HyperTool 将其泛化为 `a_t = Block(S_t)`，其中 `S_t` 是一个可在局部运行时内执行的有界程序，可组合任意数量的底层工具调用。
- **数据合成三阶段**：
  1. **组合任务构造**：基于真实种子实体，引导教师模型在 MCP 环境中交互收集多源信息，并诱导出跨工具、跨维度的约束条件，确保问题必须通过组合多步工具才能解决。
  2. **HyperTool 轨迹收集**：利用规则（推理前置于调用、块级执行、内部逻辑约束）生成边界清晰的轨迹；同时引入 **上下文压缩**（对旧工具响应进行摘要以保持当前步聚焦）和 **本地修复**（当代码块失败时自动修复语法或模式错误）。
  3. **轨迹验证**：通过执行正确性过滤（排除语法错误、缺失调用等）和**证据一致性验证**（LLM 裁判多次投票，检查答案是否被工具证据支撑、推理是否跨步一致）筛选高质量轨迹。
- **监督微调（SFT）**：保留通过验证的轨迹，转为标准聊天格式训练样例，仅在模型生成的助手轮次（包括推理文本和 HyperTool 代码块）上计算损失，工具观察被剥离。由此教会模型何时调用 HyperTool、如何内部组合工具以及何时将观察返回到主推理轨迹。

## 7. 优点

- **统一且可执行的接口**：将任意复杂度的工具组合包裹在单一 MCP 风格工具中，既保持了与原有工具模式的兼容性，又大幅提升了执行抽象层级。
- **用执行时控制替代事后压缩**：不同于以往在轨迹生成后再摘要或裁剪的方法，HyperTool 在生成时即决定哪些内容不进入主上下文，更加本质地解决了上下文膨胀。
- **数据合成与验证流程完整**：从实体种子出发构建组合任务，结合上下文压缩、本地修复和严格的执行/证据双重验证，保证了 SFT 数据的质量和模型学习的可执行性。
- **全面的实验论证**：与多种热门执行范式公平对比，并提供详尽的消融、效率、行为分类和错误分析，有力支撑了方法的有效性。

## 8. 不足与局限

- **适用范围**：当前设计最适合本地确定性工具流程（如检索、过滤、计算、聚合），对于需要频繁进行语义判断、计划调整的任务，仍需返回逐步交互模式。
- **训练数据规模与多样性**：尽管覆盖了多种组合任务，但轨迹的领域广度和数量仍可进一步扩展，以提升模型在未见工作流上的泛化能力和分块边界决策的鲁棒性。
- **安全风险**：模型动态生成并执行 Python 代码带来了远程代码执行（RCE）风险，实际部署必须依赖严格的沙箱隔离和只读权限约束。
- **可解释性降低**：内部块的复杂处理逻辑对外部推理轨迹不可见，给调试和错误定位带来困难（如错误的排序条件隐藏在 Aggregate 块内）。
- **对代码生成能力的依赖**：在小型或非代码优化模型上，直接使用 HyperTool 可能导致语法错误或模式幻觉，需辅以强大的本地修复模块。
