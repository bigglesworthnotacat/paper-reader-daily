## 摘要
递归语言模型（RLMs）表明，在模型调用上进行递归是长上下文推理的有效策略，并且生产级编码代理已经开始编写能够大规模生成子代理的代码，最近体现在Anthropic的动态工作流中。我们命名并研究了这两条工作线之间的模式，其中递归单元是一个完整的代理套件，具备文件系统工具、代码执行和规划能力，而不是无工具的模型调用。我们称之为递归代理套件（RAH），并将其定义为套件递归，即对RLM模型递归的代码优先扩展。父代理生成并运行一个可执行脚本，该脚本并行生成子代理套件以处理细粒度工作负载，并使用结构化函数调用处理小子任务。我们提供了针对长上下文推理的对照评估。在将主干模型固定为GPT-5以匹配已发布的Codex和RLM基线的情况下，RAH在Oolong-Synthetic数据集（199个样本，13个上下文长度桶，最高达400万令牌）上将Codex编码代理基线从71.75%提升至81.36%，这一增益归因于套件而非模型。使用更强的主干模型Claude Sonnet 4.5，相同设计达到89.77%。

## Abstract
Recursive language models (RLMs) showed that recursion over model calls is an effective strategy for long-context reasoning, and production coding agents have begun to write code that spawns subagents at scale, most recently in Anthropic's dynamic workflows. We name and study the pattern between these two lines of work, where the recursive unit is a full agent harness with filesystem tools, code execution, and planning rather than a model call with no tools. We call this the Recursive Agent Harness (RAH) and frame it as harness recursion, the code-first extension to the model recursion of RLMs. A parent agent generates and runs an executable script that spawns subagent harnesses in parallel for fine-grained workloads and uses structured function calls for small subtasks. We provide a controlled evaluation on long-context reasoning. With the backbone held fixed at GPT-5 to match the published Codex and RLM baselines, RAH improves the Codex coding-agent baseline from 71.75% to 81.36% on Oolong-Synthetic (199 samples, 13 context-length buckets up to 4M tokens), a gain attributable to the harness rather than the model. With a stronger backbone, Claude Sonnet 4.5, the same design reaches 89.77%.

### 2. 论文提出的方法论

- **核心思想**：将递归的基本单元定义为完整的代理套件（包括文件系统读写、代码执行、规划、子代理生成等工具），而非单一的语言模型调用。父代理可根据任务规模自动选择生成子代理的方式。
- **关键技术细节**：
  - **两种子代理生成路径**：
    - **代码执行生成（Code-Execution Spawning）**：适用于大规模细粒度任务。父代理通过Shell工具编写并运行一个Python脚本，该脚本实例化`Task()`对象，并行生成子代理套件。此路径绕过API每轮工具调用数的限制，可扩展至数千并发子代理。
    - **JSON工具调用生成（JSON Tool-Call Spawning）**：适用于1～5个小子任务，直接发出结构化函数调用生成子代理。
  - **子代理架构**：每个子代理是一个完整的代理套件，拥有独立的上下文窗口、文件系统工具、代码执行能力和再次生成子代理的能力（递归）。递归深度有可配置上限（默认3）。子代理之间无共享内存，通过写入共享输出文件聚合结果。
  - **父代理行为**：收到任务后先检查文档规模，决定分解策略；自动选择合适的生成路径。所有子代理完成任务后，父代理从聚合文件中提取答案。
  - **回答提取**：子代理原始输出经过一次基于GPT-5的格式提取调用，将内容映射为标准答案格式（如`Label: spam`），再与真实答案进行精确匹配或数值评分（`0.75^|y−ŷ|`）。
- **算法流程（文字说明）**：任务输入 → 父代理分析并生成并行脚本（或直接调用） → 脚本并发启动子代理套件，每个子代理执行“规划-工具使用-反思”循环 → 各子代理将结构化JSON结果写入预设路径 → 父代理收集聚合 → 格式化提取 → 最终答案。

### 6. 论文的主要结论与发现

- **性能大幅提升**：在GPT‑5固定骨干下，RAH将Codex基线从71.75%提升至81.36%（绝对提升9.61个点），并将RLM的64.38%远甩身后。使用Claude Sonnet 4.5时进一步达到89.77%，表明设计可与更强模型复合增效。
- **增益来源明确**：由于主干匹配Codex基线，性能跃升归因于套件递归架构本身，而非模型更新。
- **各类题型表现**：语义型问题（USER、COMPARISON、LABEL）得分均超86%；NUMERIC题因`0.75^|diff|`评分规则放大计数误差导致得分偏低（69.33%）；DATE题因样本量小（5个）结果波动大。
- **跨上下文长度能力**：在绝大多数长度区间（包括4M tokens）均超越71.75%基线，未出现随长度单调衰减。
- **失败模式**：少量案例中父代理直接回答而非生成子代理；NUMERIC评分机制惩罚离群预测；小样本题型方差大。

### 7. 优点

- **定义了新范式**：首次命名并系统评估“套件递归”，将代码执行和子代理生成结合，扩展了递归语言模型的思路。
- **控制变量证明架构价值**：完全匹配主干模型，使性能差异可归因于套件设计，增强了结论可信度。
- **工程实用性**：通过生成代码并发控制子代理，绕开了API工具调用数量限制，可扩展至成千上万个并行子代理；隔离上下文避免干扰，聚合设计简洁。
- **互补性演示**：展示了工具访问和文件系统操作如何弥补纯模型递归的短板，二者结合性能远超RLM。
- **开放性和可复现性**：代码及提示词计划公开，实验设置透明。

### 8. 不足与局限

- **基准单一**：只在Oolong‑Synthetic上测试，未在现实场景或Oolong‑Real等更具噪声的数据集上评估泛化能力。
- **消融缺失**：没有单独隔离递归深度、子代理负载粒度、代码执行 vs JSON调用路径等设计选择的影响。
- **评分机制的干扰**：NUMERIC评分函数`0.75^|diff|`对微小计数误差惩罚极重，可能导致性能被低估而掩盖了正确的推理质量。
- **父代理回避问题**：父代理偶尔跳过子代理生成直接回答，尤其在长上下文中，可能引起性能波动。
- **样本平衡问题**：某些答案类型（如DATE）样本极少，导致统计结论不稳定。
- **资源评估不完整**：缺乏详细的 Token 消耗、延迟及实际成本的精确测量，限制了应用的可行性判断。
- **依赖代码生成健壮性**：要求父代理生成的脚本语法正确，一旦出错将导致任务失败，未分析此类鲁棒性问题。
