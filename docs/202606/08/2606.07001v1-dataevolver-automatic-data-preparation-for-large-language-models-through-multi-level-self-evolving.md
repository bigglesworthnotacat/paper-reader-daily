## 摘要
高质量训练数据对于大型语言模型（LLMs）至关重要，通常需要大量且昂贵的人工整理。现有的自动数据准备方法依赖于预定义的流水线或定制的人工指令，这限制了它们对不同数据分布的适应性，并缺乏来自高质量示例的原则性指导。在本文中，我们介绍了DataEvolver，这是首个自我进化的数据准备系统，能自动构建流水线将原始数据转化为高质量数据。DataEvolver采用多层次机制来确保流水线的可执行性和有效性。在算子层面，它逐步扩展算子集合以构建逻辑计划，同时解决依赖冲突。在流水线层面，它将逻辑计划实例化为可执行代码，并通过反馈循环迭代优化流水线编排，从而缩小所准备数据与高质量示例之间的分布差距。在七个基准上的实验表明，DataEvolver显著提高了数据质量，与在原数据上训练相比，下游LLM性能平均提升了10%，凸显了LLM与数据迭代协同进化的新机遇。

## Abstract
High-quality training data is essential to large language models (LLMs) and typically requires extensive and costly manual curation. Existing automatic data preparation methods rely on predefined pipelines or customized human instructions, which limits their adaptability to diverse data distributions and lacks principled guidance from high-quality examples. In this paper, we introduce DataEvolver, the first self-evolving data preparation system that automatically constructs pipelines to transform raw data into high-quality data. DataEvolver employs a multi-level mechanism to ensure both pipeline executability and effectiveness. At the operator level, it incrementally expands the operator set to construct a logical plan while resolving dependency conflicts. At the pipeline level, it instantiates logical plans into executable code and iteratively refines pipeline orchestration through a feedback loop that reduces the distribution gap between prepared data and high-quality examples. Experiments on seven benchmarks show that DataEvolver substantially improves data quality and achieves an average 10\% gain in downstream LLM performance compared with training on original data, highlighting new opportunities for the iterative co-evolution of LLMs and data.

## 2. 论文提出的方法论

### 2.1 问题形式化
- 给定原始数据集 \(D_{\text{raw}}\)、少量高质量种子示例集 \(S\)、可选的文字描述 \(T\)，目标是从算子库 \(O\) 中构建最优流水线 \(P^*\)。
- 流水线 \(P\) 表示为算子组成的有向无环图（DAG），数据准备过程为算子组合的函数：  
  \(D_{\text{out}} = \text{Exec}(P, \Theta, D_{\text{raw}})\)。
- 优化目标：最小化生成的 \(D_{\text{out}}\) 与从种子中提炼的高质量数据特征 \(C\) 之间的差距。

### 2.2 种子引导的数据理解
- 利用少量种子数据（实验中仅用 20 条）作为高质量目标的“轻量规格说明”，结合原始数据样本和进化经验 \(M\)，生成结构化的**数据画像（profile）\(C\)**，包含目标输出 schema、格式要求、质量约束、难度偏好等。
- 该画像作为后续流水线构建与进化的核心指导。

### 2.3 算子级自演化（Operator-Level Self-Evolving）
- 目标：确保流水线的**逻辑可执行性**。
- 做法：  
  1. 根据数据画像 \(C\) 和现有算子库 \(O\)，生成一份逻辑计划（DAG）。  
  2. 通过**逻辑检查器**诊断执行性缺陷（如缺失依赖、接口不匹配、DAG 断开、顺序冲突）。  
  3. 遇到缺陷时，尝试修复（插入桥接算子、替换、重排等）；若现有算子库不足以填补能力缺口，则自动**合成新算子**并注入算子库，同时动态调整 DAG 结构。  
  4. 迭代公式：\(G^{(t+1)}, O^{(t+1)} = f_{\text{op-evolving}}(G^{(t)}, O^{(t)}, C)\)，其中 \(O^{(t+1)}\) 可能经新增算子扩充。
- 经过多轮迭代，最终产出一个逻辑上可执行的算子 DAG。

### 2.4 流水线级自演化（Pipeline-Level Self-Evolving）
- 目标：在逻辑可执行的基础上，进一步保证**数据产出的有效性**（与种子分布对齐）。
- 做法：  
  1. 将最终逻辑计划实例化为可执行代码（含参数 \(\Theta\)），形成具体流水线 \(P\)。  
  2. 对原始数据的小规模子集 \(D_s\) 进行**试运行**，得到试运行结果 \(D'_s = P(D_s)\)。  
  3. 通过**质量审察器**对比 \(D'_s\) 与种子 \(S\)，产生差异信号和进化经验 \(\text{Exp}\)。  
  4. 将经验累积到记忆 \(M\) 中，更新公式：  
     \(M^{(t+1)} = f_{\text{pipe-evolving}}(M^{(t)}, \text{Exp})\)，其中 \(\text{Exp} = f_{\text{judge}}(P^{(t)}(D_s), S)\)。  
  5. 用更新的记忆重新执行种子引导的数据理解，调整数据画像 \(C\)，并重新触发算子级和流水线级自演化，直至试运行结果满足种子隐含的质量要求。
- 最终将经过多轮进化得到的流水线应用到全量数据。

### 2.5 关键技术亮点
- **算子库演化**：系统不仅使用 21 个预定义的通用算子，还能根据任务需求自动合成领域专用算子（如数学推理的步骤拆分与答案验证、Text-to-SQL 的 schema 裁剪与证据融合），避免为不同任务手动设计流水线。
- **经验反馈利用**：通过对比试运行输出与种子，提取“隐性需求”“格式失误”“风格不一致”等可操作经验，反哺到下一轮的流水线规划中。

- **性能提升显著**：DataEvolver 处理的数据在七个基准上，较原始数据（Vanilla SFT）平均提升约 10%，较强基线 DataFlow 平均提升约 2%。仅用 1k 高质量进化数据，性能可与 5k 原始数据匹敌。

## 8. 不足与局限

- **模态单一**：目前仅针对文本数据，尚未扩展到图像、多模态等领域，作者也承认这是未来发展。
- **依赖种子质量**：系统有效性建立在种子数据能准确反映目标格式与质量标准的前提之上，若种子存在偏差或过于单一，流水线和数据质量会受到影响（尽管实验已显示一定鲁棒性）。
- **未与其他种子引导方法对比**：实验主要对比了 Vanilla SFT 和 DataFlow，但未与一些专门做种子引导数据选择的系统（如 DSIR、DoReMi、DoPAMine）直接比较，缺乏在更广数据准备谱系中的定位。
- **进化迭代开销**：虽然单次准备令牌消耗低于 DataFlow，但内部多次试运行和自演化（如调用 LLM Judge）可能带来额外计算成本，论文未细致拆解各阶段的实际算力与总进化成本。
- **流水线并行与执行复杂度**：生成的长链 DAG 可能包含多个需要调用 LLM 的算子，当算子数增多时，调度的可靠性和错误传播风险未深入探讨。
- **评测局限性**：下游效果仅用 SFT 下的基准测试衡量，未在预训练、RLHF 等更多训练范式中验证，也未评估生成数据的多样性或对模型安全性/偏见的潜在影响。
- **环境和工具绑定**：实验基于 ms-swift 和 OpenCompass，自定义性强，较少交代与其他通用框架（如 Hugging Face Transformers、vLLM 直接整合）的兼容细节。
