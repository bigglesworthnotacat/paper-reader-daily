## 摘要
大型语言模型（LLMs）正迅速演变为与外部工具和环境交互的代理系统，引入了新的安全风险，例如通过不可信外部源的间接提示注入攻击。现有防御主要侧重于在推理时阻止恶意内容，而当前的红队测试方法主要优化攻击成功率。因此，开发者对于潜在提示注入如何在代理中出现和传播的可视性有限。我们提出PI-Hunter，一个用于主动暴露LLM代理中漏洞的自动代理审计框架。PI-Hunter构建真实源感知的测试用例，并通过反馈驱动的探索迭代演进它们，以诱导代理检索并揭示嵌入在外部环境中的潜在恶意指令。跨多个基准、代理架构、攻击和防御的广泛实验表明，PI-Hunter显著提高了漏洞暴露和攻击面覆盖率，优于强大的自动红队测试基线方法，同时在现有提示注入防御下仍保持有效性。

## Abstract
Large Language Models (LLMs) are rapidly evolving into agentic systems that interact with external tools and environments, introducing new security risks such as indirect prompt injection attacks through untrusted external sources. Existing defenses mainly focus on blocking malicious content at inference time, and current red-teaming methods primarily optimize attack success. As a result, developers have limited visibility into how latent prompt injections emerge and propagate through agents. We propose PI-Hunter, an automated agentic auditing framework for proactive vulnerability exposure in LLM agents. PI-Hunter constructs realistic source-aware test cases and iteratively evolves them through feedback-driven exploration to induce agents to retrieve and reveal latent malicious instructions embedded within external environments. Extensive experiments across multiple benchmarks, agent architectures, attacks, and defenses demonstrate that PI-Hunter substantially improves vulnerability exposure and attack-surface coverage over strong automated red-teaming baselines, while remaining effective under existing prompt injection defenses.

## 1. 论文的核心问题与整体含义

大型语言模型正由独立模型快速演化为与外部工具、记忆体和动态数据源交互的**智能体**。此类范式虽然大幅扩展了 LLM 的能力，却也引入了新的系统级安全风险——**间接提示注入**。恶意指令可以不直接出现在用户输入的提示中，而是潜藏于智能体读取的邮件、网页、API 返回或检索文档等**不可信外部源**里，并在特定推理路径或交互模式下被激活，导致敏感操作、数据外泄或权限滥用。

现有防御思路主要聚焦在推理时**检测或拦截恶意内容**，而现有的自动红队方法大多以**攻击成功率为优化目标**，只构造少量高成功率的攻击提示。这导致开发者对以下问题缺乏可视性：潜在的注入指令 **什么时候、通过哪条交互路径、在何种推理状态下** 才会从潜伏走向暴露。换言之，传统的“攻击优化”范式未能回答**系统级安全审计**的核心问题：如何在实际部署前**系统性地暴露隐藏的脆弱摄入路径**。

本文提出 **PI‑Hunter**，一个自动化的智能体审计框架，将重心从“攻击优化”转向“注入暴露”。PI‑Hunter 的目标不是寻找一两条奏效的攻击，而是像“诊断猎人”一样**遍历智能体的安全边界**，挖掘潜伏在不可信源中的隐秘逻辑陷阱，定位**源‑工具‑摄入路径**，从而为安全加固提供精细化的系统级情报。

## 2. 论文提出的方法论

PI‑Hunter 由三个核心阶段构成：**静态分析**、**进化式利用**和**修补‑再探索**，其运行流程近似一种**基于 LLM 的遗传算法**。

### 2.1 静态分析
在迭代审计开始前，PI‑Hunter 对目标智能体进行一次性静态分析，识别其**工具集、检索接口、API 模式**和可访问的文件类型，特别关注那些处理不可信或非结构化外部内容的接口（如邮件客户端、网页搜索、文档检索等），以及能执行特权操作的工具（如发送消息、修改存储数据）。

### 2.2 进化式利用循环
这一阶段是 PI‑Hunter 的核心，包含三个关键步骤：

- **源感知元播种 (Source‑Aware Meta Seeding)**  
  基于静态分析给出的高风险接口，生成一组**源感知的查询测试用例** \(Q_0\)。每条用例同时满足：① 符合实际用户意图（保持真实性）；② 激活特定高风险接口（如 `get_unread_email`、`search_web`）；③ 要求智能体详细展示或总结检索到的外部内容（如“引用”“列出”“比较”），以逼迫其暴露隐藏指令。

- **轨迹执行与评估 (Trajectory Execution and Evaluation)**  
  将种群中的测试用例提交给目标智能体，记录完整的**执行轨迹**（检索内容、推理链、工具调用、最终输出）。  
  评估器基于**评分准则**对轨迹进行审计，检测信息注入的迹象：意图偏离、可疑指令传播、权威混淆、异常工具使用、敏感动作执行等。评估器不给出“是/否”攻击的二元标签，而是生成**结构化诊断反馈**，指出可疑行为及其关联的脆弱摄入路径。

- **反馈驱动突变 (Feedback‑Driven Mutation)**  
  根据评估器反馈，**选择性应用突变算子**，对查询进行修改以扩大探索范围。算子包括两大类：
  - **通用突变**：如指令层次覆盖、参数模糊化、角色反转、分隔符劫持、编码混淆等；
  - **领域特定突变**：针对 workspace、banking、travel、github 等场景定制（如假装新员工要求获取大量历史数据、模拟紧急欺诈调查、伪造生产环境宕机等）。

  此外，PI‑Hunter 还执行**LLM引导的元突变**：根据历史突变效果，总结并重组成功的算子，生成新的突变策略，使探索策略随环境自适应进化。

### 2.3 验证与共演化
一旦注入被暴露，**修补器 (Patcher)** 会对已发现的摄入路径施加**轻量级瞬时缓解**，例如在系统提示中临时黑名单恶意指令、限制脆弱接口或隔离受污染源。这些缓解**并非永久防御**，其目的是**重塑探索地貌**，迫使后续审计转向尚未被发现的攻击面，逐步扩大覆盖范围。缓解后的智能体重新进入利用循环，实现**渐进式攻击面展开**。

## 8. 不足与局限

- **真实生产环境验证缺失**：实验均在模拟基准中进行，未在如 OpenClaw 等真实、动态的生产级智能体系统上测试，可复现性和泛化边界有待确认。
- **仅发现漏洞，未提供修复方案**：PI‑Hunter 定位于审计框架，发现漏洞后由开发者自行处理，未提供自动防御或安全加固建议。
- **防御评估尚不全面**：测试的防御机制有限，未包含更多的推理时检测、安全对齐训练或执行隔离等方法；也未评估对自适应防御的对抗性鲁棒性。
- **源感知依赖静态分析质量**：若静态分析未能准确识别高风险接口，播种阶段可能遗漏关键攻击面。
- **评估依赖 LLM 评判**：轨迹评分和注入识别均由评估器 LLM 完成，可能存在误判或偏差，影响精确率和召回率的绝对可信度。
- **无法处理完全封闭的注入**：如果恶意指令完全不被检索或以极深奥的方式隐藏，即使演化探索也难以触发。
