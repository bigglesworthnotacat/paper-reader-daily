## 摘要
紧凑语言模型降低了工具代理的成本、延迟和部署风险。然而，MCP式工具使用不仅需要孤立的函数调用：代理必须从实时目录中发现工具、满足模式、保持中间输出之间的依赖关系，并将最终回答建立在执行证据之上。小型规划器经常生成看似合理的工具工作流图，但这些图在工具解析、参数验证、依赖跟踪或执行时失败。我们认为，小规模语料蒸馏难以有效应对这种失败模式。数百条教师轨迹可以教会工作流格式，但很少涵盖在不断变化的工具目录下修复失败计划所需的恢复行为。我们提出Evoflux，这是一种推理时进化搜索方法，将紧凑工具使用视为可执行工具工作流的修复。它通过结构化编辑、执行反馈、自适应强度、元引导重新设计和多样性剪枝来进化类型化工作流图。在涵盖实时MCP服务器和250个工具的留出MCP-Bench任务上，Evoflux将小型规划器的执行可行率从大约3%提升至17-24%。相比之下，在相同搜索挖掘数据上进行SFT和SFT+DPO，其表现与零样本性能相当、更差或崩溃；ReAct达到更高峰值，但方差和令牌成本更高。这些结果表明，在教师轨迹预算稀缺的情况下，基于执行基础的搜索更为可靠。

## Abstract
Compact language models (LMs) reduce cost, latency, and deployment risk for tool agents. Yet MCP-style tool use requires more than isolated function calling: an agent must discover tools from live catalogs, satisfy schemas, preserve dependencies across intermediate outputs, and ground final responses in executed evidence. Small planners often generate plausible workflow graphs that fail under tool resolution, parameter validation, dependency tracking, or execution. We argue that this failure mode is poorly handled by small-corpus distillation. A few hundred teacher traces can teach workflow format, but rarely cover the recovery behavior needed to repair failed plans over changing tool catalogs. We introduce Evoflux, an inference-time evolutionary search method that treats compact tool use as the repair of executable tool workflows. It evolves typed workflow graphs through structured edits, execution feedback, adaptive intensity, meta-guided redesign, and diversity pruning. On held-out MCP-Bench tasks spanning live MCP servers and 250 tools, Evoflux raises execution feasibility from roughly 3% to 17-24% across small planners. In contrast, SFT and SFT+DPO on the same search-mined data match, underperform, or collapse below zero-shot performance; ReAct reaches higher peaks, but with higher variance and token cost. These results show that execution-grounded search is more reliable under scarce teacher-trace budgets.

### 2. 方法论：Evoflux 推理时进化搜索

Evoflux 将紧凑代理的工具使用问题**建模为可执行工具工作流的修复问题**，在推理时通过进化循环生成、评估、变异和选择工作流图。

- **核心循环（Algorithm 1）**：
  1. **初始化**：用规划器尝试多次生成首个可执行工作流（失败时携带错误上下文重试）。
  2. **变异与进化**（每轮）：  
     - 根据**自适应强度 I** 控制探索/利用程度；  
     - **父代选择**：按探索概率随机均匀选或锦标赛选高评分个体；  
     - **收集证据**：从父代执行日志中提取节点级失败信息；  
     - **变异**：探索模式用随机堆放编辑，利用模式让 LLM 根据证据提出编辑；  
     - **可行性检查与执行**：通过编译、静态检查、执行、LLM 评判打分（零分不可行）；  
     - **生长信号 G** 通过相对提升的 EMA 更新，强度 I 随 G 动态变化（G 低时偏向探索）；  
     - 当 G 低于阈值且冷却倒计时为零时，触发**元引导重设计**，让规划器基于最佳候选和最差候选提出全新结构。

### 6. 主要结论与发现

- **推理时进化搜索显著提升执行可行性**：在留出验证分裂上，Evoflux 将小型规划器的执行成功率从约 3% 提升至 17–24%（如 Llama-3.2-3B 的 Valid Init 3% → Valid Best 17%；Qwen3.5-4B 从 3% → 24%）。
- **SFT 和 SFT+DPO 在极低轨迹预算下不可靠**：它们要么与零样本持平，要么劣于零样本，甚至出现“规划完全崩溃”（如训练后的 Qwen3.5-4B 连可执行计划都无法产生，回退至启发式方法）。

- **搜索放大的主要是工具选择、参数准确性等可修复故障**，而证据锚定和深层依赖仍具挑战。
- **Token 效率前沿**：较小模型（Nemotron-3-Nano-4B、granite-4.0-h-micro、Qwen3.5-4B）在搜索后形成较好的得分/令牌前沿。

### 7. 优点

- **问题定义清晰**：将紧凑工具代理的成败明确归因于工作流的可执行性修复，并提供了一种进化搜索的解法，而不简单追求更多训练数据。
- **以执行反馈为核心**：所有评分和变异均基于真实的编译错误、模式验证和执行痕迹，避免文本表面的“合理幻觉”。
- **自适应控制**：基于生长信号的多态强度控制，能够动态切换探索与利用，并用元引导跳出局部停滞，提高了搜索效率。
- **多样性保持**：动作哈希分桶剪枝避免了种群坍缩为近乎重复的工作流。
- **严格的可比性设计**：用相同的小量教师轨迹构建训练集，使训练与搜索在同一预算下公平对比，削弱了类似“蒸馏需要大数据”的常见反驳。
