## 摘要
自动调适系统如A-Evolve、GEPA和Meta-Harness，通过从执行反馈中优化提示、技能、工具、记忆及支持基础设施来改进大语言模型智能体，但它们通常在固定的离线基准上进行评估。实际部署则呈现开放式任务流：历史记录没有固定终点地增长，异构任务需要不同的调适，问题分布随时间变化。这些挑战使得单一且反复密集更新的调适变得脆弱，导致准确率早期达到峰值后下降，从而性能退化。这促使我们进行持续的调适构建并适配具体任务。我们提出了Adaptive Auto-Harness，一个面向此类任务流的框架和系统。该框架将到达理想调适的差距分解为进化损失和适配损失。系统通过有状态的多智能体进化器、带有求解时路由的调适树，以及在历史缺乏所需信号时的人工引导挂钩来解决这些损失。在预测市场、安全竞赛和事件预测任务流中，Adaptive Auto-Harness优于五个现有的自动调适基线，消融实验将收益归因于更好的构建、路由或定向人工引导。代码见https://github.com/A-EVO-Lab/AdaptiveHarness。

## Abstract
Auto-harness systems such as A-Evolve, GEPA, and Meta-Harness improve LLM agents by optimizing prompts, skills, tools, memories, and supporting infrastructure from execution feedback, but they are typically evaluated on fixed offline benchmarks. Real deployments instead present open-ended task streams: histories grow without a fixed endpoint, heterogeneous tasks require different harnesses, and problem distributions shift over time. These challenges make a single repeatedly and densely updated harness brittle, causing performance degradation as accuracy peaks early and then declines. This motivates sustained harness construction with task-wise adaptation. We introduce Adaptive Auto-Harness, a framework and system for such streams. The framework decomposes the gap to an oracle harness into evolution loss and adaptation loss. The system addresses these losses with a stateful multi-agent evolver, a harness tree with solve-time routing, and human-steering hooks for cases where history lacks the needed signal. Across prediction-market, security-competition, and event-forecasting streams, Adaptive Auto-Harness outperforms five existing auto-harness baselines and ablations attribute gains to better construction, routing, or targeted human steering. Code is available in https://github.com/A-EVO-Lab/AdaptiveHarness .

## 2. 论文提出的方法论

- **核心思想**：将智能体调适构建问题转化为相对于理想调适的遗憾最小化，并将遗憾分解为**进化损失**和**适应损失**，从而指导系统设计：
  - 进化损失反映进化器类别本身的能力上限（如能否构建多文件基础设施）。
  - 适应损失反映 “在求解时无法按任务重新构建最优调适” 造成的性能缺口。
- **关键技术细节**：
  - **有状态多智能体进化器**（应对无界流）：
    - 将进化过程拆分为四个阶段：**分析（Analyst）→ 研究（并行 Researchers）→ 构建（Builder）→ 验证（Verifier）**，每个角色拥有独立的完整上下文预算，避免单智能体窗口瓶颈。
    - 引入**时序揭示反馈**：仅在任务结果解析日期后才向进化器暴露标签，防止未来信息泄漏。
    - 维护**跨周期持久状态**（任务板、研究日志、架构文档、测试），使进化器能够连续积累和复用经验。
  - **求解时适配——调适树路由**（应对异构与非平稳）：
    - 进化器构建一个 **Git 分支形式的调适树**，每个分支针对一类任务（如 `/branch/crypto`、`/branch/sports`）配备独立的提示、技能和工具。
    - 在求解时，一个**路由智能体**通过 `git show` 读取各分支说明，根据任务上下文选择最匹配的分支，然后求解器切换至该分支执行。
  - **人机协同通道**（应对历史经验无法覆盖的缺失信号）：
    - **任务板引导**：人在分析阶段审阅任务板，补充领域方向或数据源访问指引。
    - **研究阶段交互协助**：当研究智能体遇到 API 鉴权等阻塞时，实时向人类求助，解锁执行。
- **分析框架与公式**（文字说明）：
  - 定义调适的效用 \(V(C, x_t)\) 为在任务 \(x_t\) 上以调适 \(C\) 求解的期望奖励。
  - 遗憾 = 全历史最优有效用 − 当前进化器构建调适的效用。
  - 将该遗憾分解为：**进化损失** \(L_{evo}\)（取决于进化器类别 \(\Phi\) 的能力上限）+ **适应损失** \(L_{adapt}\)（取决于具体进化器实例是否针对每个任务构建了专用调适）。这一分解直接指明了两类主要优化方向。

## 8. 不足与局限

- **基准覆盖有限**：仅在三个任务流上进行验证，虽覆盖了预测市场、安全挑战和事件预测，但对于其他更长时间跨度、更广空间分布的部署流尚需检验。
- **损失度量非直接量化**：进化损失和适应损失在实验中主要依靠诊断性分析和对比控制实验来推断，并未给出直接的、能精确估计理想调适的估计量。
