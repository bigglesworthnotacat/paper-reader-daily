## 摘要
大型语言模型(LLM)智能体通过将语言模型与外部工具和环境集成，日益自动化地完成复杂任务。然而，它们的自主性带来了重大安全风险：智能体可能执行破坏性指令、泄露敏感数据或违反领域约束。现有的安全方法面临根本性权衡：手工规则可解释但脆弱，过于保守的规则会阻止安全操作（高误报），而宽松的规则会遗漏不安全行为（高漏报）。神经分类器缺乏安全关键部署所需的可解释性。我们提出AutoSpec，一个通过反例引导的归纳合成（CEGIS）结合归纳逻辑编程（ILP）来自动演化已部署的专家设计安全规则的框架。从专家规则和带注释的轨迹流开始，AutoSpec迭代评估规则，挖掘假正例和假负例反例，使用ILP学习哪些谓词能区分它们，生成候选规则编辑，并验证候选以选择最佳修订。关键洞察在于，ILP能高效识别在假负例中频繁出现但在假正例中很少出现（反之亦然）的谓词，从而极大剪枝规则编辑的指数级搜索空间。该过程持续进行直至收敛，产生平衡精确率和召回率的可解释规则。我们在291条跨越代码执行和具身智能体领域的执行轨迹上评估AutoSpec。AutoSpec在两个领域分别将规则F1提升至0.98和0.93，在保持高召回率的同时实现高达94%的假正例减少，并在4-5次迭代内收敛。ILP引导的方法相比启发式CEGIS实现了高达4.8倍的F1提升。学习到的规则对人类可读、可审计，并能泛化到未见场景。

## Abstract
Large language model (LLM) agents increasingly automate complex tasks by integrating language models with external tools and environments. However, their autonomy poses significant safety risks: agents may execute destructive commands, leak sensitive data, or violate domain constraints. Existing safety approaches face a fundamental tradeoff: hand-crafted rules are interpretable but brittle, with overly conservative rules blocking safe operations (high false positives) while permissive rules miss unsafe behaviors (high false negatives). Neural classifiers lack the interpretability required for safety-critical deployments.   We present AutoSpec, a framework that automatically evolves deployed expert-designed safety rules from user safe/unsafe annotations through counterexample-guided inductive synthesis (CEGIS) guided by inductive logic programming (ILP). Starting from the expert rules and a stream of annotated traces, AutoSpec iteratively evaluates rules, mines false-positive and false-negative counterexamples, uses ILP to learn which predicates discriminate them, generates candidate rule edits, and verifies candidates to select the best revision. The key insight is that ILP efficiently identifies predicates that appear frequently in false negatives but rarely in false positives (or vice versa), dramatically pruning the exponential search space of rule edits. This continues until convergence, producing interpretable rules that balance precision and recall.   We evaluate AutoSpec on 291 execution traces spanning code execution and embodied agent domains. AutoSpec raises rule F1 to 0.98 and 0.93 across the two domains, achieving up to 94% false positive reduction while maintaining high recall, and converges within 4-5 iterations. The ILP-guided approach achieves up to 4.8x higher F1 than heuristic CEGIS. The learned rules are human-readable, auditable, and generalize to unseen scenarios.

## 1. 核心问题与整体含义（研究动机和背景）

- **问题**：LLM Agent 在执行复杂任务（如代码执行、机器人控制）时存在严重安全风险（删除文件、数据泄露、违反领域约束）。现有安全方案面临根本性权衡：
  - **手工规则**：可解释但脆弱——过于保守导致高假正例（FP），过于宽松导致高假负例（FN），且无法随 Agent 行为演变自动更新。
  - **神经分类器**（如 LLM Guard）：可自适应但缺乏可解释性，不适合安全关键部署。
- **核心洞察**：Agent 运行中积累的用户标注（safe/unsafe）可作为反例，用于自动演化已部署的专家规则，而无需从零编写。作者将规则维护问题形式化为 **反例引导的归纳合成（CEGIS）** 问题，并用 **归纳逻辑编程（ILP）** 高效剪枝指数级搜索空间。

## 2. 方法论

### 核心思想
以初始专家规则集 \(R_0\) 和带标注的执行轨迹 \(D_L\) 为输入，通过 CEGIS 循环迭代精炼规则。每轮：
1. 评估当前规则，挖掘 FP（误报警）和 FN（漏检）作为反例。
2. 使用 ILP（ILASP 求解器）从反例中学习**最具区分力的谓词**（即频繁出现在 FN 中而少见于 FP 的谓词，或反之）。
3. 基于学习到的谓词，应用受限的编辑算子生成候选规则。
4. 验证候选规则在 \(D_L\) 上的得分，选择最佳更新，迭代直至收敛。

### 关键技术细节
- **谓词库**：每个领域预定义 16–24 个布尔谓词（如 `destructive_os_inst`、`near_danger_zone`），代理工具调用事件上求值。
- **规则表示**：规则 \( r = (g, \phi) \)，其中 \(g\) 为触发器（固定），\(\phi\) 为谓词合取式。轨迹安全判定：存在任一规则在某事件上触发视为 unsafe。
- **编辑算子**（表 2）：
  - `AddConjunct`：添加谓词合取（收紧规则）
  - `AddException`：添加否定谓词（增加例外）
  - `AddDisjunct`：添加析取分支（放宽规则）
  - `Relax`：删除一个谓词（放宽）
- **ILP 学习过程**：以 FN 为正例、FP 为负例，学习最小谓词组合（合取范式），使得每个正例至少在一个事件上满足该组合，负例均不满足。输出谓词列表及推荐编辑方向。
- **候选生成与选择**：三类策略——单步编辑、混合修复链、失败驱动分支（为残余 FN 添加高精度析取分支）。评分函数 \(S(R') = (F1, Precision, -U, -|R'|)\) 按字典序排序。
- **收敛**：单调提升得分，局部最优停止，或达到迭代预算。

- **可解释性**：用户研究显示 AutoSpec 规则在可解释性（4.97 vs 3.45）、可操作性（4.83 vs 2.73）、信任度（4.80 vs 2.77）上显著优于 LLM 分类器，84.2% 的对比中偏好 AutoSpec。

## 7. 优点

- **概念新颖**：将安全规则维护形式化为 ILP 引导的 CEGIS 问题，使静态规则集具有自适应能力，同时保留可解释性。
- **技术优雅**：ILP 高效剪枝指数级编辑空间，仅需少量反例即可学习关键谓词，避免盲目搜索。
- **实用性强**：直接基于已有专家规则和运营中产生的标注，无需重新开发；输出规则可直接部署。
- **评估全面**：覆盖两个异构领域，对比多种 baseline，包含泛化测试和用户研究，实验设计严谨。
- **可操作性好**：用户研究证明从业者能理解并修改规则，而神经分类器则不可干预。

## 8. 不足与局限

- **领域覆盖有限**：仅验证了代码执行和具身 Agent，未涉及 Web Agent、数据库 Agent 或其他常见场景。
- **谓词库依赖性**：规则表达能力受限于预定义谓词；若谓词库不完整，ILP 无法学习到缺失的关键特征（论文指出 unresolved 反例可驱动谓词库扩展，但尚未自动完成）。
- **NP-hard 问题**：不保证全局最优，仅局部最优；解质量依赖谓词库丰富度和反例质量。
- **标签质量**：依赖人工标注，标注噪声可能影响演化效果；论文未讨论标签纠错机制。
- **时序/状态限制**：当前谓词仅评估单事件，未原生支持时序或跨事件状态；论文提到可用 LTL 扩展作为未来工作。
- **计算开销**：虽然算力要求不高，但每轮需调用 ILASP（有 120 秒超时），大规模轨迹或高频更新场景可能需优化。
