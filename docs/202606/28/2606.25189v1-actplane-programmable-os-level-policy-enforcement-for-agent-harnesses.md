## 摘要
AI Agent越来越多地通过框架在生产环境中运行，框架是围绕LLM的软件，包括一个执行安全和有效性策略的引擎，例如“在提交前运行测试”。执行这些策略需要弥合语义鸿沟：策略意图以未明确指定的自然语言表达，而执行必须作用于具体的系统操作，例如运行哪个测试。许多策略还定义了事件顺序或数据流操作。然而，现有方法存在不足。工具调用防护栏忽略了绕过工具层的系统操作，而操作系统沙箱控制资源访问而非操作，返回不透明的错误，使Agent困惑。我们的关键洞察是，策略上下文位于最接近任务的Agent内部，而执行必须在操作系统层面进行以覆盖所有执行路径。我们提出ActPlane，一个策略引擎，允许Agent声明策略，并在操作系统内核中通过语义反馈和隔离来执行这些策略。ActPlane使用一个简单的信息流控制（IFC）DSL来支持跨事件策略。我们使用eBPF实现ActPlane，并在实证研究、编码任务基准测试和安全基准测试的策略上评估它。ActPlane提高了策略合规性，包括工具调用拦截无法观察到的间接执行路径上的合规性，开销为1.9%-8.4%。ActPlane位于https://github.com/eunomia-bpf/ActPlane

## Abstract
AI agents increasingly run in production through harnesses, the software around the LLM, including an engine that enforces safety and effectiveness policies, e.g., 'run tests before committing.' Enforcing these policies requires bridging a semantic gap: policy intent is expressed in underspecified natural language, while enforcement must act on concrete system actions, e.g., which test to run. Many policies also define event ordering or data flow actions. Yet existing approaches fall short. Tool-call guardrails miss system actions that bypass the tool layer, while OS sandboxes control resource access instead of actions, returning opaque errors that confuse the agent. Our key insight is that policy context lives within the agent closest to the task, while enforcement must happen at the OS to cover all execution paths. We introduce ActPlane, a policy engine that lets agents declare policies and enforces them in the OS kernel with semantic feedback and isolation. ActPlane uses a simple information-flow control (IFC) DSL to support cross-event policies. We implement ActPlane with eBPF and evaluate it on policies from the empirical study, coding-task benchmarks, and safety benchmarks. ActPlane improves policy compliance, including on indirect execution paths that tool-call interception cannot observe, with 1.9%-8.4% overhead. ActPlane is at https://github.com/eunomia-bpf/ActPlane

## 1. 论文的核心问题与整体含义

### 研究动机与背景
- AI Agent在生产环境中通过“框架”（harness）运行，框架需强制执行安全和有效性策略（如“提交前先运行测试”）。
- 当前存在**语义鸿沟**：策略意图以自然语言表达（模糊、未明确指定），但执行必须作用于具体的系统动作（如具体的测试命令、文件路径）。
- 现有方法均不足：
  - **工具调用防护栏**：只能捕获Agent直接调用的工具，无法观测间接执行路径（如Agent编写的脚本中执行`git commit`）。
  - **OS沙箱**：控制资源访问（如允许/禁止文件读写）而非动作语义，返回不透明错误（如`EPERM`），Agent无法理解违规原因。
- 关键洞察：策略上下文（如“测试命令是什么”“数据目录在哪”）最接近任务的Agent，但执行必须在OS层面覆盖所有路径（包括子进程、shell输出）。

### 整体含义
- ActPlane是第一个让Agent用简单DSL声明策略，并在OS内核（基于eBPF）中强制执行，同时提供语义反馈的系统。

## 2. 方法论

### 核心思想
- **信息流控制（IFC）**：在OS对象（进程、文件、网络端点）上附加标签，跟踪数据流向，实现跨事件策略执行。
- **分层策略域**：多级域允许子域继承父域策略，但禁止削弱高层安全约束。
- 提供**语义反馈**：当策略违反时，返回可读的错误原因（如“提交前未运行测试，请先运行`npm test`”）。

### 关键技术细节
- **DSL设计**：五要素——源（source）、目标操作（target）、效果（effect）、可选时序门（gate）、原因（reason）。效果包括`block`（阻止）、`kill`（终止）、`notify`（通知）。
- **编译器与eBPF引擎**：DSL规则编译为eBPF程序，通过BPF-LSM（预操作）和tracepoints（后操作）挂载到exec、open、write等系统调用。
- **标签传播**：每个对象维护64位掩码，通过`fork`、`exec`、`read`、`write`等操作逐位OR传播，保证单调递增。
- **时序门**：`after ... since ...`模式，确保测试在最近修改之后运行；`exits N`限定退出码。
- **分层域**：通过进程树关联策略域，子域可添加局部规则但不能移除父域规则；运行时通过环缓冲区提交增量。

### 算法流程（文字说明）
1. 策略生成：Agent读取任务描述和仓库，调用LLM翻译自然语言策略为ActPlane DSL规则。
2. 编译加载：Rust编译器将DSL编译为固定大小配置，通过eBPF加载器放入内核。
3. 运行时执行：
   - 系统调用触发eBPF钩子，检查当前进程/文件/网络端点的标签掩码。
   - 匹配规则后，执行`block`（拒绝syscall并返回语义错误）、`kill`（终止进程）、`notify`（记录日志）。
   - 标签沿进程、文件、网络传播。
4. 反馈循环：用户空间读取内核报告，格式化后返回给Agent，指导其修正行为。

## 6. 主要结论与发现

1. **DSL覆盖**：agent可自动将607个策略全部翻译为可编译规则，成本低（$0.028/策略）。
2. **合规性**：ActPlane DCR达75.8%，比基线高22-31个百分点；在间接执行路径（脚本、隐藏）上尤其显著（基线近零检测）。

4. **开销低**：端到端1.9%-8.4%，单syscall延迟增加几到几十微秒，远小于LLM推理时间。
5. **安全预防**：在361个安全任务中预防74%基线不安全行为。
6. **跨模型稳定**：在Qwen3.6和DeepSeek-Pro V4上保持排名一致。

## 7. 优点

- **新颖性**：首次将IFC思想应用于agent策略的OS级执行，弥合自然语言与系统动作的语义鸿沟。
- **完整性**：从实证研究、系统设计、DSL、编译器、eBPF实现到全面评估，闭环完整。
- **工程可行性**：基于eBPF、低开销、开源，可直接集成到现有agent框架（如Claude Code、Codex CLI）。
- **安全性分层**：层次域确保高层约束不可被绕过，适合多Agent、多租户场景。
- **反馈实用性**：语义反馈使agent能自我修正，而非简单拒绝。

## 8. 不足与局限

- **覆盖范围有限**：仅处理OS可执行策略（事件相关，占45%），内容策略（需linter）和非OS可观行为（聊天）不在范围内。
- **粒度限制**：文件级/进程级IFC，不支持字节级细粒度跟踪，可能错过间接流（如通过云服务API绕过）。
- **钩子覆盖缺口**：未覆盖所有协议（如WebDAV上传、数据库突变），部分服务端操作无法拦截。
- **衰减问题**：单调标签可能导致过度限制（over-tainting），虽可通过新建子进程清除，但缺乏动态降级机制。
- **单机假设**：目前主要面向单机agent执行，分布式/微服务场景下标签传播复杂，未验证。
- **依赖LLM生成策略**：质量受限于LLM能力，虽可迭代修正，但极端情况下可能错失策略。
