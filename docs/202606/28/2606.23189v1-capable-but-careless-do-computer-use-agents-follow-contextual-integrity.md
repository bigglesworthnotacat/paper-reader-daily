## 摘要
计算机使用代理（CUAs）现在代表用户在个人应用程序（如电子邮件、日历和待办事项列表）中执行操作。这种跨应用程序访问很有用，但也带来了一个在很大程度上被忽视的隐私风险：当代理在一个上下文中工作时，它可能会从另一个上下文中拉入在该上下文中不适当的信息。因此，我们引入了AgentCIBench，这是一个评估框架，可将这种风险转化为可执行、确定性评分的场景。我们针对CUA中的三种常见故障模式：视觉共位（代理拉入UI中任务目标旁边被禁止的项）、任务模糊过度分享（代理针对不明确的提示输出大量个人状态信息）以及收件人不匹配（代理将内容发送给不适当的收件人）。我们评估了15个前沿代理，发现失败率高得惊人：15个中有11个在超过50%的场景中发生泄露，平均泄露率为67.9%，并且当代理在环境中端到端执行任务时，同样的失败仍然存在。我们发布AgentCIBench以鼓励开发更安全的计算机使用代理，并将上下文披露测试定位为部署前的安全检查。

## Abstract
Computer-use agents (CUAs) now act on a user's behalf across personal applications such as email, calendars, and to-do lists. This cross-application access is useful, but it also creates a privacy risk that has been largely overlooked: when an agent works in one context, it can pull in information from another that is inappropriate in that context. Hence, we introduce AgentCIBench, an evaluation harness that turns this risk into executable, deterministically scored scenarios. We target three common failure modes in CUAs: visual co-location, where the agent pulls in prohibited items that sit next to the task target in the UI; task-ambiguity overshare, where the agent dumps dense personal state in response to an under-specified prompt; and recipient misalignment, where the agent sends content to an addressee for whom it is inappropriate. We evaluate 15 frontier agents and find a surprisingly high failure rate: 11 of 15 leak on more than 50% of scenarios, with an average leakage of 67.9%, and the same failures persist when agents act end-to-end in the environment to complete the task. We release AgentCIBench to encourage the development of safer computer-use agents and position contextual disclosure testing as a pre-deployment safety check.

## 1. 论文的核心问题与整体含义（研究动机和背景）

- **核心问题**：计算机使用代理（CUA）在跨个人应用程序（如邮件、日历、待办列表）执行任务时，可能从某一上下文中拉取信息并将其泄露到另一个不适当的上下文中，造成隐私风险。现有基准仅衡量任务完成能力或对抗鲁棒性，未评估代理在正常使用中是否泄露个人状态。
- **理论框架**：采用 **上下文完整性（Contextual Integrity, CI）** 理论（Nissenbaum, 2004, 2009），将隐私定义为适当的信息流动；同一信息对某些收件人合适，对另一些则不合适。二进制保密模型无法区分“分享给家人”与“分享给同事”。
- **研究动机**：CUA已进入消费者和企业系统（如Claude Cowork, OpenAI Codex, OpenClaw），其安全不仅取决于任务完成，更取决于是否仅披露任务和收件人适当的上下文信息。现有评价体系缺失对这一失败模式的测量。

## 2. 论文提出的方法论：核心思想、关键技术细节

### 2.1 核心思想
- 构建一个可重新运行的评估框架 **AgentCIBench**，将CI风险转化为确定性可评分的场景。
- 每个场景包含：用户应用初始状态 $S_{apps}$、自然语言任务提示 $q$、收件人 $r$、必选分享集合 $V_{share}$、必须禁止泄露集合 $V_{leak}$。代理的输出 $a^*$（发送消息、保存日历事件、发布笔记）被评分：实用率 $u = [V_{share} \subseteq a^*]$，泄露率 $\ell = [V_{leak} \cap a^* \ne \emptyset]$。
- 引入**参与条件泄露率** $L_{eng} = L/(1-R)$，其中 $R$ 为拒绝率，以区分真正的选择性抑制与仅仅拒绝执行任务。

### 2.2 三种CI故障模式
| 模式 | 缩写 | 描述 | CI参数扰动 |
|------|------|------|------------|
| 视觉共位 | VCL | 代理将UI中与任务目标相邻的禁止项一同拉入输出 | 从状态中呈现的属性 |
| 任务模糊过度分享 | TAO | 在未明确指定范围时，代理默认倾倒所有可用状态 | 传输原则（under-specified prompt） |
| 收件人不匹配 | RMA | 相同内容对某些收件人合适，对另一些不合适 | 收件人角色/关系 |

### 2.3 场景生成引擎
- 使用**蒙特卡洛树搜索（MCTS）** 从手工种子出发，通过三种CI靶向变异操作（Semantic_Entanglement, Ambiguity_Trap, Identity_Bleed）生成候选场景。
- 每个候选由3个开源代理（proxy）执行，用LLM法官评分奖励 $R = u \times CI\_Violation$（$u \in \{0,1\}$, CI_Violation ∈ [1,5]），并加入新颖性奖励。仅保留高奖励、非重复场景。
- 最终池：117个场景（75个TAO, 24个RMA, 18个VCL），来自28个种子。

### 2.4 混合评分
- 确定性匹配器（标准化包含、令牌覆盖率、序列相似度） + LLM法官（捕捉同义改写）。
- 泄露计数取两者并集，且法官的主张必须有文本支持。

## 6. 论文的主要结论与发现

1. **泄露广泛**：15个代理中12个在>50%场景泄露，平均泄露率67.9%。
2. **任务完成能力不能预测泄露安全性**：高实用代理之间参与泄露率跨度达84个百分点（Claude-Opus-4.7仅14.0%，Gemini-3.1-Pro高达98.3%）。Pearson r=0.49 (p=0.06)。
3. **拒绝行为掩盖泄露**：GPT-5.4通过41.9%的拒绝率使原始泄露仅18.8%，但参与泄露升至32.4%。Kimi-K2.6也有类似现象。
4. **端到端泄露持续/加剧**：Claude-Opus-4.7参与泄露从14.0%升至42.9%；Sonnet-4.6从54.5%升至80.0%。
5. **三种防御均有效**：Restrictive、Rubric-informed、Recipient-typed将平均参与泄露降低33-36百分点，同时实用率提升16-23百分点。最强防御（Recipient-typed）将平均泄露从51.7%降至16.2%。
6. **防御在三种模式上均有效**：VCL泄露从70.6%降至19.6%（Recipient-typed），TAO从45.9%降至11.8%，RMA从60.9%降至17.2%。

## 7. 优点：方法或实验设计上的亮点

- **理论锚定**：以语境完整性理论为核心，定义了三种实际可操作的CUA失败模式，超越了二值保密模型。
- **对抗式场景生成**：MCTS自动搜索高难度场景，避免手工编写偏见，且可随模型演化重新运行。
- **混合评分机制**：确定性匹配+LLM法官，平衡精度与泛化能力，减少漏报与误报。
- **去混淆设计**：引入参与泄露率（$L_{eng}$）分离真正的选择性抑制与单纯拒绝执行，避免拒绝率对泄露指标的扭曲。
- **双重评估设置**：状态化可快速规模化，端到端验证现实UI中结果的可转移性。
- **防御实验**：三种轻量级提示干预均有效且不牺牲实用，具有实际部署价值。
- **开源与可复现**：发布完整生成管道与场景池，鼓励社区拓展。

## 8. 不足与局限

- **环境不真实**：OpenApps是六个应用的控制工作空间，非真实安装环境；绝对泄露率不应外推到现实。
- **场景设计偏差**：场景由对抗引擎生成，较随机样本更难，可能高估泄露频率。
- **端到端覆盖有限**：仅2个模型、50场景，且步数预算（20步）导致高截断率，样本量小，置信区间宽。
- **模式重叠**：三种失败模式并非互斥，真实泄露可能同时涉及多种机制，标签分配可能产生下界估计。
- **长期记忆/多轮交互未评估**：论文聚焦单次任务输出，未考虑持久记忆或多轮个性化调优。
- **防御实验有限**：仅覆盖3个模型和3种提示干预，不能推广到其他模型或非提示干预（如微调）。
- **伦理风险**：基准可能被误用于优化泄露，但作者已通过使用合成数据、仅评分指定信息单元、将框架定位为安全基准来缓解。
- **文化适应性**：CI标签基于单场景判断，可能在不同文化/组织/个人偏好下不适用。
