## 摘要
使用工具的大语言模型（LLM）智能体面临两类不同的安全失效：未经授权的外部操作，以及在最终输出检查介入之前运行环境内部敏感明文数据的泄露。现有防御通常只保护一个边界，要么是规划器/运行环境，要么是操作汇点，因此无法同时保障两个表面。我们提出 SecureClaw，一种双边界架构，在效果汇点进行授权，在读边界进行明文限制。敏感读取通过一个可信网关，将原始值替换为不透明句柄，并在评估部署中以有界摘要作为显式的降密接口。改变外部状态的写操作遵循 PREVIEW→COMMIT 协议，只有可信执行器才能提交由策略授权的确切规范请求。运行环境仍可基于摘要和符号引用进行规划，但无法直接解引用机密或执行副作用。在 AgentDojo、AgentLeak 和 Agent Security Bench（ASB）的评测中，SecureClaw 是我们通用测试框架下唯一同时保持可用任务效用并在 ASB 上实现 0% 攻击成功率、AgentDojo 上 0.64% 攻击成功率，以及在 AgentLeak 的攻击对照通道上整体泄露率仅 3.23%（该通道衡量最终输出和内部中继泄露）的防御方法。

## Abstract
Tool-using large language model (LLM) agents face two distinct security failures: unauthorized external actions and exposure of sensitive plaintext inside the runtime before any final output check can intervene. Existing defenses usually protect one boundary, either the planner/runtime or the action sink, and therefore do not by themselves secure both surfaces. We present SecureClaw, a dual-boundary architecture that places authorization at the effect sink and plaintext confinement at the read boundary. Sensitive reads pass through a trusted gateway that replaces raw values with opaque handles and, in the evaluated deployment, bounded summaries as an explicit declassification interface. Writes that change external state follow a PREVIEW$\rightarrow$COMMIT protocol in which only a trusted executor may commit the exact canonical request authorized by policy. The runtime can still plan over summaries and symbolic references, but cannot directly dereference secrets or perform side effects. Across AgentDojo, AgentLeak, and Agent Security Bench (ASB), SecureClaw is the only defense we evaluate in a common harness that simultaneously retains usable task utility and achieves 0\% attack success rate (ASR) on ASB, 0.64\% ASR on AgentDojo, and 3.23\% overall leak on AgentLeak's attacked parity lane, which measures final-output and internal-relay leakage.

### 1. 论文的核心问题与整体含义

这篇论文针对的是**使用工具的大型语言模型智能体面临的两个关键且互不包含的安全失效表面**。

*   **研究背景**：随着LLM智能体从聊天助手转向工作流自动化，它们能够读取邮件、发票等敏感数据，并执行发送消息、更新记录等外部操作。这带来了严重的安全威胁，包括间接提示注入攻击可能操纵智能体行为。
*   **两类核心安全问题**：
    1.  **效应授权**：到达外部汇点的确切请求是否经过授权？一个被攻击的运行时可能在不被允许的情况下发送邮件或转账。
    2.  **运行时明文限制**：不可信的运行时在运行过程中是否会接收到可用的敏感原始数据？即使阻止了未授权的最终操作，运行时也可能通过内部通道（如智能体间消息、共享内存）泄露机密。
*   **现有方案局限**：现有的防御方案通常只保护上述两个边界中的一个（如仅过滤规划器，或仅在操作端点进行权限控制），无法同时保障两个表面的安全。
*   **论文核心主张**：**智能体运行时不应控制外部效应何时发生，也不应控制原始受保护值驻留何处。** 必须构建一个双边界架构，从系统层面分离效应控制和明文控制。

### 2. 方法论：SecureClaw 架构设计

SecureClaw 的核心思想是建立一个**双边界信任架构**，将安全执行从不可信的LLM运行时中剥离，转移到一组可信组件中。

**核心思想与架构组件**：
*   **架构角色划分**：
    *   **不可信的智能体运行时**：继续负责规划、搜索和编排工具调用，但它不可信，无权直接接触明文或提交外部操作。
    *   **可信网关**：位于读路径，负责分类返回数据，将原始敏感值替换为不透明句柄和有界摘要。
    *   **可信句柄存储**：在运行时内存之外保留受保护的明文及其元数据。
    *   **可信策略引擎**：对规范化的请求进行授权或拒绝，并返回绑定到请求摘要的授权凭证。
    *   **可信执行器**：位于写路径终点，是唯一能接触效应端汇点的组件，负责在提交前进行最终验证。

**关键机制流程**：
*   **读路径——不透明句柄限制 (Opaque-Handle Confinement)**：
    *   当工具返回敏感内容时，可信网关将原始明文存储到句柄表中，并仅向运行时返回一个**高熵的、符号化的不透明句柄**。
    *   同时，为了支持规划，网关会返回一个**有界摘要**，该摘要被视作一种**显式的降密接口**，而非原始数据的泄露。
    *   运行时可以在后续计算中传递句柄，但**无法将其解析为明文**。这切断了通过智能体间消息和内存等内部通道泄露数据的路径。
*   **写路径——PREVIEW → COMMIT（预览-提交）协议**：
    *   运行时只能**提议**一个效应操作（包括操作意图和输入参数）。
    *   可信网关会附加上经认证的调用者、会话等上下文信息，形成完整请求 `ρ`。
    *   网关通过 `Canon(ρ)` 函数对请求进行**确定性规范化**，并计算绑定摘要 `h ← HMAC-SHA256(k_bind, Canon(ρ))`。
    *   策略引擎根据 `h` 返回绑定的授权决策凭证。
    *   可信执行器在提交前会**独立重新计算 `h`**，验证授权凭证的完整性、新鲜度，并检查重放状态和句柄解析权限。**只有通过所有验证，执行器才会解引用句柄并提交外部效应。**
*   **拒绝感知的恢复**：当执行器拒绝提交时，不会直接终止工作流，而是根据固定模板返回安全提示，引导运行时进行下一步（如从“发送邮件”切换到“起草待确认”）。

### 6. 论文的主要结论与发现

*   **双边界保护是必要的**：实验证明，仅靠执行边界（如Faramesh）无法阻止内部明文泄露，仅靠运行时防御（如IPIGuard, DRIFT）无法阻止未授权提交。SecureClaw通过**将效应授权放在汇点，将明文限制放在读边界**，是唯一同时保护两个表面的方案。
*   **前所未有的安全性**：SecureClaw是唯一一个在三个基准上同时实现极低攻击成功率（ASB: 0%，AgentDojo: 0.64%）和极低内部泄露率（AgentLeak内部中继泄露近乎为零）的防御方案，且这一成果是在攻破运行时的强威胁模型下取得的。
*   **安全与可用性的平衡**：通过拒绝感知的恢复机制，SecureClaw在被攻击状态下仍保持了较高的任务可用性（ASB: 88.90%），证明了“默认拒绝”的安全设计不一定导致工作流崩溃。
*   **残余风险可解释**：剩下的极少数攻击成功案例并非核心机制被绕过，而是因为策略授权粒度过粗或降密接口的自然语言摘要泄露，这为未来的研究方向指明了道路。

### 7. 优点

*   **清晰的安全分解**：论文提出的双表面安全分析框架（效应表面 vs. 明文表面）逻辑清晰，深刻揭示了现有防御的根本性局限。
*   **系统级架构设计**：SecureClaw借鉴了操作系统/能力系统的原则，将安全执行从不可信的应用（LLM运行时）转移到可信的计算基上，设计优雅且原理性强。
*   **极为扎实的实验评估**：实验设计全面、严谨，不仅在多基准上与基线对比，还通过消融实验、对抗性压力测试（绕过套件）和残余分析，从多角度充分验证了核心主张。

### 8. 不足与局限

*   **良性任务效用损失**：在AgentDojo上，SecureClaw的良性任务效用（60.82%）相较于无防御方案（70.10%）有明显下降。这源于其严格的安全控制，论文将此归因于策略粒度不足。
*   **对策略引擎的依赖**：安全性极大地依赖于策略引擎的配置和正确性。如果策略过于宽松，仍可能允许“错误”但“授权”的操作发生（残余风险）。
*   **明文处于降密接口的风险**：虽然内部通道泄露被阻断，但通过有界摘要这一“显式降密通道”仍可能泄露足以区分秘密的信息（Theorem 3），这是有意为之的设计妥协，但在高机密场景下构成风险。
*   **信任根假设**：架构假设可信网关、句柄存储、策略引擎和执行器不被攻破。若这些核心组件被攻破，安全保证将完全失效（Table 23）。
*   **实现与部署复杂性**：引入多个可信组件增加了系统复杂性和潜在的性能开销（单服务约15ms），虽然被论证为可接受，但对部署和运维有更高要求。
