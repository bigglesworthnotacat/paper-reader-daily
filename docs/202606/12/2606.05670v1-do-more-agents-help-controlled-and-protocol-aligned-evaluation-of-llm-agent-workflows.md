## 摘要
当比较的系统共享相同的基准加载器、工具访问权限、答案契约、用量统计和轨迹记录时，增加更多智能体对LLM工作流有帮助吗？我们引入了BenchAgent，一个评估框架，将单智能体、固定多智能体（MAS）和演化MAS工作流置于一个归一化的执行和记录协议下。BenchAgent使用GPT-4.1在十个推理、编码和工具使用基准上评估这些基底内部工作流，并单独报告了一项运行时生成工作流的协议对齐外部（PAE）GAIA研究。在SI条件下，六个受测MAS中至多有一个超过匹配的单智能体锚点在基准平衡平均准确率上：EvoAgent位于Wilson单次运行指导范围内，而其余五个落后2.56-11.29个百分点，并且占据更昂贵的准确率-成本权衡点。在PAE GAIA快照中，一个Claude-Code风格的运行时工作流总体达到66.72%，在第3级上达到69.23%，比最强的非Claude基线Jarvis（一个固定MAS）高出20个百分点以上。

## Abstract
Does adding more agents help an LLM workflow once compared systems share the same benchmark loader, tool access, answer contract, usage accounting, and trajectory logging? We introduce BenchAgent, an evaluation framework that places single-agent, fixed multi-agent (MAS), and evolving MAS workflows under one normalized execution and logging protocol. BenchAgent evaluates these substrate-internal workflows across ten reasoning, coding, and tool-use benchmarks with GPT-4.1, and separately reports a Protocol-Aligned External (PAE) GAIA study of a runtime-generated workflow. Under SI conditions, at most one of six tested MAS exceeds the matched single-agent anchor on benchmark-balanced average accuracy: EvoAgent lies within the Wilson one-run guidance, while the remaining five trail by 2.56-11.29 points and occupy more expensive accuracy-cost trade-offs. On the PAE GAIA snapshot, a Claude-Code-style runtime workflow reaches 66.72% overall and 69.23% on Level 3, more than 20 points above the strongest non-Claude baseline, Jarvis, a fixed MAS.

## 1. 论文的核心问题与整体含义

本研究聚焦于一个根本性问题：**在 LLM 智能体工作流中，增加更多智能体是否真的能带来性能提升？** 当前该领域的评估普遍存在协议不一致的问题，不同系统在基准加载、工具访问、答案格式、用量统计和轨迹记录等方面各有差异，导致观察到的准确率差距往往混杂了工作流组织本身的贡献与这些协议层面的优势，难以公平归因。论文的核心意图是**在可控、协议对齐的条件下分别考察固定多智能体系统（MAS）、演化 MAS 以及运行时动态生成的工作流，揭示“智能体数量或角色”本身能否独立产生可靠的工作流增益**，从而为智能体架构设计提供更科学的评判标准。

## 2. 方法论

- **BenchAgent 评估框架**：提供一个统一的执行“衬底”，将单智能体、固定 MAS、演化 MAS 置于相同的基准加载、工具注册、答案契约、用量统计和轨迹记录接口下，使得跨范式的比较仅反映工作流组织的差异。
- **工作流提升（Workflow lift）定义**：在保持基座模型、基准、工具、评估器等不变的前提下，将单智能体工作流替换为多智能体工作流所带来的准确率与成本变化。单智能体锚点（BenchAgent Core）不是最强可能的单控制器，而是一个共享执行基线的参照点。
- **两类比较协议**：
  - **衬底内部比较（SI）**：工作流运行在 BenchAgent 内部，暴露全部消息、工具调用、用量和终止状态，真正隔离工作流结构对性能的影响。
  - **协议对齐外部比较（PAE）**：当系统（如 Claude-Code 风格运行时工作流）无法被重新实现在衬底内部时，对齐输入、评估器、输出格式、工具能力类、后端模型族和权限配置，但保留控制器细节的透明性。
- **Wilson 置信区间指导**：由于实验为单次 pass@1 运行，使用 Wilson 95% 二项置信区间半宽作为可解读差距的下界。差距小于该半宽度的结果被视为描述性的，而非稳定的排序证据。
- **工作流类别定义**：文中将工作流明确划分为单智能体、固定 MAS（角色和拓扑预先设定）、演化 MAS（在预设工作流家族中搜索/变异）和运行时生成工作流（在执行过程中动态创建任务专用智能体、分配上下文、定义工具作用域和验证/恢复分支）。

## 3. 实验设计

### 3.1 数据集与基准
- **广泛的衬底内部基准**（十个）：MATH、AIME、GSM8K、DROP、BBH、MMLU-Pro、HumanEval、MBPP、HotpotQA、IFEval。对于超过 400 样本的分集，随机固定种子抽样 400 例，其余使用完整分集（如 AIME 30 例、HumanEval 131 例、MBPP 341 例）。
- **GAIA 验证集**：用于运行时工作流研究，共 165 个样本（Level 1: 53, Level 2: 86, Level 3: 26），考察多步检索、文件检查、状态保持和最终答案延迟发布等复杂工具使用任务。

### 3.2 工具分配
- 除 HotpotQA 和 GAIA 使用 BenchAgent 完整工具注册表外，其余基准仅暴露 Python 解释器（加最终答案发射接口）。
- CC-workflow 在 GAIA 上匹配了同类的工具能力类（网络搜索、文件读写、Shell 执行等），但工具实现细节随外部系统而异，属于 PAE 对齐范畴。

### 3.3 比较方法
- **单智能体锚点**：BenchAgent Core，在衬底内部运行。
- **固定 MAS**：Jarvis（HuggingGPT 风格）、LLM-Debate（辩论式）、AutoGen、CAMEL（角色交流）、ChatEval（多评判者）。
- **演化 MAS**：EvoAgent（进化搜索提示脚手架）。
- **运行时生成工作流**：Claude-Code 风格（CC-workflow），仅参与 GAIA 的 PAE 对比，其内部通过动态创建 Planner/Solver/Verifier/Writer 子智能体、派遣任务、持久化证据、实施验证门控等方式进行。

### 3.4 模型与超参
- 全部主实验使用 GPT-4.1（`gpt-4.1-2025-04-14`），温度 0.2，top_p 1.0，最大 token 8192。
- 额外的后端泛化检查使用了 Qwen3-32B（BenchAgent 内部 GAIA 对比）和 GLM-5（BenchAgent 单智能体锚点校准），以判别主要工作流趋势是否依赖于特定基座模型。

## 6. 主要结论与发现

- **多智能体并非简单地在数量或固定协调上必然胜出**：在 SI 条件下，六种受测固定和演化 MAS 中至多只有 EvoAgent 在基准平衡平均准确率上以极小优势（+1.44 百分点）略高于单智能体锚点，且该优势处于 Wilson 不确定性尺度内；其余五个 MAS 准确率落后 2.56–11.29 个百分点，同时成本（token、时间）更高，占用的精度-成本权衡点更差。
- **工作流结构与任务的匹配程度决定增益，而非智能体数量**：LLM-Debate 的辩论式聚合在可独立验证提议的任务（HumanEval、MATH）上获益；EvoAgent 的进化搜索在需要多样推理模式的任务（BBH）上产生显著提升，但在计算密集型任务上（AIME）表现不佳；ChatEval 在指令遵循任务（IFEval）上最强，却以大量 token 消耗为代价，而在其他基准上落后于基线。即 MAS 的协议当且仅当其匹配任务的错误模式时才有助于提升，否则反而引入开销和信息损失。
- **运行时生成的工作流在复杂长程工具使用上展现出大幅优势**：CC-workflow 在 GAIA 上达到 66.72% 总体准确率，较最优非 Claude 基线（Jarvis）高出 20.06 个百分点；在最具挑战性的 Level 3 上优势扩大到 42.31 个百分点。且保留日志显示其消耗的可见 token 数和墙钟时间反而更少。
- **两种失败模式的结构性差异**：固定和演化 MAS 的失败常因线性交接中丢失约束而将错误“固化”；CC-workflow 的失败更倾向于局部的工具或状态问题，可被工具重试、证据重读或验证器纠正。
- **评估协议统一至关重要**：分解准确率、轨迹和成本后，发现相似准确率的系统在 token 用量、延迟和工作流拓扑上可能完全不同，只有对齐协议才能避免错误归因。

## 7. 优点

- **开创性评估框架**：BenchAgent 首次为多范式 LLM 智能体工作流提供了统一的执行与记录衬底，实现了真正的“公平比较基础”，而非引入新数据集。
- **概念清晰的方法论**：明确提出“工作流提升”作为测量目标，定义了衬底内部和协议对齐外部两种比较模式，并利用 Wilson 区间对单次运行结果的解读给出保守指导，严谨可靠。
- **全面的对比分析**：涵盖了从固定、演化到运行时生成的多种工作流类别，在 10 个多样化基准和 1 个高阶工具使用基准上进行评估，使结论具有较强的覆盖面。
- **过程证据的深度挖掘**：不仅报告最终准确率，还提供 token 消耗、墙钟时间、工具调用分布、代理数量、委派深度、失败步骤位置等丰富的轨迹统计，帮助解析工作流行为，避免“唯分数论”。
- **透明且可复现**：详细公开了提示契约、系统角色、工具配置和轨迹片段，并对各种检查（后端泛化、控制器校准、转移合理性）进行了补充实验，增强了研究的可审计性。

## 8. 不足与局限

- **非因果推断**：尽管 SI 比较控制了执行衬底，但各 MAS 的再实现可能引入细微差异；PAE 下的 CC-workflow 优势无法归因于单一机制（如动态委派、上下文管理、验证门控等），需要进一步的消融实验才能分离具体设计的影响。
- **单次运行的统计限制**：所有主实验均为 pass@1 一次运行，未提供多次重复后的平均和方差，因此结论偏向描述性，不能严格支撑排序。
- **CC-workflow 内部不完全透明**：Claude Code 的供应商侧上下文压缩、重试策略、缓存等内部行为无法通过保留日志观测，可能导致成本记录不完全准确。
- **工具表面并非完全一致**：PAE 中仅对齐了工具能力类，而非逐工具匹配，可能残留工具实现的差异。
- **未包含更多后端模型**：虽然进行了少量后端泛化检查（Qwen、GLM），但主要结论基于 GPT-4.1，进一步验证不同模型族下的鲁棒性将增强研究的普遍性。
- **应用范围**：研究集中在推理、编码、工具使用基准和 GAIA，未覆盖开放域对话、多模态交互或真实世界部署中的其他挑战（如安全、延迟约束、可维护性等）。
