## 摘要
数据代理将基于LLM的推理与关系数据访问、可执行分析工具及多步骤工作流编排相集成，使其在企业分析中日益核心。这种集成在数据资源、数据库执行和代理推理方面引入了新的安全脆弱性，将数据库安全与通用LLM代理安全中的关切重新组合成两类工作均无法单独捕获的失效模式。为弥补这一空白，我们对数据代理进行了系统的安全研究，主要贡献有三。首先，我们构建了一个分层脆弱性框架，识别出解释层、执行层和策略层共八种数据代理特有风险。其次，我们引入一种按对手目标、战术和技术组织的攻击分类法，涵盖三种目标、七种战术和十四种技术，并配以基于真实数据库模式的LLM驱动载荷生成流水线。第三，我们在六个系统上评估了这些攻击，包括四个开源数据代理和两个生产级云分析服务。实验揭示了当前系统中存在大量安全脆弱性，并得出四项关键发现。

## Abstract
Data agents integrate LLM-driven reasoning with relational data access, executable analytical tools, and multi-step workflow orchestration, making them increasingly central to enterprise analytics. This integration introduces new security vulnerabilities across data resources, database execution, and agent reasoning, recombining concerns from database security and general-purpose LLM-agent security into failure modes that neither line of work captures on its own. To address this gap, we present a systematic security study of data agents. Our contributions are threefold. First, we develop a layered vulnerability framework that identifies eight data agent-specific risks across interpretation, execution, and policy layers. Second, we introduce an attack taxonomy organized by adversary goal, tactic, and technique, covering three goals, seven tactics, and fourteen techniques, and pair it with an LLM-driven payload generation pipeline grounded in real database schemas. Third, we evaluate these attacks on six systems, including four open-source data agents and two production cloud analytics services. Our experiments reveal substantial security vulnerabilities across current systems and yield four key takeaways.

### 2. 方法论
- **核心思想**：将数据代理抽象为`Σ = ⟨(M,S), T, D, Π, E⟩`组成的系统模型，沿**解释层** → **执行层** → **策略层**的管道化视角识别脆弱性，并将其映射为可执行的攻击技术。
- **脆弱性发现**：基于对数据资源、数据库执行与代理推理的交叉分析，提取出8种特有脆弱性（V1–V8），例如：
  - 解释层：V1 (隐性信任偏差)、V2 (缺乏数据源验证)
  - 执行层：V3 (不受控查询成本)、V4 (跨引擎语义不一致)、V5 (无界多步查询链)
  - 策略层：V6 (安全策略遗忘)、V7 (过度特权数据库连接)、V8 (缺乏组合泄露控制)
- **攻击分类法**：按**目标-战术-技术**三层组织：
  - 三大对手目标：**劫持 (Hijack)**、**误导 (Mislead)**、**耗尽 (Drain)**
  - 7种战术、14种具体攻击技术（如T1.1查询注入、T5.1数据类型偏差利用、T7.1低价值分支扩展）
- **载荷生成流水线**：采用检索增强生成（RAG）方法。先检索攻击技术描述，用LLM生成机制上不同的攻击模板，再结合真实数据库的模式信息（表名、列名、类型），实例化为25个/技术、总计350个模式特异的攻击载荷，确保覆盖多领域变体。

### 3. 实验设计
- **数据集/场景**：采用**DAComp-DA**基准数据集，包含100个独立SQLite数据库实例，覆盖金融、电商、数字营销等企业领域。每个实例原本是良性分析工作负载，通过在可写字段植入载荷或构造多轮对话序列转换为安全测试。
- **评估对象**：
  - 4个开源数据代理：DataInterpreter、DB-GPT、DeepAnalyze、LAMBDA（主要使用DeepSeek-V3.2作为基础LLM，DeepAnalyze自带蒸馏模型）
  - 2个闭源商用服务：Databricks Genie Code、BigQuery Conversational Analytical Agent
- **对比方法**：不涉及方法对比，而是横向比较不同系统针对相同攻击向量的**攻击成功率（ASR）**。
- **评估指标**：
  - **ASR**：基于阈值判断攻击是否成功（劫持：BR≥1；误导：RE≤10%；耗尽：整体RAR≥5）。
  - **爆炸半径 (BR)**：劫持攻击造成的未授权影响传播层级（0~4级）。
  - **相对误差 (RE)**：误导攻击输出与攻击者目标值的偏差百分比。
  - **资源放大比 (RAR)**：耗尽攻击相对于良性工作负载的资源（令牌数、执行时间）放大倍数。

### 6. 主要结论与发现
- **普遍暴露**：每一个受测系统都可被利用至少5种脆弱性，没有系统能阻止全部攻击。商用服务对劫持攻击防护较强，但仍显著暴露于误导和资源耗尽技术。
- **脆弱性不具系统专一性**：8种脆弱性均至少在3个不同系统中出现，表明它们是数据代理管道固有的安全缺陷，而非单一实现引入的bug。
- **主要风险**：
  - **资源耗尽 (Drain)** 是首要失效模式，攻击者无需违反显式策略，仅通过诱导看似合理的分析工作即可耗尽计算资源。
  - **信任偏差主导**：解释层隐性信任偏差（V1）是曝光度最高的脆弱性，代理在面对冲突证据时经常陷入攻击者设定的偏好。
  - **基于推理的数据泄露**：多步分析推理使得通过组合单个合法的查询来重构敏感记录成为可能，需要会话级泄露控制。
- **关键防御启示**：衍生出四项设计原则（Takeaway），包括：将数据库连接推理视为新攻击面、将资源耗尽作为主安全风险、追踪累积泄露、用显式出处而非隐式偏好解决证据冲突。

### 7. 优点
- **问题独创性**：首次将数据代理安全作为独立于数据库安全和通用Agent安全的交叉领域进行系统研究，填补空白。
- **方法论完备**：给出了从形式化系统模型、脆弱性框架、攻击分类法到自动化载荷生成的端到端研究管线，结构清晰，逻辑严密。
- **实验设计严谨**：评测系统覆盖面广（含开源与闭源商业系统），指标定义明确（BR/RE/RAR/ASR），实验可复现性好，考量化测试次数庞大（3075次），并包含对阈值和基础LLM的敏感度分析，结论稳健。

### 8. 不足与局限
- **数据库类型与规模局限**：实验仅基于SQLite进行，可能无法完全反映分布式、大规模生产环境中特有的执行层脆弱性（如复杂优化器、分布式算子）。
- **攻击场景预设**：威胁模型假设攻击者可写入数据或上传文件，未考虑外部API、第三方组件等更广泛的横向扩展攻击面。
- **缺乏防御方案验证**：作为安全分析论文，提出了防御原则，但未设计或评估任何相应的防御机制。
- **商用系统评估受限**：由于黑盒访问，对商用服务仅能评估 ASR，无法量化 BR、RE 和 RAR 等影响程度指标，削弱了对比深度。
- **时间的敏感性**：评估的时间点是2026年5月，商用系统自身的更新迭代可能使具体攻击的时效性变化较快。
