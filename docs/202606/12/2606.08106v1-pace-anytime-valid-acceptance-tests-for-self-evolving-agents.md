## 摘要
自演化代理通过反复对其自身的提示、技能或工作流提出更改，并保留那些在小型留出集上得分更高的更改来实现改进。几乎所有的努力都集中在生成候选的提议器上；我们认为薄弱环节在于接受器，即决定是否提交更改的规则。在相同的噪声开发估计上应用数百次，普遍采用的“只要分数上升就保留”规则是一种未受控制的自适应多重检验：代理实际上在自我p-hacking，积累错误提交，导致其剧烈波动和漂移而非改进。我们将提交重新定义为序贯假设检验，并提出PACE（配对随时有效的提交评价），一种无需训练、随时有效的提交闸门。每个候选与现有方案在相同实例上比较，仅当基于博弈的检测e过程积累决定性证据时才提交，提前停止以节省评估，并控制每个候选的错误提交概率在用户设定的水平，即使在可选停止下也是如此（每次决策保证）。在Qwen2.5代理（0.5B-3B）于GSM8K、SVAMP和ARC-Challenge上进行提示级自演化的实验中，当噪音提议中隐藏着一个真正的改进时，贪心接受提交了30-42%的错误编辑和10-33%的有害编辑，而PACE提交了真正的改进且基本上没有其他，以明显更低的方差实现了与贪心相同的留出集准确率，评估成本降低约18%。当没有真正的收益可用时，贪心每次运行提交13-21个虚假自我修改（72-100%错误），并使最脆弱的代理性能下降4.9点，而PACE保持在基线。自演化的可靠性取决于接受器，而不仅仅是提议器。

## Abstract
Self-evolving agents improve by repeatedly proposing changes to their own prompts, skills, or workflows and keeping those that score higher on a small held-out set. Almost all effort has gone into the proposer that generates candidates; we argue the weak point is the acceptor, the rule that decides whether to commit a change. Applied hundreds of times against the same noisy dev estimate, the ubiquitous "keep it if the score went up" rule is uncontrolled adaptive multiple testing: the agent effectively p-hacks itself, accumulating false commits that make it churn and drift rather than improve.   We recast committing as a sequential hypothesis test and propose PACE (Paired Anytime-valid Commit Evaluation), a training-free, anytime-valid commit gate. Each candidate is compared to the incumbent on identical instances and committed only when a testing-by-betting e-process accumulates decisive evidence, stopping early to save evaluations and controlling each candidate's false-commit probability at a user-set level even under optional stopping (a per-decision guarantee).   On Qwen2.5 agents (0.5B-3B) self-evolving at the prompt level on GSM8K, SVAMP, and ARC-Challenge, greedy acceptance commits 30-42% false and 10-33% harmful edits when a genuine improvement is hidden among noisy proposals, while PACE commits the real one and essentially nothing else, matching greedy's held-out accuracy at sharply lower variance and about 18% lower evaluation cost. With no real gain available, greedy commits 13-21 spurious self-modifications per run (72-100% false) and degrades the most fragile agent by 4.9 points, while PACE holds at baseline. Reliability of self-evolution depends on the acceptor, not only on the proposer.

## 1. 论文的核心问题与整体含义
自演化代理（self-evolving agents）通过重复提议对自身提示、技能或工作流的修改，并保留在一个小型留出集上得分更高的修改来实现自我改进。目前几乎所有研究精力都集中在**提议器**的设计上，而**接受器**（即决定是否提交某一修改的规则）却普遍采用一条未经审视的启发式规则：“只要留出集评估得分上升就提交”（贪心接受）。  
作者指出这一做法存在根本性的统计缺陷：在大量回合中重复使用同一个带噪声的开发集进行数百次比较，相当于**未受控制的、自适应的多重假设检验**。代理实际上在“p-hacking”自己，积累了大量虚假提交（false commits），导致系统反复修改却没有真正进步，甚至出现性能漂移和退化。  
因此，核心问题是：**自演化代理的接受决策必须得到统计控制**，可靠性不仅取决于提议器，更取决于接受器。论文提出将提交决策重新建模为序贯假设检验，并设计了一个符合随时有效（anytime-valid）原则的接受闸门。

## 2. 方法论：PACE（配对随时有效的提交评价）

### 核心思想
- 将每个候选修改与当前配置进行**配对比较**（McNemar 风格）：在同一批实例上评估两者，仅关注出现分歧的配对（候选正确而当前配置错误，记为 `w=1`；反之 `w=0`），平局则丢弃。
- 将检验视为一场**下注游戏（testing-by-betting）**：从初始财富 `E0=1` 开始，对每一个分歧对下注 `λ`。若候选不比当前配置好（零假设 H0），分歧对中双方获胜概率相等，财富期望保持不变（公平博彩）；若候选确实更优，则财富有望增长。
- 使用 **e过程** 积累财富，一旦财富 `E ≥ 1/α`，就拒绝零假设，提交候选修改。该过程**随时有效**：即使中途停止、根据观察结果决定何时停止，对单个候选的虚假提交概率始终控制在预设水平 α 以下。

### 财富更新公式
```
E ← E · (1 + λ(2w − 1)),  λ ∈ [0,1)
```
其中 `w` 为当前分歧对的输赢标志（1 表示候选赢）。零假设下 `E` 是非负上鞅，满足 Ville 不等式：`Pr[sup_t E_t ≥ 1/α] ≤ α`。

### 算法流程（Algorithm 1 概括）
1. 输入代理配置、开发集 D、显著性水平 α、下注比例 λ。
2. 每一轮自演化：
   - 提案器生成候选 `c'`。
   - 初始化财富 `E=1`。
   - 逐一从 D 中抽取实例，评估 `c` 和 `c'`，若平局则跳过。
   - 根据输赢更新 `E`。
   - 若 `E ≥ 1/α`，则提交 `c'` 并退出评估循环；若评估预算耗尽仍未达阈值，则拒绝候选。

**关键属性**：该门控是**随时有效**的，可提前停止评估以节省算力；**无需训练**；仅消耗配对的正确性结果，不依赖提议器内部信息，因此对任何自修改循环通用。

## 3. 实验设计

### 使用场景与基准
- **代理**：Qwen2.5-Instruct（0.5B, 1.5B, 3B），通过可编辑的系统提示进行自演化。
- **任务**：
  - GSM8K：小学数学应用题（精准匹配答案）。

  - ARC-Challenge：多项选择科学题（非数学领域）。
- **实验体制**：
  1. **受控体制**：植入一个有害指令，提案器能够发现并移除它，从而获得一个**已知的真实改进**。用于检验接受规则能否保留真实改进并拒绝噪声。
  2. **随机体制**：代理在温度 0.7 下采样（模拟真实部署的随机性），没有真正的大改进存在。用于检验接受规则是否会追逐采样噪声。

### 比较方法
- **贪心接受**（Greedy）：当前留出集得分上升即提交（主流做法）。
- **固定 n 配对检验**（Paired (n)）：在固定数量的配对后执行一次 McNemar 检验。
- **在线 FDR**（Online-FDR）：一种考虑多重检验修正的在线方法。
- **PACE**：本文提出的随时有效门控。

## 6. 主要结论与发现
1. **贪心接受的诊断成立**：在受控体制下，贪心提交 30–42% 错误修改和 10–33% 有害修改；在随机体制下，每次运行提交 13–21 个虚假修改（72–100% 虚假），且可能导致脆弱的代理准确度下降（0.5B 模型下降 4.9 个点）。
2. **PACE 有效抑制虚假提交**：在所有实验中，审计标记的虚假提交率基本为 0，仅提交真正改进，且评估成本较贪心低约 18%。
3. **可靠性提升**：PACE 在保持真正改进的同时，显著降低了准确率的方差（尤其是对 3B 模型），且不牺牲平均增益。
4. **通用性**：上述模式在 arithmetic（GSM8K、SVAMP）和 non‑math（ARC‑Challenge）任务上均成立，表明问题根源在于接受规则本身，而非特定任务。
5. **自演化的可靠性取决于接受器**：即使拥有强大的提议器，若接受决策无统计控制，整个自演化循环仍会崩溃。

## 8. 不足与局限
- **实证范围局限**：只在提示级自演化上验证，虽声称接口可推广至技能、代码、多代理拓扑等更复杂的自演化系统，但未提供直接证据。
- **模型和提议器单一**：仅使用 Qwen2.5 一个模型家族和一种提议器，通用性尚待更广验证。
- **审计非真值**：虚假提交率基于温度 0 的审计池估计，并非绝对真实准确度。
- **统计保证边界**：控制的是**每候选**的虚假提交概率，而非整个运行级别的族系错误率（但作者论证这已经足以解决贪心引入的复合效应）。

- **开发集覆盖局限性**：门控只能验证开发集所覆盖的改进，无法保证分布外提升。
