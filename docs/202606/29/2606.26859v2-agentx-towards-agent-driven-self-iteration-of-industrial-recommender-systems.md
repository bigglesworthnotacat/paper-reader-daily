## 摘要
推荐算法的迭代正从一种依赖工程师的手工流程，转向工业化的研究循环，但这一转变仍受到结构性执行瓶颈的阻碍：从想法到上线的周期仍然依赖人类工程师生成假设、修改生产代码、启动A/B实验并归因线上结果。因此，创新与人力线性增长，而非与证据、算力和积累的实验知识复合增长。我们提出了AgentX，一个在生产环境中部署的多智能体系统，它从根本上重构了这一生产函数。AgentX作为一个自我进化的开发引擎运行：它自主生成、实施、评估并从推荐实验中学习，其规模和速度是任何手动工作流都无法维持的。

该系统在一个闭环中协调四个紧密耦合的阶段。头脑风暴智能体综合来自历史实验、系统架构、数据分析和外部研究的证据，生成排名后的可执行方案。开发智能体通过基于仓库的生成和多维可靠性验证，将每个方案转化为生产就绪的代码。评估智能体通过带有护栏否决的A/B判断进行安全在线部署，将成功和失败都转化为结构化的知识资产。然后，一个进化层（SGPO）将执行轨迹提炼为语义梯度更新，持续优化智能体本身——使系统不仅自动化，而且自我改进。

## Abstract
Recommendation algorithm iteration is moving from an artisanal, engineer-bound process toward an industrialized research loop, but this transition remains blocked by a structural execution bottleneck: the idea-to-launch cycle still depends on human engineers to generate hypotheses, modify production code, launch A/B experiments, and attribute online results. Innovation therefore scales linearly with headcount rather than compounding with evidence, compute, and accumulated experimental knowledge. We present AgentX, a production-deployed multi-agent system that fundamentally restructures this production function. AgentX operates as a self-evolving development engine: it autonomously generates, implements, evaluates, and learns from recommendation experiments at a scale and pace that no manual workflow can sustain.   The system orchestrates four tightly coupled stages in a closed loop. A Brainstorm Agent synthesizes evidence from historical experiments, system architecture, data analysis, and external research into ranked, executable proposals. A Developing Agent translates each proposal into production-ready code through repository-grounded generation and multi-dimensional reliability verification. An Evaluation Agent conducts safe online rollout with guardrail-vetoed A/B judgment, converting both successes and failures into structured knowledge assets. A Harness Evolution layer (SGPO) then distills execution trajectories into semantic-gradient updates that continuously sharpen the agents themselves -- making the system not merely automated, but self-improving.

## 2. 方法论：核心思想、关键技术细节、公式或算法流程

### 2.1 核心思想
- **闭环自迭代**：将推荐迭代分解为四个紧密耦合的阶段（头脑风暴、开发、评估、进化），形成一个从想法生成到上线验证再到智能体自身改进的完整闭环。
- **真实线上奖励**：以线上A/B测试的反馈作为权威奖励信号，而非离线指标或人类评分。
- **语义梯度提示优化（SGPO）**：将执行轨迹转化为自然语言诊断，作为“语义梯度”来更新子智能体的提示、验证规则和输出合约，并通过配对回放（paired replay）准入，确保安全进化。

### 2.2 关键技术细节

#### (a) 头脑风暴智能体（Brainstorm Agent）
- **输入**：模糊的用户意图（如“提升用户时长”）。
- **工作流程**：
  1. **边界设定**：将意图规范化为结构化任务边界，包括目标、允许的变更范围、禁止项等。
  2. **有界方案探索**：分批次生成方案，每个方案标注成熟度（立即实施、需先探测、长期备选）；使用“避免集”记录已拒绝或重复的方向。
  3. **证据加权生成**：从四个知识源（实验知识库、系统知识库、数据分析、模型研究）中按权重检索证据，评估方案质量。最终得分公式：
     ```
     S(c|q) = λ_o·O + λ_b·B + λ_f·F + λ_h·H + λ_e·E - λ_r·R
     ```
     其中O为目标对齐度，B为业务语义有效性，F为实施可行性，H为交接完整性，E为加权证据得分，R为风险惩罚。
  4. **验证与交接**：通过人工审查和生产检查（A/B参数检查、知识库一致性等），将通过的想法转化为结构化的实验记录和交接计划。

#### (b) 开发智能体（Developing Agent）
- **两条并行轨道**：
  - **在线策略轨道**（特征/策略级变更）：实现为生产代码变更。
  - **离线模型轨道**（模型架构变更）：实现为训练实验，要求可证伪的因果链归属。
- **可靠性保证**：
  - **基于仓库的代码生成**：通过项目特定知识库和案例工具箱（schema查询工具、DSL检查器、C++语法检查器等）防止属性幻觉、DSL误用等。
  - **验证导向的实现循环**：每轮实施后运行准确性循环和dryrun管道，通过八个维度的质量评分（表1）衡量可靠性，其中属性幻觉、正确性循环次数、人工干预占比最高。
- **模型开发专用机制**：
  - 每轮声明**政策（policy）**：包括声称的因果机制和预期可观察量。
  - 代码实现后由**验证器**检查git diff语义匹配，并由**专家智能体群**（物理隔离）投票（超三分之二多数）达成共识。
  - 指标提取由确定性Python状态机完成（无LLM参与）。
  - **可证伪归属**：逐链判定因果链的每个环节（已验证/断裂/不明），只有全链“已验证”才记录增益；否则即使AUC提升也弃置。
  - **鲁棒执行**：对平台故障（OOM、作业挂起等）按原因代码分类处理（确定性错误直接放弃、瞬态故障重试一次、网关故障切换）。

#### (c) 评估智能体（Evaluation Agent）
- **安全部署**：通过OpenAPI映射实验到正确的业务域、世界、流量桶；执行金丝雀路径（灰度窗口）；参数变更需通过工程白名单。
- **A/B判断与护栏否决**：
  - 使用CUPED、双重差分等方差缩减方法，辅以缺失数据处理。
  - 护栏设计三原则：
    1. **业务域范围护栏**：每个业务域使用自己的核心指标和否决阈值。
    2. **复合经济交换指标**：使用聚合生命周期价值（LT）得分，而非单一指标硬阻断。
    3. **阈值作为关注信号**：严重下降硬阻断，中等程度升级为人工审查，轻微仅监控。
  - 最终输出结构化裁决：KEEP / EXTEND / DISCARD，附带效果、护栏状态、统计方法等。
- **负面结果资产化**：将失败的实验（DISCARD或不明确）记录下来，包含根因、阶段、业务目标等，供未来头脑风暴检索，避免重复错误。

#### (d) 进化层（SGPO）
- **SGPO-I（基于轨迹的提示优化）**：
  1. 从积累的轨迹池中采样与目标子智能体相关的轨迹。
  2. 评估器写出自然语言损失报告并浓缩为**语义梯度**（g_{t,i} = 缺失约束、错误步骤顺序、证据要求不足等）。
  3. 精炼智能体根据梯度生成候选修订提示（仅限该子智能体的指令、验证规则等）。
  4. 通过**配对回放**：将旧/新提示在相同回放任务上运行，比较得分ΔJ_i。只有当ΔJ_i > ε且安全检查通过时，才接受该更新。
- **SGPO-II（基于代码回放的提示优化）**：
  - 证据来源改为历史合并请求（MR），过滤后转换为需求任务。
  - 评估标准五个维度（语义正确性40%、需求覆盖率25%、文件覆盖率20%、默认安全10%、代码风格5%），加权得分≥4且语义正确性≥4为通过。
  - 失败时生成语义梯度并精炼开发智能体的提示。

#### (e) 模型研究探索循环（第7.2节）
- **三阶段穷举**：
  1. 阶段1：对每篇论文生成一个复现方案，运行后生成leaderboard，取Top K=16。
  2. 阶段2：对Top K论文的可消融模块，枚举原选择与LLM推断的替代，评估各模块独立贡献（在32个实验中至少24次正AUC为有效）。
  3. 阶段3：跨论文组合，每4个常规轮后插入挑战轮（将Top有效模块应用于排名17-32的论文），防止收敛到少数方案。
- **记忆引导剪枝**：论文级（核心前提被证伪则整篇排除）、模块级（消融无效则排除组合）、饱和度检测（架构字段哈希重复率>80%连续两轮则终止）。
- **经验飞轮**：每个实验结论（成功/失败）附加到只增事件日志，通过阈值门（至少2个独立运行）和对抗审查门（专用智能体尝试证伪）后，才成为可用的playbook或anti-pattern。

## 6. 主要结论与发现

1. **闭环可行且高效**：AgentX在真实工业推荐系统中完成了从想法到上线再到自我进化的完整闭环，不依赖人类逐环节拼接。
2. **显著业务增益**：三周内10个可上线结果，主Feed用户消费时长累计提升0.561%，生活服务年化收入超1亿元人民币。每个Worker每周LR数1.1个，是工程师的13.8倍，但每想法转换率略低（2.7% vs 5.1%）。
3. **自我进化加速**：系统通过SGPO每周并发实验量翻倍、想法通过率从15%提升至45%、每周LR数从2增至5。
4. **瓶颈主要在平台侧**：91.4%的想法拒绝源于平台/基础设施约束（参数冲突、缺失属性等），而非智能体推理错误；编码失败中>95%是基础设施侧问题。
5. **模型研究闭环有效**：AgentX能自主复现论文、消融模块、跨论文组合，最终2个上线模型在快手直播时长上提升0.865%。
6. **SGPO有效提升智能体能力**：通过语义梯度和配对回放，Brainstorm子智能体得分从75%提升到98%，开发智能体在复杂模块上得分从2.60提升到4.90。
7. **负结果资产化**：系统将失败实验的结构化根因存储为可检索的知识，避免重复犯错。
8. **可证伪归属机制防止虚假增益**：在模型实验中，若因果链条不完整，即使AUC上升也不记录，从而维护知识库质量。

## 7. 优点

1. **完整工业闭环**：首次在真实生产环境中实现了从想法到A/B测试到自我进化的全链路自动化，验证了工业级可行性。
2. **真实在线奖励信号**：直接使用线上用户反馈作为优化目标，避免了离线指标与线上效果的不一致问题。
3. **自进化机制（SGPO）**：通过语义梯度和配对回放，使智能体系统本身持续改进，而非一次性自动化工具。
4. **严格的安全性设计**：护栏否决、金丝雀部署、工程白名单、人工审查门控等多层安全措施保障生产环境稳定。
5. **可证伪归属与知识质量保障**：模型实验要求因果链条验证，负面结果结构化存储，知识进入知识库前需经过阈值和对抗审查，防止噪声积累。
6. **模块化与可扩展性**：四个智能体独立但协作，支持在线策略和离线模型两条轨道，模型研究探索采用三阶段穷举+记忆剪枝，搜索效率高。
7. **透明且可追溯**：监控平台记录每个环节转化率，实验拒绝原因分类明确，SGPO进化过程可通过配对回放验证。

## 8. 不足与局限

1. **实验覆盖范围有限**：
   - 仅在一个平台（快手）上进行，未在其他推荐系统（如抖音、YouTube、Netflix）上验证泛化性。
   - 只测试了主Feed和生活服务两个场景，其他业务线（如广告、短视频、直播）未涉及。
   - 模型研究部分仅在公开数据集和单一工业场景验证，未与其他AutoML/智能体方法（如AIDE、MLE-STAR、自进化推荐系统[Wang 2026b]）进行直接对比。
2. **对比基线不够全面**：
   - 生产端仅与一名手工工程师对比，未与其他自动化系统（如传统AutoML、规则基系统）对比。
   - 模型研究侧仅以RankMixer为基线，未包含当前SOTA模型（如DIN、DIEN、SIM等）作为基线。
3. **算力消耗未量化**：
   - 未报告GPU型号、数量、训练时长、总计算成本等关键资源信息，难以评估效率和可复制性。
   - 三周内运行374个想法、100个部署、214个模型实验，成本可能较高，但原文未讨论。
4. **依赖特定平台基础设施**：
   - AgentX高度依赖快手的内部平台（AB平台、DSL、训练框架、知识库等），迁移到其他系统需要大量适配工作。
   - 系统中约91.4%的拒绝原因是平台/基础设施约束，说明系统对平台弹性要求高。
5. **SGPO进化速度可能较慢**：
   - 每轮只更新一个子智能体的一种提示，通过配对回放准入，整体进化速度受限于计算和数据。
   - 进化过程中可能出现暂时回撤（如图8所示），需要多轮才能稳定提升。
6. **实验数量与充分性方面的局限**：
   - 生产实验仅持续三周，时间较短，长期效果和季节性影响未考察。
   - 模型实验中，模块组合未覆盖所有可能的组合（仅测试了部分），可能遗漏最优组合。
   - 负面结果资产化的长期积累效果未量化。
7. **人工依赖仍然存在**：
   - 头脑风暴阶段需要人工审查（虽然审查项精简化）。
   - A/B实验的创建仍需人工输入（实验名和world）。
   - 高度复杂的策略或需要多团队协调的变更可能仍需人工介入。
8. **安全风险未完全消除**：
   - 虽然有多层安全设计，但智能体可能通过从未见过的路径触发危险变更（如绕过白名单的遗留配置）。
   - 外部知识库的知识可能过时或错误导致错误假设。
