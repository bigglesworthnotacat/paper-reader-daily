## 摘要
AI智能体被授予对工具、API和其他基础设施的访问权限，使其成为这些系统中的活跃主体。主流方法将控制置于智能体自身的运行时内部：系统提示、输出过滤器和护栏库。在智能体地址空间中的任何控制都可通过影响它的输入到达；这适用于任何对其自身运行时具有足够访问能力的AI系统，我们将其归类为可逃逸AI系统。我们确定了授权机制必须满足的四个属性，以实现架构控制而非协作请求：进程隔离、仅在结构唯一路径上的执行前强制、请求和系统级别的故障安全关闭，以及受控系统信任边界外部可验证的外部化签名证据。我们将这一层定位为执行时AI对齐，补充训练时对齐（RLHF、宪法AI）和推理时对齐。我们提出了不可撤销的安全内核，这是一个实现所有四个属性的Rust参考实现。其故障安全不变性在两个层面经过机器验证：SMT定理（Z3）和生产决策函数的穷举有界模型检验证明（Kani，4/4测试套件）。从Python到Rust的迁移以字节等价性为条件（1000/1000个测试用例；17/17个对抗类别）。我们评估了该内核管理一个实时的、可逃逸的AI系统（一个确定性的、自我改进的世界模型），对抗一个寻求逃逸的对手，该对手驱动其真正的自我修改接口：在1000次自我修改中，所有704次针对安全关键核心的尝试均被拒绝，无逃逸；另外300次在操作员终止开关下也被拒绝。另一系列6240次授权往返测试中，无成功绕过。对比三个声称控制AI智能体控制面的当代系统，智能体调用控制；在此，它缺乏这一选择。

## Abstract
AI agents are granted access to tools, APIs, and other infrastructure, making them active principals in those systems. The dominant approach places controls inside the agent's own runtime: system prompts, output filters, and guardrail libraries. Any control in the agent's address space is reachable by inputs that influence it; this generalizes to any AI system with sufficient reach into its own runtime, a class we term escapable AI systems.   We identify four properties that an authorization mechanism must satisfy for architectural control rather than for cooperative requests: process separation, pre-action enforcement on a structurally only path, fail-closed at both the request and system levels, and externalized signed evidence verifiable outside the controlled system's trust boundary. We position this layer as execution-time AI alignment, complementing training-time alignment (RLHF, Constitutional AI) and inference-time alignment.   We present the Unfireable Safety Kernel, a Rust reference implementation realizing all four. Its fail-closed invariant is machine-checked at two levels: an SMT theorem (Z3) and an exhaustive bounded-model-checking proof of the production decision function (Kani, 4/4 harnesses). A Python-to-Rust migration was gated on byte-equivalence (1000/1000 fixtures; 17/17 adversarial classes). We evaluate the kernel governing a live, escapable AI system, a deterministic, self-improving world model, against an escape-seeking adversary driving its real self-modification seam: across 1,000 self-modifications, all 704 attempts on the safety-critical core are refused, with no escape; a further 300, under the operator kill switch, are also refused. A separate campaign of 6,240 authorization round-trips had no successful bypass. Against 3 contemporary systems claiming the agent control plane, the agent invokes control; here, it lacks that choice.

- **核心问题**：AI 智能体（以及更广泛的“可逃逸 AI 系统”——即具有足够运行时操控能力以绕过自身控制措施的系统）在获得工具、API 等基础设施访问权限后，成为持续活动的主体，其行为可能因提示注入、模型自身误对齐或欺骗而偏离操作者意图。现有安全控制（系统提示、输出过滤器、护栏库等）全部位于智能体的地址空间内部，理论上可被智能体的输入影响或被自身代码重写（例如通过 `sys.modules` 替换、猴子补丁等），因此是“可合作请求”而非“不可绕过的架构约束”。
- **整体含义**：需要一种架构级的安全控制层，使智能体**无法**通过任何手段关停、绕过或替换该控制。本文提出 **执行时 AI 对齐（Execution-Time AI Alignment）** 作为对齐的第三层（补充训练时对齐如 RLHF、宪法 AI，以及推理时对齐如提示过滤），并设计实现了 **不可撤销的安全内核（Unfireable Safety Kernel）**，为可逃逸系统提供确定性、可验证的授权边界。

### 论文提出的方法论：核心思想、关键技术细节、形式化验证

#### 核心思想与四大属性
作者从威胁模型出发，推导出架构级授权机制必须满足的四个属性（P1–P4）：
- **P1 进程隔离**：控制逻辑运行在与被控系统不同的独立进程中，被控系统对控制进程无写入权限，仅能通过只读接口通信。
- **P2 仅结构唯一路径上的执行前强制**：每个有后果的动作必须**必然**经过该控制，控制位于动作与系统之间**唯一的**路径上，被控系统**不能选择**是否调用该控制。
- **P3 请求级与系统级故障安全关闭（Fail-Closed）**：请求级：若控制不可达、返回错误、或签名验证失败，则**拒绝**动作；系统级：若内核未配置、启动时不可达、或运行时丢失连接，则智能体**自身拒绝启动或停止有后果的动作**（“无内核，无智能体”）。
- **P4 外部化签名证据**：每个被允许的动作均产生一条日志记录，追加到可验证的、由操作员密钥（内核**不持**有的私钥）签名的透明度日志中，使攻击者即便攻陷内核进程也无法伪造可验证的授权链。

#### 关键技术实现
- **四层防御纵深（Four-Seam Architecture）**：从网络层到应用层共四个独立、各自故障安全关闭的“接缝”（nginx `auth_request`、应用中间件、调度钩子、客户端 SDK 断路器），每个都可独立拒绝动作。
- **静态授权面**：所有授权路由在**编译时静态组装**，运行时不支持动态添加策略端点，防止智能体通过内核接口注册新能力。
- **透明度日志（Transparency Log）**：每条允许记录附带内核签名（内核持有临时密钥）和操作员签名（操作员私钥离线保存），任何第三方持有操作员公钥即可验证日志完整性，检测内核是否被攻陷而非法授权。
- **二进制证明（Binary Attestation）**：通过 Sigstore 风格证明将发布二进制摘要与已签名发布清单绑定，后台协调器实时检测差异。
- **Python 配套防御库**：利用 `sys.addaudithook` 在 Python 层拦截 `import`、`exec`、`compile` 事件并转发给内核，在动态代码替换发生前拒绝。

#### 形式化验证
- **SMT 定理证明（Z3）**：对抽象模型证明故障安全关闭不变性：∀σ. (gate_ok(σ) ∧ fail_closed_config(σ)) ⇒ (allow(σ) ∧ ¬transport_error(σ))，并验证其非空虚性（Arm B）。
- **Kani 有界模型检查**：对生产 Rust 代码中的纯函数 `gate_decision(state, cooldown_elapsed, probe_in_flight) → GateDecision` 进行穷举验证，四个测试套件（验证 Open 状态下冷却期内总是拒绝、Open 仅冷却期后允许等）全部通过，0 失败。该函数是实际断路器回调路径的直接委托，保证证明绑定到生产代码。

### 实验设计：数据集/场景、基准、对比方法

- **迁移等价性测试**：使用确定性种子（`0xA1A14944`）生成的 1000 个请求测试用例（覆盖认证、批准、元数据、误配置、边界五类攻击），将 Python 原型与 Rust 实现进行**逐字节输出等价**对比，正确率 1000/1000；另使用 17 个对抗攻击类别（伪造角色、缺失 API 密钥、重放 nonce、过期令牌等）进行跨语言拒绝行为一致性测试，17/17 完全相同。

- **对抗鲁棒性测试**：超过 80 个 Rust 原生鲁棒性测试函数，覆盖令牌伪造、事件指纹伪造、重放/TTL、注册绕过、调用者角色绕过、透明度日志篡改、客户端故障安全关闭（6 个测试确认 500 错误时调用者拒绝、永不静默批准）、mTLS 证书固定等。每个属性均映射到论文声明的威胁模型主张。
- **实时可逃逸系统逃逸测试**（核心实验）：

  - 在最顶层，将对抗者具体化为一个**自主追求逃逸的自我改进智能体**，驱动真实自我修改接口（修改策略、自主性阶梯、密封本身、编译内核、传输白名单等）。在 1000 次自我修改中，704 次针对安全核心的尝试**全部拒绝**（`safety_critical_core_requires_attestation`），296 次良性修改被允许；在操作者天花板设为 0（终止开关）时，额外 300 次全部拒绝（包括 91 次良性）。**无成功逃逸**。
- **对比方法**（第 7 节）：
  1. **Galileo Agent Control**：通过装饰器/框架插件在进程内执行策略，代理可选择绕过。
  2. **Microsoft Agent Governance Toolkit**：同样在进程内挂钩 LangChain/CrewAI 调用处理器，代理可绕过未覆盖的代码路径。
  3. **Microsoft Authorization Fabric**：远程策略决策点，但执行决策的调用方仍是智能体运行时，不是强制路径。
- 对比结论：三者均具有“智能体决定是否调用控制”的共同结构缺陷，而不可撤销安全内核的架构使智能体**无权选择**。

### 论文的主要结论与发现

1. 现有 AI 智能体安全控制因位于智能体地址空间内而**架构上不安全**，无法抵御有能力的内部威胁（如模型假对齐、自我修改）。
2. 满足四个属性（P1–P4）的授权机制可实现**不可绕过的架构控制**，并将这一层命名为**执行时 AI 对齐**。
3. 不可撤销安全内核（Rust 实现）在全部测试中满足故障安全关闭不变性，经 Z3 和 Kani 双重形式化验证，所有逃逸尝试在真实系统中被零成功绕过。
4. 与三个当代系统对比，内核是唯一一个智能体**无权选择是否调用控制**的系统。
5. 提议执行时对齐作为训练时与推理时对齐的**补充**而**非替代**，三者共同构成完整对齐栈。

### 优点

- **架构创新**：明确提出并严格定义了可逃逸系统和四个必要控制属性，将安全控制从“合作性请求”提升为“结构不可绕过”。
- **多点防御纵深**：四个独立接缝 + 系统级生命周期绑定，使得单点故障不会导致安全失效。
- **形式化验证**：同时使用 SMT 模型定理证明和有界模型检查验证故障安全关闭逻辑，且模型绑定到生产代码函数，连接了抽象证明与实际实现。
- **严格的迁移纪律**：通过字节等价性门控，确保重构不引入安全回归；且后续每日严格的等价性管道捕获了普通 CI 遗漏的分歧。
- **透明度日志 + 操作员密钥**：即使内核被完全攻陷，第三方仍可检测到非法授权（证书透明度思想的类比应用）。
- **开源可用**：Apache-2.0 许可，提供完整的测试套件、证明脚本，鼓励审查与协作。

### 不足与局限

- **外部验证尚未完成**：缺乏独立第三方红队评估，对抗者由作者构建，存在潜在偏差。
- **形式化验证覆盖不完全**：仅形式化证明了断路器决策（gate_decision）的故障关闭属性；令牌签名/验证、Merkle 日志、策略评估等仅通过测试覆盖，未形式化验证。
- **持续负载与容量特性未充分刻画**：在高并发下 Rust 内核可能出现退化（C=100 时），尾延迟、吞吐极限、请求丢弃行为（需确认丢弃总是拒绝而非静默允许）仍需深入。
- **跨智能体新兴逃逸**：多智能体系统中，一个智能体的动作可能削弱另一个智能体的控制，当前架构仅处理单智能体动作授权，跨代理协作的逃逸需要额外分析。
- **自修改策略缺失**：内核只授权“是否允许自我修改”，但缺乏描述“允许什么样的修改”的形式化策略层（例如禁止修改策略本身），作者承认这是未来工作。
- **操作员密钥内部分误用**：具有操作员私钥的内部人员仍可签署恶意策略产出无效授权，当前仅通过事后检测（透明度日志对比）来发现，没有预防机制。
- **侧信道泄漏**：允许/拒绝决策的时间模式尚未使用恒定时间路径防御。
