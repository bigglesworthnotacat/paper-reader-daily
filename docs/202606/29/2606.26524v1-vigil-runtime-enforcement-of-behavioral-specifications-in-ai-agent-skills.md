## 摘要
智能体系统越来越多地通过第三方技能执行操作，使得模型生成的决策能够影响文件、通信渠道和网络物理设备。这些技能通常包含自然语言规范，定义了访问权限、披露限制、执行权限和必要前置条件。尽管此类规范描述了技能行为的预期边界，但它们本身并不提供可执行的运行时强制实施。实施这些规范带来了上下文粒度挑战：即使针对特定任务上下文编写了策略，监控器仍需决定观察哪些事件、保留哪些状态、推理执行的范围以及何时干预。选择错误的粒度可能要么阻止良性执行，要么遗漏仅跨多个操作出现的违规。然而，大多数现有的强制实施机制假设了固定的事件模型或强制实施点。在这项工作中，我们提出了VIGIL，一个面向智能体系统的端到端运行时强制实施框架。VIGIL根据来自技能规范、操作员定义的约束以及跨多个技能的全局规则的行为策略，检查智能体的实际执行轨迹。为了使这类策略可执行，VIGIL引入了一种策略语言，该语言捕获了智能体-工具事件上的上下文特定强制实施需求，包括时间依赖、参数约束和值流条件。该语言配以符号评估规则，将策略转化为有限轨迹上的SMT约束，使得VIGIL能够检测依赖于事件顺序、参数关系或跨调用值流的违规，而非依赖固定的单调用过滤器。在涵盖办公文档、操作和工程任务的真实LLM智能体运行中，VIGIL检测策略违规的召回率超过95%，假阳性率低于10%。

## Abstract
Agentic systems increasingly act through third-party skills, allowing model-generated decisions to affect files, communication channels, and cyber-physical devices. These skills often include natural-language specifications that define access permissions, disclosure limits, execution privileges, and required preconditions. Although such specifications describe the intended boundaries of skill behavior, they do not by themselves provide executable runtime enforcement. Enforcing them raises a contextual granularity challenge: even when a policy is written for a particular task context, a monitor must still decide which events to observe, what state to retain, how far across the execution to reason, and where to intervene. Choosing the wrong granularity can either block benign executions or miss violations that emerge only across multiple actions. Most existing enforcement mechanisms, however, assume a fixed event model or enforcement point.   In this work, we present VIGIL, an end-to-end runtime enforcement framework for agentic systems. VIGIL checks an agent's actual execution trace against behavioral policies from skill specifications, operator-defined constraints, and global rules spanning multiple skills. To make such policies executable, VIGIL introduces a policy language that captures context-specific enforcement requirements over agent-tool events, including temporal dependencies, argument constraints, and value-flow conditions. The language is paired with symbolic evaluation rules that translate policies into SMT constraints over finite traces, allowing VIGIL to detect violations that depend on event order, argument relationships, or cross-call value flow rather than relying on fixed single-call filters. On real LLM-agent runs spanning office-document, operational, and engineering tasks, VIGIL detects policy violations with over 95% recall and a false-positive rate below 10%.

## 2. 方法论：核心思想、关键技术细节
- **核心思想**：严格分离行为规范的“通用含义”与“运行实例化”。通过**轨迹抽象**将原始工具调用转化为带类型事件的事实基，**策略编译**将规范实例化到该轨迹的词汇表上，**SMT检查**将每条策略转化为一个量词自由的SMT可满足性查询。
- **关键技术细节**：
  - **轨迹抽象（Abstract）**：将每个工具调用转化为**类型化事件** `e = (action, arguments map, status, bounded output)`。保留执行顺序 `e1 ≺ e2 ≺ ...`。提取**签名 Σ = (动作集, 参数名集, 值集)** 和**事实基 JτK**（所有原子事实的集合）。事实基是封闭世界的，缺失即表示未发生。
  - **策略语言（Policy Language）**：支持六种**时间形态**：`abs`（禁止）、`prec`（前置条件）、`resp`（响应）、`bresp`（有界响应）、`rslv`（最终解决）、`until`（区间保护）。模式（pattern）支持动作条件、参数约束、变量绑定（x用于连接两个模式中的相同值）。另有降级 `□Θ`（任意时间体）。
  - **编译（Compile→Rφ）**：语法导向，每条策略 φ 编译为对每个事件的约束合取 `Rφ = ∧ Rφ(e)`。每个约束是量词自由的，使用有限事件集上的析取/合取。例如 `prec(ψ,ψ′)`: 每个匹配 ψ 的事件必须有一个更早的匹配 ψ′ 的事件；若共享变量 x，则要求两者参数值相等。
  - **SMT检查（Verify）**：将 JτK 与每个 Rφ 同时断言，检测 `unsat(JτK ∧ Rφ)` 即表示轨迹违反 φ。使用Z3求解器的**不可满足核**（unsatisfiable core）定位具体违规事件。
  - **在线部署**：在每个待执行调用前，VIGIL基于已观测轨迹前缀进行检查。违反禁止或前置条件则阻止调用；待响应义务在运行结束时判断。
- **算法流程**（Algorithm 1）：
  1. `(Σ, JτK) ← ABSTRACT(L)` 提取签名和事实基。
  2. `P ← COMPILE(D, Σ)` 将自然语言规范编译为策略集合。
  3. 对于每条 φ ∈ P，编码为 Rφ，检查 unsat(JτK ∧ Rφ)。如不可满足，则从核中定位违规事件 loc(φ)，加入 W。
  4. 若 W 为空返回 safe，否则返回 unsafe 及违规事件集。

## 7. 优点
- **端到端运行时监控**：无需每个技能编写专门的强制代码；策略编译和SMT检查完全确定。
- **可检测跨动作的时序和数据流违规**：通过变量绑定和时间形态，发现单步检查无法察觉的隐藏违规。
- **定位精确**：利用SMT不可满足核返回违规事件，便于回滚或再规划。
- **低假阳性率**：封闭世界假定和有限轨迹事实确保缺省判定安全。
- **可扩展性**：运行时词汇饱和，开销不随轨迹长度增长；编译步骤可缓存。
- **系统性强**：覆盖学术界（SkillsBench）和工业界（NVIDIA、Databricks等）真实技能，验证了实际价值。

## 8. 不足与局限
- **可观测性边界**：策略相关行为若隐藏在脚本内部（如 `secure_archive.py`），不转化为工具调用，则VIGIL无法监控，产生假阴性。
- **编译保真度**：LLM生成策略时可能过度泛化（如忽略“批量”限定），导致假阳性。译者提示工程可改进但非完全可靠。
- **依赖可信收集器**：假设收集器完整记录所有工具调用；被攻击或绕过的情况超出模型范围。
- **仅限有限轨迹安全属性**：无法处理无限路径或不终止的程序；对实时性要求极高的系统，0.27s检查额外延迟可能仍不可接受。
- **SMT求解扩展性**：实验未测试超长轨迹或极复杂约束下的求解时间；当事件集和约束巨大时，Z3查询可能变慢。
- **人工标签集规模较小**：SB+SI仅152条，可能未覆盖所有违规模式。额外两个基准为现有benchmark，但主要对比限于SB+SI。
- **仅支持工具调用级事件**：无法审计模型内部的推理或未表示的副作用（如全局变量修改）。现实中的脚本可能混合多种执行路径。
- **未评估攻击场景下对抗性绕过**：论文假设技能内容被攻击者控制，但未测试攻击者如何通过学习边界构造隐蔽攻击序列。
