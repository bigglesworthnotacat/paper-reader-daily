## 摘要
我们提出HORIZON，一种将硬件设计视为仓库级代码演化的自演化智能体框架。一个Markdown工具链被编译成包含领域知识、可执行评估器、接受谓词以及git/运行时策略的项目包；随后，一个无需人工干预的智能体循环通过仓库操作用于状态管理、追踪和重放，从而演化一个独立的git工作树。这将先前从EDA软件系统扩展至硬件设计工件本身的仓库级自演化工作进行了延伸。我们在ChipBench、RTLLM、Verilog-Eval以及九个CVDP类别上评估了我们的方法，通过完全无需人工干预的智能体循环在所有测试套件上实现了100%的基准完成率。然而，我们并不声称硬件设计的智能体AI问题已解决：这些基准测试是芯片设计中更广泛工程问题的受控代理。第\ref{sec:discuss}节探讨了当前研究的局限性，并指出了开放的研究挑战。

## Abstract
We present HORIZON, a self-evolving agent framework that treats hardware design as repository-level code evolution. A Markdown harness is compiled into a project pack containing domain knowledge, an executable evaluator, an acceptance predicate, and a git/runtime policy; a hands-free agent loop then evolves an isolated git worktree, using repository operations for state management, tracing, and replay. This extends prior works of repository-scale self-evolution from EDA software systems, to hardware-design artifacts themselves. We evaluate our approach on ChipBench, RTLLM, Verilog-Eval, and nine CVDP categories, achieving 100\% benchmark completion across all suites with a fully hands-free agentic loop. However, we do not claim that agentic AI for hardware design is solved: these benchmarks are controlled proxies for a much broader engineering problem in chip design. Section~\ref{sec:discuss} examines the limitations of the current study and highlights open research challenges.

## 1. 核心问题与整体含义（研究动机和背景）

- **研究问题**：硬件设计（尤其是RTL设计）的自动化长期以来依赖单次生成或简单修复循环，缺乏可迭代、可追踪、可复现的仓库级代码演化能力。现有仓库级自我演化工作（如AlphaEvolve、SATLUTION、ABCEvo）仅限于演化EDA软件系统（工程师运行的程序），而非硬件设计工件本身。本文提出将**硬件设计本身**视为一个仓库级代码演化问题，从而让智能体在版本控制、可执行反馈和接受门控的框架下自动迭代改进设计。
- **动机**：RTL设计的正确性依赖于周期级行为、复位约定、位宽、接口协议等精细约束，单纯语法正确的Verilog远远不够。需要一种框架，使智能体能够将候选工件放入可运行工作区，调用领域工具，解释失败原因并反复修订，直到满足显式接受条件。作者希望将软件工程中成熟的仓库级自演化范式扩展到硬件设计领域。
- **整体含义**：这项工作标志着从“智能体辅助硬件设计”向“硬件设计作为自演化仓库”的转变，提供了一种通用框架，可以统一处理RTL生成、补全、修改、测试生成、调试等多种任务，而无需为每个问题重新设计修复管道。但作者明确指出，当前基准是受控代理，实际问题远未解决。

## 2. 论文提出的方法论：核心思想、关键技术细节与流程

### 2.1 核心思想
- 将硬件设计任务封装为一个**版本控制的、可自动评估的、自包含的git工作树**。任务目标、领域知识、评估器和接受谓词通过一个Markdown harness定义，由bootstrap agent编译为**项目包（project pack）**，然后启动一个免手动智能体循环，自动编辑、评估、提交或拒绝候选版本。
- 核心原则：**候选变更只有在通过可执行正确性检查并改善性能指标时才被接受**——这一原则原本用于演化EDA软件，现在应用于硬件设计本身。

### 2.2 关键技术细节
1. **Markdown Harness → 项目包**：用户提供结构化Markdown描述，包含目标、领域知识、评估器规范、接受谓词等。Bootstrap agent `G_φ` 将其转换为项目包 `p = (π_agent, E_p, A_p, Γ_p, Ω_p)`，分别对应智能体策略提示、可执行评估器、接受谓词、版本控制/工件策略、领域技能和仓库指令。
2. **仓库追踪公式化**：
   - 状态 `s_t`：包含当前工作树git树、项目包、活动状态、日志、声明内存。
   - 选项 `a_t`：包含提议补丁、内部工具调用、最终提交/拒绝决策。
   - 评估证据 `y_t = E_p(w_t ⊕ Δ_t)`。
   - 状态转移：通过接受谓词`A_p(y_t)`决定提交或拒绝。
   - 奖励 `r_t`：可包含通过率、覆盖率、QoR变化、负的token和时间。本文只报告通过率、覆盖率和token消耗。
   - 轨迹 `τ_{0:D}`：记录所有外步骤。
3. **免手动智能体循环**：一次外迭代包含多次内步骤（编辑、调用工具、解释失败、修复）。循环完全自动，无需人工干预。每次外迭代后，通过git diff检查变更，git commit记录接受版本，git notes附加评估结果。Git历史即经验缓冲区。
4. **会话重用与缓存**：智能体保持持久模型会话，避免重复输入harness、项目包和稳定源，使每次修复迭代的新增费用很小。文中指出约91%的token是缓存输入。
5. **记忆与状态管理**：不假设马尔可夫状态，记忆由会话缓存和git工作树共同承载。评审（review）步骤保持独立，避免信息污染。

### 2.3 算法流程（文字说明）
- 输入：用户提供Markdown harness。
- Bootstrap：编译为项目包p。
- 循环（迭代次数由预算或收敛决定）：
  1. 智能体读取当前状态（工作树+项目包+记忆+日志）。
  2. 分析当前问题，制定行动计划。
  3. 编辑文件、调用工具（如编译、仿真）。
  4. 运行评估器E_p，生成证据y_t。
  5. 应用接受谓词A_p：若通过（Ap=1），git commit并附加证据；若不通过，记录拒绝日志。
  6. 更新奖励r_t（本文仅记录，不用于训练）。
  7. 重复直到所有任务通过或达到预算。
- 输出：完全通过所有任务的设计工件及完整git历史。

## 6. 论文的主要结论与发现

1. **框架有效性**：HORIZON在所有评估的RTL基准套件上实现了100%通过率，且完全免手动运行。这是首个端到端自动化完成所有RTL基准的智能体系统。
2. **收敛路径差异大**：简单生成套件（Verilog-Eval、RTLLM）在2次迭代内完成；CVDP中最简单的CID 014（断言生成）只需1次迭代，最难的CID 002（代码补全）需要82次迭代。
3. **token消耗高度集中**：CVDP类别消耗97.1%的token；CID 002单独消耗26.7%。约91%的token来自缓存输入，实际计算成本可控。
4. **覆盖率作为副观测信号**：在测试生成任务中，经过率门控的循环无法将覆盖率推到100%（CID 012收敛时平均覆盖率为97.9%），反映了接受谓词与真实验证目标之间的差距。
5. **当前基准的局限性**：暴露了奖励黑客风险、反馈延迟问题以及对真实芯片设计的简化。

## 7. 优点

1. **框架通用性**：将硬件设计任务统一转化为git仓库演化，可扩展至RTL生成、修改、测试生成、调试等多种任务，甚至潜在支持PPA优化和EDA工具自演化。
2. **自动化和可复现性**：完全免手动循环，从harness到收敛无需人工介入；完整的git历史保证了可重播和可审计性。
3. **成本意识**：明确报告token消耗和缓存比例，指出当前开销主要来自少数困难任务，为后续效率优化提供方向。
4. **诚实讨论局限性**：在结论部分深入分析了奖励黑客、延迟反馈、基准过拟合等问题，并引用了SWE-bench的经验教训，体现了严谨的学术态度。
5. **结果全面**：覆盖了多种任务类型（生成、修改、验证、调试），且全部达到100%，尽管有基准缺陷，但总体成果令人印象深刻。

## 8. 不足与局限

1. **单次运行与缺乏统计**：没有多轮重复实验报告方差，无法判断稳定性和随机性影响。
2. **没有对比基线**：未与现有迭代RTL方法（如AutoChip、RTLFixer、VerilogCoder、MAGE、ACE-RTL）进行公平比较，无法证明HORIZON的相对优势。
3. **奖励黑客风险**：智能体可访问详细评估日志，可能过度拟合特定测试，而非实现真正的设计意图。作者承认这一点，但未设计防御措施（如隐藏测试）。
4. **反馈延迟问题未量化**：文中评估时间较短，但在真实设计中（如综合、布局布线）延时可达天级，HORIZON的快速循环不适用，而论文未提供解决方案。
5. **覆盖率和QoR未作为优化目标**：覆盖率仅作为观测，QoR完全不涉及，限制了在真实芯片设计中的实用性。
6. **领域知识依赖**：需要用户提供结构化Markdown harness，包括领域知识和评估器，这本身可能构成瓶颈。
7. **基准有限的代表性**：CVDP等基准是受控问题，缺少设计约束变化、PPA权衡、验证完整性等实际挑战。
