## 摘要
使用工具的LLM智能体越来越多地读取不受信任的内容，同时持有具有副作用的工具，如支付、电子邮件、CRM和基础设施API，但常见的框架默认设置仍然将工具暴露与授权混为一谈。我们审计了LangChain/LangGraph、LlamaIndex和Stripe Agent Toolkit是否在每次模型发出的调用执行前，使用具体的参数值重新授权。在固定的公共源代码提交中，所有三者默认提供能力门，但没有一个默认提供确定性的、故障关闭的每次调用值授权门。我们引入了ScopeGate，一个用于智能体工具调用的五阶段PDP/PEP：范围、授权、资金上限、幂等性和默认拒绝。评估显示，相同的未经授权的支付调用在LangChain的默认调度下执行（附带LlamaIndex PoC），但被ScopeGate拒绝；测试的控制报告0/48静态绕过，0/29未经授权尝试（40次迭代自适应运行），0/10良性误拒绝，以及Latam-GPT支付智能体遏制率为10/10。ASR表示尝试的未经授权行动，遏制不是治愈方法，部署层声明是对测量模型类别的推断，并且未声称任何CVE。

## Abstract
Tool-using LLM agents increasingly read untrusted content while holding side-effecting tools such as payments, email, CRM, and infrastructure APIs, yet common framework defaults still conflate tool exposure with authorization. We audit whether LangChain/LangGraph, LlamaIndex, and the Stripe Agent Toolkit re-authorize each model-emitted call, with concrete argument values, before execution. Across pinned public-source commits, all three provide capability gating by default, but none provides a deterministic fail-closed per-call value authorization gate by default. We introduce ScopeGate, a five-stage PDP/PEP for agent tool calls: scope, authorization, money ceiling, idempotency, and default deny. Evaluation shows the identical unauthorized payout call executes under LangChain's default dispatch (with a companion LlamaIndex PoC) but is denied by ScopeGate; the tested control reports 0/48 static bypasses, 0/29 unauthorized attempts (40-iteration adaptive run), 0/10 benign false-denies, and Latam-GPT payment-agent containment at 10/10. ASR denotes attempted unauthorized action, containment is not a cure, deployment-tier claims are inference over measured model classes, and no CVE is asserted.

# 论文中文详细总结

## 1. 论文的核心问题与整体含义（研究动机和背景）

- **核心问题**：当前LLM智能体框架普遍将“能力门控”（capability gating，即哪些工具可被调用）等同于“授权”（authorization），忽略了每次工具调用的具体参数值是否需要授权。这导致经典的“混淆代理”（confused-deputy）漏洞：攻击者通过间接提示注入控制模型输出，使模型发出具有真实权限的工具调用（如转账、退款、修改CRM数据等），而框架默认仅检查工具名称和参数类型，不检查参数值是否合法。
- **背景**：LLM智能体常被集成到支付、邮件、CRM等敏感应用中，读取不可信内容（如用户邮件、网页、文档）。间接提示注入已被证实可行。论文关注的关键问题是：当模型发出恶意工具调用时，运行时系统如何响应？现有主流框架（LangChain/LangGraph、LlamaIndex、Stripe Agent Toolkit）默认未提供每次调用前的值级授权检测，存在安全缺口。

## 2. 论文提出的方法论：核心思想、关键技术细节

- **核心思想**：将“能力门控”和“每次调用授权”明确分离。能力门控决定哪些工具可被暴露给智能体；每次调用授权决定特定调用（含具体参数值）在当前主体和会话上下文中是否被允许。论文提出 **ScopeGate**，一个位于模型输出和工具执行之间的确定性PDP/PEP（策略决策点/策略执行点），在每次工具调用生效前强制检查五阶段策略，且策略完全由操作员在带外配置，不受模型输出影响。
- **关键技术细节**：
  - **ScopeGate五阶段决策流程**（算法伪代码见论文Algorithm 1）：
    1. **Scope（作用域）**：检查工具名称是否在策略允许列表中，不在则拒绝。
    2. **Authorization（授权）**：检查模型提供的每个参数值是否在已批准的允许列表中（例如 `account` 必须在验证集 `{acct_MERCHANT_001}` 中）。
    3. **Money ceiling（资金上限）**：对于支付/退款工具，检查金额是否为有限实数且满足 `0 ≤ amount ≤ ceiling`，拒绝NaN、无穷、非数字类型等。
    4. **Idempotency（幂等性）**：要求副作用工具必须提供可信的幂等性键，防止重放和循环放大。
    5. **Default deny（默认拒绝）**：只有全部通过才允许，任何错误也默认为拒绝。
  - **策略的带外性**：所有策略元素（允许工具列表、允许参数值、金额上限、幂等性要求等）均由操作员从可信根源（非LLM输出）加载，不可被模型上下文或注入影响。
  - **部署方式**：可作为SDK中间件、sidecar代理或中心化网关部署，安全属性不依赖部署位置。
