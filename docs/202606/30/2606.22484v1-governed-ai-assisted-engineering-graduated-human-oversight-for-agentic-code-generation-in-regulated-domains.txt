# Governed AI-Assisted Engineering: Graduated Human Oversight for Agentic Code Generation in Regulated Domains 

Dr. Richard Kang 

DoiT International richard@doit.com 

Abstract —The adoption of agentic AI coding systems — where autonomous agents generate, review, test, and deploy code with minimal human intervention — creates a governance challenge in regulated industries. Existing frameworks address AI-assisted development maturity or the productivity-reliability tension but offer no mechanism for calibrating human oversight intensity to regulatory impact. We present the Governed AI-Assisted Engineering (GAIE) framework, a three-tier graduated human oversight model for agentic code generation in regulated domains. GAIE introduces the Oversight Classification Model (OCM), a deterministic decision function that classifies code generation tasks by regulatory impact, customer proximity, reversibility, and data sensitivity to route them through one of three oversight tiers: human-in-the-loop (strategic functions), human-over-the-loop (customer-impacting), or automated-with-monitoring (inter-nal). Each tier defines required evidence artifacts for compliance auditability. We map GAIE against the Bank of Thailand’s 2025 AI risk-management policy and demonstrate cross-jurisdiction applicability to MAS (Singapore), NIST AI RMF, ISO/IEC 42001, and the EU AI Act. Evaluation through regulatory coverage analysis, comparative framework analysis, and analytical pro-ductivity modeling suggests that graduated oversight preserves 84–97% of agentic coding velocity (central estimate: 91%) while maintaining compliance evidence coverage for regulated functions. GAIE contributes a framework that explicitly bridges AI-assisted development maturity with regulatory governance through proportionate human oversight.

I. I NTRODUCTION 

A. The Agentic Coding Paradigm Shift 

AI-assisted software development has undergone a qualita-tive transformation. Where early tools such as GitHub Copilot operated at the granularity of a single line or function — suggesting completions within human-directed workflows — modern agentic systems perform multi-step reasoning, au-tonomous planning, tool use, and long-horizon code generation with minimal human intervention [1], [2]. Systems such as Claude Code, Devin, SWE-agent, OpenHands, and MetaGPT can receive a high-level task description and autonomously decompose it into subtasks, generate code across multiple files, write tests, debug failures, and prepare deployment artifacts [2], [3]. The speed is unprecedented. Agentic coding systems pro-duce in minutes what human developers produce in hours [4]. Controlled studies report productivity gains of 20–56% on well-scoped tasks [5], though the most rigorous randomized controlled trial documents a 19% slowdown for experienced developers and telemetry across 10,000+ developers shows 98% more pull requests but 91% longer review times with flat delivery metrics [5], [6]. This tension between generation speed and reliability assurance forms the core challenge this paper addresses. 

B. The Governance Challenge in Regulated Domains 

Regulated industries — banking, insurance, healthcare, en-ergy — face supervisory expectations that extend to any AI system influencing decisions with customer or public impact. Financial regulators including the Bank of Thailand (BOT), the Monetary Authority of Singapore (MAS), the UK’s Pru-dential Regulation Authority (PRA), and the US Office of the Comptroller of the Currency (OCC) require human oversight for high-impact AI-informed decisions [7], [8], [9]. When AI agents write code that governs regulated functions — credit approval logic, customer onboarding workflows, anti-money laundering screening rules — the oversight ques-tion transfers from model output governance to development pipeline governance. The institution remains responsible for code running in production regardless of whether a human or an agent wrote it [10]. The Bank of Thailand’s 2025 AI risk-management policy is explicit: human participation or human oversight is required whenever AI is used for strategic functions, defined to include credit approval, account opening approval, and approval of deposits, withdrawals, or transfers [7]. The policy further mandates lifecycle-wide controls across data quality, model evaluation, explainability, and AI-specific cyber defenses.

D. The Problem: Uniform Oversight Does Not Scale 

Two naive approaches to governing agentic code generation both fail: 

“Review everything” creates velocity collapse. Atlas-sian’s deployment of human-in-the-loop software development agents found that high cost of human feedback and integration challenges significantly constrained the value proposition [11], [12]. When every AI-generated artifact requires human review at traditional code-review speed, the productivity benefit of agentic coding is entirely consumed by review overhead — creating what Farrag terms the “Productivity-Reliability Para-dox” [5]. 

“Review nothing” creates regulatory non-compliance and unauditable code. Without governance, AI-generated code affecting strategic functions reaches production without evi-dence that human oversight occurred — a direct violation of regulatory expectations [7], [8]. The analogy to automotive autonomy is instructive. SAE J3016 [13] defines six levels of driving automation, each with different human attention requirements. The automotive industry does not debate “should cars be autonomous or not?” — it asks “what level of autonomy is appropriate for this driving context?” We apply the same principle to code generation: not “should agents code autonomously?” but “what level of oversight is appropriate for this code’s regulatory context?” 

E. Contributions 

This paper makes the following contributions: 1) The Governed AI-Assisted Engineering (GAIE) framework — a three-tier graduated human oversight model for agentic code generation that preserves velocity for low-risk tasks while enforcing human accountability for regulated functions. 2) The Oversight Classification Model (OCM) — aformal, deterministic decision function that routes code generation tasks to appropriate oversight tiers based on four risk dimensions. We establish key properties (mono-tonicity, fail-safety under correct or uncertain metadata, completeness) by construction. 3) A per-tier evidence artifact model — specifying what compliance evidence each tier produces, enabling regu-latory auditability as a byproduct of development rather than a separate workstream. 4) Regulatory mapping and cross-jurisdiction analysis —providing a traceability mapping from GAIE components to selected regulatory control expectations across five frameworks. 5) Multi-method evaluation — including regulatory cov-erage analysis, comparative framework analysis, and an-alytical productivity impact estimation with sensitivity analysis.

B. Human-in-the-Loop in Software Engineering 

CentaurEval [4] introduces a benchmark for measuring the value of human presence in agentic coding, finding that human reasoning combined with AI efficiency outperforms either alone on complex tasks. Takerngsaksiri et al. [11] and Pasuksmit et al. [12] report on Atlassian’s deployment of HITL agents, identifying high cost and the need for structured in-teraction protocols. Mozannar et al. [18] present Magentic-UI with co-planning and action-guarding patterns. Peng et al. [19] find that reviewers calibrate effort based on perceived risk — empirical evidence that humans naturally apply graduated attention. 

C. Maturity Models for AI in Software Engineering 

Anderson [20] presents the AI Codebase Maturity Model (ACMM), a six-level framework (Level 0–5) describing how codebases evolve from basic AI-assisted coding to fully au-tonomous systems. ACMM describes what level of AI assis-tance exists but not how to govern it proportionately . Zohaib et al. [21] present a layered maturity model for agentic AI in 6G software businesses. The CMMI lineage [22] provides precedent for staged maturity models. 

D. The Productivity-Reliability Tension 

Farrag [5] identifies the “Productivity-Reliability Paradox”: AI coding shows 20–56% productivity gains on well-scoped tasks but 19% slowdown for experienced developers, with 91% longer review times. Farrag proposes specification-driven governance as the resolution — applying uniformly regardless of regulatory risk. Our work observes that the paradox is an artifact of uniform governance applied to heterogeneous risk . Liang et al. [6] provide corroborating evidence (N=4,867 developers). 

E. Graduated Autonomy in Safety-Critical Domains 

The principle of graduated autonomy has precedent: SAE J3016 [13] (automotive, L0–L5), Parasuraman et al. [23] (aviation), Zabolotnii et al. [24] (clinical AI staged autonomy), and NUREG [25] (nuclear safety classification). These do-mains share: oversight intensity should be proportionate to consequence severity . GAIE applies this principle to software engineering. 

F. Governance Frameworks for Agentic AI in Software Engi-neering 

Several recent works address fragments: Ait et al. [26] (governance DSL), Treude et al. [10] (accountability gaps), Hasanli [27] (TDD governance), Boyuan et al. [28] (dual-helix governance), Zietsman et al. [29] (governance prompt quality), and Alenezi et al. [30] (SE reorientation). None provides a comprehensive tiered oversight framework calibrated to regulatory requirements. 

G. Regulatory Frameworks for AI 

The Bank of Thailand [7] requires human participation for strategic functions, lifecycle governance, and FEAT principles. MAS FEAT [8], NIST AI RMF [31], ISO/IEC 42001 [32], BIS [33], the FSB [34], and the EU AI Act [35] all mandate 

proportionate oversight but do not specify how to implement proportionality in development workflows. GAIE fills this gap. 

H. Summary and Gap Statement 

No existing work combines: (a) graduated human oversight, (b) calibrated to regulatory impact classification, (c) for agentic code generation specifically, (d) with per-tier evidence arti-facts, (e) validated against real regulatory frameworks. III. T HE GOVERNED AI-A SSISTED ENGINEERING (GAIE) FRAMEWORK 

A. Framework Overview 

Figure 2 presents the GAIE framework architecture. The framework operates as a governance layer between task in-take and code deployment, routing each code generation task through the appropriate oversight tier based on its risk classification. 

B. Design Principles 

GAIE is built on five design principles: 

P1. Proportionality. Oversight intensity scales with regula-tory impact. 

P2. Evidence-by-design. Every oversight event produces au-ditable artifacts automatically. 

P3. Fail-safe default. When classification is uncertain, the OCM escalates to a higher oversight tier, never a lower one. 

P4. Separation of generation and approval. The agent that generates code cannot approve its own deployment. 

P5. Regulatory defensibility. Every framework element maps to at least one specific supervisory requirement.

Definition 2 (Risk Feature Space). Each task t ∈ T is characterized by a risk feature vector: 

φ(t) = ( RI (t), CP (t), RV (t), DS (t)) (2) 

2) Classification Decision Function: Figure 3 presents the OCM classification decision tree. 

3) Formal Properties: Theorem 1 (Monotonicity). For any task t, increasing any risk dimension never decreases the assigned oversight tier. 

Proof sketch. The classification rules are ordered by decreas-ing tier assignment. If φ(t1) ≤ φ(t2) component-wise, then OCM (t1) ≤ OCM (t2).

Theorem 2 (Fail-Safety under correct or uncertain metadata). If risk dimensions are correctly observed, or if uncertainty is detected below the confidence threshold, OCM does not under-classify tasks relative to their true risk profile.

D. Three-Tier Graduated Oversight 

Figure 4 illustrates the human involvement, agent behavior, and evidence flow at each tier. 

1) Tier 1: Human-in-the-Loop (HITL): Trigger: RI =strategic. The coding agent halts and escalates control (RE-TURN CONTROL). The human reviews the proposed ap-proach, modifies if needed, and explicitly approves before generation proceeds. At deployment, the human again reviews all evidence and authorizes. 

Evidence produced: Classification log, RE-TURN CONTROL event, human identity, review timestamps, approval decision with rationale, modification diff, deployment authorization with cryptographic signature. 

2) Tier 2: Human-over-the-Loop (HOTL): Trigger: CP = direct, OR (CP = indirect AND DS = personal), OR (CP = indirect AND RV = irreversible). The agent generates and tests autonomously; a human approves deployment. 

Evidence produced: Classification log, generation trace, test results, security scans, human deployment approval. 

3) Tier 3: Automated with Monitoring (AWM): Trigger: All risk dimensions indicate low impact. Full autonomous pipeline with post-deployment monitoring and exception-based human escalation. 

Evidence produced: Pipeline log, test results, deployment record, monitoring baseline, anomaly absence confirmation.

G. Supporting Control Dimensions 

Three cross-cutting controls operate across all tiers: 

Data boundary enforcement. Pre-transmission filtering (regex + NER-based PII detection), access scoping, and in-ference isolation. 

Prompt/response safety controls. Aligned with OWASP LLM Top 10 [36] and MITRE ATLAS [37]: prompt injection detection (LLM01), output validation for backdoors and ex-filtration (ATLAS T0040–T0048), sandboxed execution, and supply chain validation (LLM05). 

Immutable audit trail. Complete provenance chain from task request through deployment and monitoring.

A. Positioning Against Related Work 

GAIE complements ACMM (describes where ; GAIE pre-scribes how ), resolves Farrag’s Paradox (by tiering rather than uniform governance), extends graduated autonomy to SE (from automotive, aviation, clinical domains), and operationalizes regulatory intent (translating principles into engineering pro-cesses). 

B. The Proportionality Principle 

Graduated oversight is superior on four grounds: regulatory alignment (aligns with the proportionality principle that reg-ulators have established), productivity evidence (HITL adds value on complex judgment tasks, not routine generation [4]), 

governance quality (concentrating attention prevents theatri-cal compliance), and information theory (graduated oversight allocates reviewer bandwidth to high-information decisions). 

C. Limitations and Threats to Validity 

L1. Single jurisdiction for primary mapping; cross-jurisdiction lighter. 

L2. No production deployment data; estimates are analytical. 

L3. Classification boundary ambiguity at edges. 

L4. Expert validation pending (5–8 practitioners planned). 

L5. Technology coupling in reference architecture. 

L6. Adversarial robustness: confident but incorrect metadata is a failure mode. 

L7. Regulatory mappings not externally validated by author-ities. 

D. Implications for Practice 

For regulated institutions: Adopt graduated oversight rather than the false dichotomy of “review everything” vs. “ban AI coding.” 

For AI coding tool vendors: Build RETURN CONTROL / escalation mechanisms as first-class features. 

For regulatory design (research observation): Our anal-ysis suggests that clarity of regulatory function definitions directly affects governance implementation efficiency. Where strategic function boundaries are clearly defined, institutions can calibrate oversight precisely. 

For researchers: Empirical studies needed on OCM ac-curacy, tier calibration, false-escalation rates, and evidence artifact effectiveness.

Agentic AI code generation in regulated industries requires governance, but uniform oversight creates a productivity-reliability paradox. GAIE resolves this through graduated human oversight calibrated to regulatory impact. The OCM routes tasks through three tiers based on regu-latory impact, customer proximity, reversibility, and data sen-sitivity. Human-in-the-loop applies to strategic-function code (estimated 5–15%). Human-over-the-loop applies to customer-impacting code (15–25%). Automated-with-monitoring ap-plies to internal tooling (60–80%). Each tier produces defined evidence artifacts. Evaluation suggests: (a) functional traceability to all ap-plicable BOT lifecycle requirements, (b) differentiation from existing frameworks, (c) formal OCM properties (monotonic-ity, fail-safety, totality), and (d) 84–97% velocity preservation (central: 91%) vs. 45–65% under uniform HITL. GAIE contributes a framework that explicitly bridges AI-assisted development maturity with regulatory governance through proportionate human oversight — making governance the enabler of velocity rather than its constraint.
