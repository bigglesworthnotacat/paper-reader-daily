## 摘要
AI智能体协议定义了智能体如何使用工具、委派工作以及跨软件系统进行协调，但其安全要求仍不完整，且在不同部署中执行不一致。我们提出了AgentThread，一个从规范文本到运行中SDK的源关联框架，用于智能体协议的安全保障分析。AgentThread贡献了分层安全范围、形式化为TLA+不变量的协议派生检查，以及一个两阶段检查器：它将协议规范编译成可模型检查的模型，并通过协议适配器针对真实SDK回放可执行反例。对于每个发现，AgentThread记录检查背后的源文本，并将违反的协议要求与缺失的建议、强化缺口以及未分配的跨协议责任分开。针对五个新兴的智能体协议，AgentThread识别出35个规范层面的发现，针对生产SDK和参考服务器进行了80次实现测试，并发现了30个仅在协议组合下出现的额外失败。我们进一步表明，只有一种协议在实践中执行了安全相关的控制，且没有协议为跨协议行为分配执行责任。因此，智能体协议的不安全性不仅是一个规范或实现问题，更是跨协议、SDK和部署的责任缺口。

## Abstract
AI agent protocols define how agents use tools, delegate work, and coordinate across software systems, but their security requirements remain incomplete and inconsistently enforced across deployments. We present AgentThread, a source-linked framework for security assurance analysis of agent protocols, from specification text to running SDKs. AgentThread contributes a layered security scope, protocol-derived checks formalized as TLA+ invariants, and a two-phase checker that compiles protocol specifications into model-checkable models and replays executable counterexamples against real SDKs through protocol adapters. For each finding, AgentThread records the source text behind the check and separates violated protocol requirements from missing recommendations, hardening gaps, and unassigned cross-protocol responsibilities.   Across five emerging agent protocols, AgentThread identifies 35 specification-level findings, supports them with 80 implementation tests against production SDKs and reference servers, and finds 30 additional failures that emerge only under protocol composition. We further show that only one protocol enforces a security-relevant control in practice and no protocol assigns enforcement for cross-protocol behavior. Insecurity in agent protocols is therefore not only a specification or implementation problem, but also a responsibility gap across protocols, SDKs, and deployments.

## 1. 核心问题与整体含义（研究动机和背景）
- **核心问题**：AI智能体协议（如MCP、A2A、ANP、ACP等）正成为软件基础设施的关键组件，但现有安全要求不完整且在不同部署中执行不一致。当多个协议通过运行时、中继或桥接组合使用时，不安全行为会跨越协议边界涌现，而没有一个协议或组件单独拥有全部安全责任。
- **研究动机**：现有工作要么专注于单个协议的表层验证（如消息格式、认证），要么只研究攻击案例和基准测试，缺乏从规范文本到实现SDK的系统性、源可追溯的安全保障分析方法。特别是，组合场景下的跨协议责任缺口（responsibility gap）尚未被正式研究。
- **含义**：智能体协议的不安全性不仅是一个规范缺陷或实现错误问题，更是一个跨协议、SDK和部署的责任分配问题。

## 2. 方法论：核心思想与关键技术细节
- **核心思想**：AgentThread是一个**源关联**（source-linked）的框架，从协议规范文本到运行中SDK，建立一条完整的证据链：提取规范条款→构建类型化中间表示（Protocol IR和Responsibility IR）→编译为TLA+形式化模型→使用TLC模型检查器生成反例→通过协议适配器在真实SDK/参考服务器上回放反例。
- **分层安全范围**：分为RPC层（L1消息/线格式完整性，L2会话生命周期，L3身份/能力绑定）和Agent层（L4语义操作控制，L5审计/问责）。每个层对应一组可检查的验证目标。
- **关键流程**（图3）：
  1. **源收集**（手动）：收集权威规范、模式、SDK文档、参考实现。
  2. **规范子句提取**（机器辅助+验证门控）：提取带有模态（MUST/SHOULD/MAY/未指定）、源引用、参与者绑定的子句。
  3. **类型化IR构建**（机器辅助+验证门控）：建立子句记录、转换记录、属性记录，保留源链接和层派生属性。
  4. **IR到TLA+编译**（自动化）：将IR中的行动者、前置条件、状态写入编译为TLA+行动，模态决定性质类型（安全不变量/强化检查/未指定发现）。
  5. **TLC模型检查**：每个不变量独立运行，生成反例轨迹。
  6. **实现回放**：通过协议适配器将反例轨迹转换为SDK/API调用，在真实实现上验证。
- **责任IR**：独立记录控制的所有者、SDK执行情况、缺口类型（拥有缺口、分离职责、执行缺口、范围缺口、组合孤儿）。

## 6. 主要结论与发现
- 1. **规范层面**：没有一个协议通过所有检查。五个协议均存在审计完整性和凭证/注册表完整性缺口。最严重的单个协议发现：ACP-Client在文件写入时缺乏路径限制，可能允许恶意内容修改安全敏感位置。
- 2. **实现层面**：MCP的内容完整性检查是直接的非遵守案例（规范要求净化工具输出，但SDK完全不净化）。A2A的代表性框架缺口出现在委托、同意、审计和凭证生命周期。
- 3. **组合层面**：30/43（70%）的组合安全义务存在反例。主要失败模式包括：隐藏中间人、权威/同意语义不匹配、身份与内容/范围正交。组合失败不是单个协议可见的，仅在边界出现。
- 4. **责任缺口**：仅ANP的DID身份验证被强制执行；组合安全被普遍“孤儿化”——没有任何协议拥有或执行桥接安全。最大安全责任往往落在运行时桥接或部署策略上，而非协议规范内。

## 7. 优点
- **系统性**：从规范文本到SDK的全链条、分层、源关联的分析框架。
- **形式化与实现结合**：TLA+模型检查生成反例后，通过协议适配器在真实实现上回放，将形式证据转化为可复现的测试用例。
- **责任分析创新**：引入责任IR，明确谁拥有控制、谁执行、谁缺失，填补了以往“谁修复”的空白。
- **自动化边界清晰**：源收集和适配器为人工，后续提取、编译、模型检查、回放均为自动化，便于协议演进时的重复分析。
- **组合方法论可扩展**：通过桥接动作和能力契约，将组合分析降级为单向协议分析+桥接分析，无需全新框架。

## 8. 不足与局限
- **实验覆盖有限**：仅分析5个协议，且均为当前主流但非全部；TLC为界限模型，可能遗漏某些深层攻击。
- **人工成分潜在偏差**：源收集、子句提取、IR构造、适配器编写均为人工主导，虽有机辅助但仍有主观偏差风险。
- **仅成对组合**：未分析三个及以上协议的组合，真实部署中可能出现更复杂的链式组合。
- **版本快照问题**：协议快速演进（MCP、A2A等），分析结果可能随时间过时。
- **无对比基线**：未与任何现有安全分析工具（如ProVerif, Tamarin等）进行对比，无法评估AgentThread相较于纯形式化方法的效率或覆盖率优势。
- **未覆盖所有部署场景**：仅测试了部分SDK和参考服务器，未包含所有第三方实现或自定义桥接。
