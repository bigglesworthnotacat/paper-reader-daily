## 摘要
所有主流LLM代理框架都将LLM置于编排者角色；模型决定下一步做什么、何时调用工具、何时停止。我们认为，令牌爆炸、控制流幻觉和不可靠的完成并非实现缺陷，而是将循环、分支和顺序执行的确定性工作分配给概率系统的架构性后果。更好的提示或更强的模型无法保证LLM代理的可靠性。因此，我们提出代理式编程，其中程序掌控所有控制流，而LLM本身是其组成部分，我们称之为LLM即代码，仅在任务需要推理或生成时才调用。在每次调用中，模型保持完全的灵活性，但不能改变程序的执行路径。由于控制权在程序手中，LLM的上下文由执行历史的调用树构建，形成有向无环图（DAG）。每次调用的上下文长度由其调用深度决定，而非步骤累积。一项计算机使用代理的案例研究表明，该设计是切实可行的，而不仅仅是理论立场，显著提升了长视觉操作序列的稳定性。

## Abstract
Every major LLM agent framework gives the LLM the role of orchestrator; the model decides what to do next, when to call tools, and when to stop. We argue that token explosion, control-flow hallucination, and unreliable completion are not implementation bugs but architectural consequences of assigning the deterministic work of looping, branching, and sequencing to a probabilistic system. A better prompt or a stronger model cannot guarantee the reliability of the LLM agent. We therefore propose Agentic Programming, in which the program governs all control flow, and the LLM is itself part of it, an adaptive component we call LLM-as-Code and invoke only where a task calls for reasoning or generation. Within each call the model keeps full flexibility, but it cannot alter the program's execution path. With control in the program, the LLM's context is built from the execution history's call tree and forms a directed acyclic graph (DAG). Each call's context length is then determined by its call depth rather than by accumulation over steps. A case study of computer-use agents shows that the design is practical, not just a theoretical stance, substantially improving the stability of long visual operation sequences.

## 2. 论文提出的方法论：核心思想、关键技术细节

### 2.1 核心思想：Agentic Programming（代理式编程）

*   程序负责所有控制流（循环、分支、顺序、错误处理），LLM 作为“代码中的 LLM”嵌入其中，只在需要推理、生成或判断的节点上被调用。
*   模型在每次调用中仍拥有完全的推理灵活性，但**不能改变程序的执行路径**，从而将“确定性保障”与“概率性推理”解耦。

### 2.2 关键技术细节

#### a) 代码驱动的工作流（Code‑driven Agent Workflow）
*   代理技能由普通代码编写，运行时强制执行每一步规则，而非通过提示要求模型遵守。
*   灵活性由**递归调用**实现：一个 LLM 调用可以根据其判断调用另一个 `@agentic` 函数，运行时动态展开调用图（`DAG`），既非硬编码管道，也非模型完全自决。

#### b) DAG 结构的上下文（DAG‑structured Context）
*   执行历史不再是一个平铺的对话日志，而是由调用树构成的有向无环图（DAG）。
*   **上下文伸缩规则**：当前运行的调用节点仅保留其**完整祖先链**的上下文，已返回的子调用被压缩为一句话摘要。上下文长度由**调用深度**决定 \\( O(\\text{depth}) \\)，而非累计步骤 \\( O(\\text{steps}) \\)，从根本上缓解了上下文爆炸和长上下文推理退化问题。
*   整个 DAG 保留在程序层用于复现与调试，但任何单次调用都无需看到全部历史。

#### c) 多代理协作（Multi‑agent Collaboration）
*   每个代理是一个普通函数，并行执行多个代理即并发调用多个函数，每个代理维护自己作用域内的上下文，结果在汇聚点（Join）通过程序逻辑合并，无需一个超级模型进行协调。
*   失败被局部化：一个代理的失败只是一个失败的函数调用，程序可以按自身规则重试、替换或忽略，不影响其他代理。

#### d) 自编程进化（Self‑programmed Evolution）
*   代理函数的改进过程本身也遵循同一范式：提议新函数的步骤是概率性 LLM 调用，但**验收标准（如通过测试）是确定性的**，一旦通过即作为固定代码提交，后续执行享有与任何普通代码相同的确定性保障，而非让模型每次运行时重新阅读并选择是否遵守。
