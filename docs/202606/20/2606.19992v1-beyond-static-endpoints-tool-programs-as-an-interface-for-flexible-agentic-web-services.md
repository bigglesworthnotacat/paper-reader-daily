## 摘要
在代理网络时代，基于大语言模型的代理越来越多地将网络服务作为工具来调用，然而大多数接口仍然是“静态端点”，难以表达包含循环、条件、连接和重试的长期工作流。我们提出了ToolPro，它将代理的工具意图表示为“可执行工具程序”，通过显式效果类型紧凑地编码多步骤服务交互。ToolPro结合了约束引导的程序构造、面向效果的重放以实现恰好一次的状态修改调用，以及基于配置文件的策略来决定程序执行何时优于逐步调用。我们通过WebAssembly沙箱在MCP风格的服务上实例化了ToolPro，并在现实世界中多样工作流上进行了评估。ToolPro将端到端延迟降低了高达53.4%，客户端流量减少了高达96.1%，在更高网络延迟和工作流复杂度下收益更大。

## Abstract
In the agentic web era, LLM-based agents increasingly invoke web services as tools, yet most interfaces remain \emph{static endpoints} that poorly express long-horizon workflows with loops, conditionals, joins, and retries. We present ToolPro, which represents an agent's tool intent as an \emph{executable tool program} that compactly encodes multi-step service interactions with explicit effect types. ToolPro combines constraint-guided program construction, effect-aware replay for exactly-once state-modifying calls, and a profile-driven policy that decides when program execution outperforms stepwise calling. We instantiate ToolPro over MCP-style services with WebAssembly sandboxing and evaluate it on diverse workflows of real-world applications. ToolPro reduces end-to-end latency by up to 53.4\% and client-side traffic by up to 96.1\%, with larger gains under higher network latency and workflow complexity.

### 1. 论文的核心问题与整体含义

*   **研究背景**：在“代理网络”时代，基于大语言模型（LLM）的智能体（Agent）需要编排复杂的、长时间跨度的网络服务来完成工作流。这些工作流通常包含循环、条件判断、中间变量绑定和重试等程序性逻辑。
*   **核心问题**：目前主流的服务接口是**静态API端点**。这种接口是为单次查询设计的，无法有效表达多步骤的程序性交互。当智能体使用端点时，必须将复杂工作流分解为一系列脆弱的、与多轮LLM推理相互交织的请求。这导致了三大性能瓶颈：
    1.  **延迟膨胀**：多轮客户端-服务端往返（RTT）和每步决策开销导致延迟增高。
    2.  **流量浪费**：控制逻辑在客户端外部执行，导致系统性地过度或不足地抓取数据。
    3.  **失败脆弱**：部分执行失败后的重试可能重复产生副作用（如重复写入），导致服务状态损坏。
*   **整体含义**：论文认为，低效的根源是**表示性**问题。作者提出将“工具程序”作为智能体与服务交互的一等接口，将多步骤交互打包为一个可执行、可委托、可优化和可检查的单一对象，从而从根本上解决上述瓶颈。

### 2. 论文提出的方法论：ToolPro

*   **核心思想**：将一个智能体对工具调用的**意图**，显式地表示为一个**可执行工具程序**。该程序能够紧凑地编码多步服务交互，包含控制流、中间变量绑定，并明确区分**读**和**写**两类操作的效果类型。
*   **关键技术细节**：
    *   **约束引导的程序构造与修复**：
        1.  **客户端合成与检查**：LLM 根据任务意图合成候选程序，客户端进行轻量级检查（如端点覆盖、控制流骨架），过滤掉明显不合格的程序。
        2.  **服务端投影**：服务端将候选程序重写到一个**受限的接口-程序表面**。该表面仅允许结构化控制流、统一的 `CALL` 外部调用接口、显式的读写效果注解，并禁止异常、线程等不安全操作。此投影确保了程序的可分析性和可执行性。
        3.  **编译与反馈驱动修复**：投影后的程序在 WebAssembly 沙箱中编译和执行。若编译或运行失败，服务端利用编译器诊断信息和运行时追踪进行**有界就地修复**，然后重新编译执行，直至成功或预算耗尽。
        4.  **安全回退**：若修复失败，系统会安全地 Fallback 到传统的“逐步调用端点”模式，并输出诊断信息。
    *   **面向效果的重放以实现重试安全**：
        *   这是解决“写入”副作用重复问题的关键机制。ToolPro 运行时拦截每一次外部调用。
        *   **读取** 调用总是直接转发到服务。
        *   **写入** 调用则受到重放保护：系统维护一个按意图实例划分的、已成功执行的写入操作日志。
        *   在程序修复后的重执行过程中，若遇到一个写入调用，系统会先在日志中查找匹配项。若找到，则直接返回缓存的结果，不再向底层服务发出调用，从而实现了接口级别的**恰好一次**语义。
        *   如果修复后的程序试图改变之前已提交的写入操作，重放机制会禁用并回退到逐步调用，以避免语义分歧。
    *   **基于配置文件的策略决定何时整合**：
        *   程序执行需要承担构造和编译的一次性成本。为判断何时整合更有利，ToolPro 引入了一个基于**性能模型**的决策策略。
        *   **模型公式**：系统通过在线画像维护最近的网络延迟、每步决策开销、构建时间等参数，并估算工作流所需的端点调用数 \( N \)。它通过以下公式来预测整合的净收益：
            \[ \Delta T = (N - 1) \times (\text{网络往返时间} + \text{决策开销}) - \text{构建开销} \]
        *   只有当 \(\Delta T > 0\)，即预测收益大于成本时，系统才选择“程序执行”模式；否则，选择“逐步调用”模式。

### 3. 实验设计

*   **场景与基准**：
    *   选取了三个来自 GitHub 的、真实且广泛使用的开源应用程序作为服务：**Memos**（知识管理平台）、**Directus**（API 层）和 **MinIO**（对象存储系统）。
    *   为每个应用构建了两个程序性工作流：一个**只读**（`.r`）和一个**读写**（`.w`）。每个工作流通过参数 \( N \) 控制其程序长度（即逐步调用所需的 API 调用次数）。
    *   此外，还设计了四个补充性复杂基准（`cbench1`-`cbench4`），用以测试非确定性检索、分支协调写入、非幂等副作用和跨服务执行等更复杂的情况。

### 6. 论文的主要结论与发现

*   **显著性能提升**：与静态端点逐步调用相比，ToolPro 能够在多样化的现实工作流中，将**端到端延迟降低高达 53.4%**，将**客户端流量减少高达 96.1%**。
*   **性能优势随条件增强**：在网络延迟越高、工作流越复杂（\( N \) 越大）的情况下，ToolPro 的性能优势越明显，验证了其减少网络往返次数的核心优势。
*   **策略的有效性**：基于配置文件的决策策略能有效地在程序执行和逐步调用间切换，确保了在各种网络条件下性能的鲁棒性（在低延迟时避免构建开销，在高延迟时通过整合获取收益）。

### 8. 不足与局限

*   **LLM能力依赖性强**：系统的成败很大程度上取决于 LLM 生成可编译、逻辑正确的代码的能力。论文也指出，模型能力不足（如 qwen3-coder-flash 在复杂任务上的编译失败）是导致性能下降的主因。这限制了该系统在当前较弱模型上部署的普适性。
*   **受限的程序表面**：为了确保可分析和安全性，工具程序被限制在一个非常严格的语言子集中。这可能会限制其处理极端复杂或非结构化逻辑的能力，尽管目前的工作流已经覆盖了主要场景。
*   **修复的保守性**：重放机制对“已提交的写操作前缀被修改”的情况采取保守的 Fallback 策略。这意味着某些从逻辑上本可以安全修复并重放的情况（例如调整了不影响服务状态的操作顺序），也会被放弃而退回到低效的逐步调用。
*   **单节点服务假设**：论文的实现和效果评估似乎是基于单节点服务实例的。对于大规模分布式服务，维护全局统一的“写入操作历史日志” \( H \) 会带来一致性和扩展性方面的挑战，论文对此尚未深入探讨。
