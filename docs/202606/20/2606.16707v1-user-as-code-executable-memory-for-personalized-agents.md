## 摘要
个性化 AI 代理需要用户记忆：一个持久化的用户模型，它通过多轮对话构建，并在每个新对话中被查询。如今，这种记忆几乎总是存储为非结构化文本、知识图谱或扁平化的事实库，并通过检索进行查询——提取与当前请求最相似的条目。这种“事实袋”式记忆虽然能较好地回忆单个事实，但由于存储事实和依据事实行动是分离的步骤，它难以解决矛盾、聚合多条记录或执行规则。我们认为用户记忆应该是可执行的。我们提出了“用户即代码”（UaC）范式，其中代理对用户的模型是一个活软件项目：带类型的 Python 对象保存用户状态，普通的 Python 函数编码管理状态的规则，因此对用户的表示和推理都在解释器可执行的同一媒介中进行。其实现机制是一个两阶段流水线：一个仅追加的日志，从不丢弃事实，并定期将其检查点化（checkpoint）为带类型的代码。这改变了记忆的能力。在标准的长期对话基准测试中，UaC 在回忆能力上（LOCOMO 上达 78.8%）既达到了全上下文的上界，也与最强的先前记忆系统相当。其优势在表示最为重要之处显现。对于关于用户历史的聚合问题——“去年我进行了多少次国际旅行？”——基于检索的记忆系统表现不佳（6-43%），而 UaC 保持近乎完美（99%），因为答案是对带类型状态的一行计算，而非对文本的搜索。而且，由于每次状态变化时其规则都会确定性地执行，UaC 能够主动浮现出安全关键警报——例如新开的处方药与数月前记录的过敏史冲突——这是查询驱动的记忆所无法提供的能力。

## Abstract
A personalized AI agent needs a user memory: a persistent model of who the user is, built across many conversations and consulted on each new one. Today this memory is almost always stored as unstructured text, a knowledge graph, or a flat store of facts, and consulted by retrieval -- fetching the entries most similar to the current request. Such "bag-of-facts" memory recalls individual facts well, but because storing a fact and acting on it are separate steps, it struggles to resolve contradictions, aggregate over many records, or enforce rules. We argue that user memory should instead be executable. We introduce User as Code (UaC), a paradigm in which an agent's model of a user is a living software project: typed Python objects hold the user's state and ordinary Python functions encode the rules that govern it, so representing and reasoning about the user happen in one medium an interpreter can run. The enabling mechanism is a two-phase pipeline: an append-only log that never discards a fact, periodically checkpointed into typed code.   This changes what memory can do. On standard long-term conversation benchmarks, UaC matches both a full-context upper bound and the strongest prior memory systems on recall (78.8% on LOCOMO). Its advantage emerges where representation matters most. On aggregate questions over a user's history -- "how many international trips did I take last year?" -- retrieval-based memory collapses (6-43%) while UaC stays near-perfect (99%), because the answer is a one-line computation over typed state rather than a search over text. And because its rules execute deterministically whenever the state changes, UaC can surface unsolicited, safety-critical alerts -- such as a newly prescribed drug that conflicts with an allergy recorded months earlier -- a capability query-driven memory cannot provide.

- **整体含义**：提出可执行记忆范式，将用户模型构建为可被解释器直接运行的 Python 代码项目，使记忆不仅能被检索，还能被计算和主动执行。

## 2. 方法论
- **核心思想（User as Code）**：用户状态用类型化 Python 数据类（dataclass）表示，管理状态的规则用普通 Python 函数编码，使得对用户的“表示”与“推理”在同一介质中由解释器统一处理。
- **关键技术细节**：
  - **两阶段流水线**：
    1. **Phase 1（记录）**：从每段会话中提取事实，以附加日志（append-only log）形式存储，从不覆盖或删除，并解析相对日期为绝对日期。
    2. **Phase 2（结构化）**：定期基于完整事实语料，通过 LLM 重新生成类型化 Python 代码（数据类实例），而非增量改写，从而避免信息丢失。
  - **生成-验证-复审循环**：编码代理由 LLM 编写约束函数，解释器确定性执行验证，复审结果并决定是否持久化，最终在清单（manifest）中展现预计算警报。
  - **多策略检索**：回答用户查询时，组合三条通道——结构化代码状态、事实向量检索和原始对话归档检索，各通道互补覆盖。
  - **设计与原则**：存储与验证统一、结构与数据分离、按生活领域模块化、渐进式信息披露、代理原生文件系统。
- **中间表示格式**：每个用户是一个目录，包含 `state.py`（数据类实例）、`constraints.py`（规则函数）和清单文件（展示活跃警报及领域摘要）。

## 6. 论文的主要结论与发现
- UaC 在标准回忆基准上与全上下文上界持平（LOCOMO 78.8% vs. 79.8%），在 LongMemEval 上与 MemMachine 和全上下文形成统计上无显著差异的头部集群。
- 在分析推理任务中，检索式系统如 MemMachine 和 Mem0 随记录数增加骤降（低至6-43%），而 UaC 保持 99% 正确率，因为聚合计算是对类型化状态的一行代码执行。
- 主动服务方面，UaC 的约束管线在标准场景达到 100% 警报检出，困难场景保持 85%，显著优于多数基线，主动触发安全警报是纯检索无法实现的能力。
- 追加式事实提取是提高回忆的决定性机制（+19pp）；两阶段分离结合了覆盖和类型化结构，比增量编码改写高出 12.3pp；约束管线对主动服务至关重要，对 QA 回忆无影响。
- 分析推理的单次查询中 UaC 成本高于全上下文+REPL，但随着对相同用户状态重复查询，结构化成本被快速摊销（3~11次后即超过 FC+REPL），长期成本优势显著。
- 模块化按需加载能在保持精度的同时大幅降低 token 成本，并避免单次过大上下文带来的精度下降。

## 7. 优点
- **范式创新**：将用户记忆从被动存储提升为可执行代码项目，融合存储与验证、表示与推理。
- **结构化解耦**：追加日志与定期结构化的两阶段设计，兼顾了记忆覆盖度和类型的结构性。
- **能力扩展**：从简单回忆拓展到分析聚合和主动约束检查，填补了现有记忆系统的功能空白。
- **实验扎实**：多维度基准测试、多基线对比、丰富消融和统计分析，跨 LLM、跨评判者的鲁棒性验证，以及成本和延迟的量化。
- **可组合性**：结构化代码与 SOTA 检索机制可叠加（附录B显示混合方案进一步提升），且模块化设计具备可扩展性。

## 8. 不足与局限
- **写入端开销**：两阶段引入两次 LLM 调用，结构化成本在单次查询中高于直接全上下文，对偶发查询不利。
- **结构化状态对纯回忆中性**：通道消融表明，类型化 Python 状态在 LOCOMO 式回忆任务上贡献微弱（-1.3pp，统计不显著），其主要价值体现在分析推理和约束检查任务上。
- **主动服务结果的局限性**：困难场景样本量较小（n=20），与最强基线 Mem0 的差异未达统计显著，需要更大规模验证。
- **分析推理基准为合成数据**：记录最“干净”，真实噪音更多，且单次查询设计不利于评估 UaC 的摊销优势。
- **扩展瓶颈**：当前第二阶段的全量再生在大规模会话中会遇到 500K 字符输入截断，需分级结构化。
- **同骨架重构与基础设施差异**：基线系统用简化的 ChromaDB 替代了原文献中的复杂检索栈，无法完全分解表征与检索栈的贡献。
- **代码执行安全**：可执行记忆需要沙箱隔离，生产环境需防止跨用户泄漏与代码注入风险。
- **模型依赖**：可移植性仅验证了两个闭源大模型，未覆盖较小型的代码生成模型。
