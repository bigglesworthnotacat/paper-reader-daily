## 摘要
我们提出GRACE-DS，一个面向LLM驱动的AutoML智能体部署前评估的数据科学中受保护的奖励引导智能体纠错环境。GRACE-DS是一套位于隔离环境中的评估指标，可应用于特定组织的表格型机器学习任务。它将智能体暴露于真实的工作流阶段，从规划和数据检查到特征工程、模型开发、验证、代码修复直至最终提交，同时隐藏的可执行验证器不仅评估最终预测性能，还衡量泄露避免、可复现性、协议有效性、纠错行为和奖励对齐。最强的结构化机制，即灵活迭代交互（我们的方法），相比单次生成、无结构交互和基于重启的基线，获得了更高的端到端标准化隐藏测试质量，同时提高了协议有效完成率。在超过7000个回合中的验证结果表明，GRACE-DS是一个稳健的平台，用于评估基于LLM的AutoML智能体在类似生产环境下并遵循组织特定要求执行机器学习工作流的能力。

## Abstract
We introduce GRACE-DS, a Guarded Reward-guided Agent Correction Environment in Data Science for pre-deployment evaluation of LLM-powered AutoML agents. GRACE-DS is a set of evaluation metrics in an isolated environment that can be applied to tabular ML tasks specific to a particular organization. It exposes agents to realistic workflow stages, from planning and data inspection through feature engineering, model development, validation, and code repair to final submission, while hidden executable validators measure not only final predictive performance but also leakage avoidance, reproducibility, protocol validity, correction behavior, and reward alignment. The strongest structured regime, flexible iterative interaction (our approach), achieves higher end-to-end normalized hidden-test quality than single-shot generation, unstructured interaction, and restart-based baselines, while also improving protocol-valid completion. Validated across more than 7,000 episodes, these results establish GRACE-DS as a robust platform for assessing the capacity of LLM-based AutoML agents to execute machine learning workflows under production-like conditions and in accordance with organization-specific requirements.

### 1. 论文的核心问题与整体含义

这篇论文针对的是**大语言模型（LLM）驱动的AutoML智能体在企业级生产环境下的部署前安全性、可靠性及流程合规性的评估难题**。

-   **核心问题**：现有的基准测试（如MLE-bench、TML-Bench）大多只关注Kaggle竞赛式的**最终预测得分**，而忽略了生产环境中的关键风险。例如，智能体可能通过**泄露目标变量**、过度拟合验证集、使用不可复现的代码或非法访问受保护数据来获得虚高的分数。这种“只看结果，不问过程”的评估方式，可能导致在公开基准上表现优异的模型，在实际内部混乱的数据和严格治理要求下完全失效。
-   **整体含义**：GRACE-DS弥补了这一差距。它不仅衡量模型的预测能力，更核心的是建立一个“部署契约”，通过结构化的交互工作流和隐蔽的过程验证器，全面鉴定智能体在方法规范性、代码可复现性、数据泄露避免、错误纠正能力以及奖励对齐等方面的行为，从而为组织提供一个可信、可操作的**部署决策依据**。

### 2. 方法论核心

GRACE-DS的核心是一个**隔离的、带反馈的、基于状态机的交互式评估环境**，它将表格型机器学习任务分解为多个显式阶段，并对智能体的每一步行为进行隐蔽审计。

-   **环境设计与状态机**：
    -   **模拟真实工作流**：智能体在一个受限制的沙盒中工作，按照`PLAN`（规划）、`EDA`（数据探索）、`FEATURE_ENGINEERING`（特征工程）、`MODEL`（模型训练）、`VALIDATE`（请求验证）、`CODE_FIX`（代码修复）和`FINAL_SUBMIT`（最终提交）等明确状态进行操作。
    -   **信息不对称**：评估者（Evaluator）持有私有的验证集标签和隐藏测试集。智能体只能看到训练集（含标签）和验证集特征，从而杜绝其直接拟合验证集。
    -   **可复现的提交要求**：最终提交必须是一个可调用`predict_fn(raw_dataframe)`函数或一个完整的`Pipeline`对象，确保整个工作流能在原始数据上重放。

-   **隐蔽的多维验证器套件**：
    -   评估并非由一个单一的评分函数完成，而是由一组独立的、隐蔽的验证器组成，覆盖五大类指标：
        1.  **完整性与防泄漏**：检查目标变量是否被用作特征、是否在验证/测试集上训练、以及是否存在经典的 `train+validation` 重拟合等关键错误。
        2.  **数据质量**：检查缺失值、重复值等。
        3.  **建模规范**：确保候选模型可复现、预处理步骤在Pipeline内。
        4.  **流程规范**：评估规划覆盖率、迭代行为等。
        5.  **最终比较**：与一个简单的基线模型和一个未经调优的`HistGradientBoosting`预言机模型进行对比，以标准化最终得分。

-   **结构化反馈与奖励信号**：
    -   **分级反馈**：智能体不会直接看到私有的验证分数或检查清单。它仅在`VALIDATE`行动后获得一个基于验证集的门控指标，并针对未满足的过程清单项，获得抽象的、引导性的提示，而非具体的检查代码。
    -   **分解式奖励函数**：最终奖励 `r_final` 由预测性能项（0.55）、规划覆盖率（0.15）和代码质量（0.30）加权组成，公式如下：
        `w = 0.55 * r_perf + 0.15 * r_plan + 0.30 * r_code`
        任何关键方法论错误（如目标泄露）会强制将最终奖励归零。这种设计让过程奖励成为任务质量的代理信号，但无法替代最终的隐藏测试评估。

-   **核心交互机制：灵活迭代**：被称为“灵活迭代”`flexible_iterative`的模式是GRACE-DS的精髓。它允许智能体在反馈驱动下自主选择工作流阶段，但其行为受到“候选模型优先”等纪律的约束（如：必须先生成一个简单可复现的基线模型，才能进行深度探索）。这种机制旨在评估智能体的自我纠错和迭代优化能力。

### 3. 实验设计

-   **数据集**：
    -   使用了**10个类似生产环境的表格型任务**，覆盖四个类别：
        1.  **TML-bench**：4个截止期后的Kaggle竞赛数据集（回归和分类）。
        2.  **TabReD**：1个带有官方时间划分的行业数据集（回归）。
        3.  **Benchmark**：3个来自UCI/OpenML的不常见基准（分类和回归），降低记忆化风险。
        4.  **Synthetic**：2个由scikit-learn生成的合成数据集，具有已知的特征相关性真值。
    -   这些数据集的多样性确保了评估的全面性，并且由于使用了截止期后数据和合成数据，**有效防止了智能体通过记忆公共答案来获得高分**。

-   **基准与对比方法**：
    -   论文将`flexible_iterative`（灵活迭代）作为核心方法与一系列基线机制进行了全面的对比，包括：
        -   **单次生成（single_shot）**：仅一次生成模型并提交。
        -   **无结构智能体（unstructured_agent）**：持续交互，但无显式工作流阶段标签和阶段化反馈。
        -   **从零开始重启（restarts_from_scratch）**：多次独立完成“单次生成”任务。
        -   **调用次数匹配的重启上限（restarts_call_matched）**：确保用于比较的重启基线获得了与迭代机制等量或更多的LLM调用次数。
    -   同时，还设计了消融实验（移除`PLAN`, `EDA`, `FE`等阶段）和压力测试（奖励最大化智能体、红队攻击）来评估各个组件的贡献和系统的鲁棒性。

-   **评估指标**：
    -   **E2E Q（端到端质量）**：核心指标，计算所有回合的标准化隐藏测试得分，将未完成的回合计为0。
    -   **Obs. Q（观测质量）**：仅统计成功提交的回合的平均得分。
    -   **PV（协议有效完成率）**：成功提交一个可复现、无违规、未被强制修复的解决方案的回合比例。
    -   **分解奖励和错误统计**：用于解释性能背后的原因，如关键错误数（泄漏等）、错误恢复率等。

### 4. 资源与算力

-   **LLM推理**：所有8个前沿LLM均通过**OpenRouter API远程调用**，未提及本地部署大模型所需的GPU。
-   **沙箱执行环境**：所有的模型训练、验证和测试脚本在CPU上运行。论文明确说明使用了一台配备**AMD Ryzen 7 5700X CPU**和**64 GB RAM**的机器，并行Worker数为**8**。

### 5. 实验数量与充分性

实验规模庞大且设计严谨，具有统计说服力：

-   **核心实验**：覆盖**8个LLM × 10个任务 × 7种核心机制**，每个配置下运行 **6个回合**，共产生 **480个回合每机制**，总计 **3360个回合**。
-   **扩展实验**：还包括5个消融机制（480回合/个）、2个奖励优化探针（480回合/个）以及红队测试网格。
-   **统计显著性**：所有配对对比（如灵活迭代 vs. 基线）均使用了**Wilcoxon符号秩检验和配对自助法置信区间**，确保了结论的统计可靠性。
-   **公平性**：通过固定所有随机种子（数据划分、LLM采样）、统一每回合行动和Token预算、以及设计“调用次数匹配”的基线，确保了不同机制之间的对比是公平和客观的。

### 6. 主要结论与发现

-   **结构化迭代机制显著优于所有基线**：`flexible_iterative`在端到端质量（E2E Q）和协议有效完成率（PV）上均大幅领先。其E2E Q达到0.754，对比单次生成（0.536）、无结构智能体（0.527）和调用次数匹配的重启上限（0.686）均有显著提升。
-   **结构化反馈能有效遏制关键错误并提升鲁棒性**：灵活迭代机制将致命性错误（如目标泄露）爆发的回合数降至接近零（1/480），而重启基线则出现了5.2%到7.1%的泄漏问题。同时，它将大多数失败回合成功转化为有效提交，展现了强大的纠错能力。
-   **过程中奖励是有效的进度信号，但不能替代最终评估**：奖励优化实验显示，尽管“奖励最大化”智能体可以小幅提升过程奖励，但这是以牺牲隐藏测试集上的真实质量为代价的。
-   **模型规模与机制有效性存在交互作用**：对于能力较弱的开放权重模型，僵化的、固定阶段的工作流会导致其协议完成率崩溃（如Qwen3-Next-80B降至38.3%），而灵活迭代机制则具备更好的鲁棒性和能力均衡作用。

### 7. 优点

-   **面向生产的深度评估范式**：区别于传统的“唯分数论”，GRACE-DS从流程合规、数据安全、代码可复现、自我纠错等多维度评估智能体，更贴近企业真实需求。
-   **精巧的隐蔽审计设计**：通过评估者持有私有标签、隐藏验证器具体实现、仅提供抽象提示等设计，巧妙地避免了智能体通过“刷榜”或模式匹配来欺骗评估系统，迫使其必须真正理解数据和方法论。
-   **高度可解释的诊断能力**：通过对奖励的分解、最终路径的拆解以及错误和恢复的统计，GRACE-DS不仅能给出“好不好”的结论，更能解释“为什么好”或“为什么坏”，使评估结果极具可操作性。
-   **实验全面性和严格性**：在多模型、多任务、多机制上进行了超过7000个回合的测试，并采用了严谨的统计检验和有差别的基线对比（如调用次数匹配），保证了结论的坚实性。

### 8. 不足与局限

-   **任务范围局限**：当前仅限于**表格型监督学习**任务，尚未扩展到深度学习、多模态、强化学习或开放性科学发现等AutoML同样活跃的领域。
-   **简化的模型搜索**：实验过程中明确指示并验证了智能体仅使用少量手动设定的模型配置，排除了自动超参数搜索。这可能低估了现实场景中智能体通过自动搜索发现更优模型的能力。
-   **缺乏美元成本分析**：虽然讨论了延迟、Token消耗和效率比，但并未将其转换为具体的API调用费用，使得成本效益权衡不够直观。
-   **计算资源异构性问题**：LLM是通过API远程调用的，其响应延迟和推理速度受API提供商影响很大，这使得不同模型之间的“时间-质量”对比可能包含了网络与服务端的干扰，不完全反映模型本身的效率。此外，沙箱仅用CPU，对于评估需要GPU的深度学习方法不够充分。
