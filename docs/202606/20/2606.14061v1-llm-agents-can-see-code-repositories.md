## 摘要
基于大语言模型的编码智能体在软件工程任务中表现出色。然而，大多数智能体几乎完全以文本形式处理代码仓库，这与人类开发者利用文件夹层次结构和依赖关系等视觉结构在大型代码库中定位有所不同。随着多模态大语言模型的出现，智能体能否有效利用代码仓库的视觉表示成为一个开放问题。本文首次对基于大语言模型的智能体在仓库级问题解决中使用视觉仓库表示进行了系统的实证研究。我们评估了四种最新的多模态模型。结果表明，纯视觉设置会降低准确率并增加 token 消耗，因为智能体缺乏足够的符号细节，转而通过重复的视觉查询来弥补。相比之下，将仓库结构的视觉图作为补充模态与标准文本界面相结合，可以帮助智能体更高效地理解结构：输入 token 消耗最多降低 26%，同时问题解决准确率保持不变或有所提升。可视化在故障定位以及智能体自主控制探索深度时最为有用。这些发现为下一代编码智能体提供了一种实用的文本与视觉混合设计。

## Abstract
Coding agents powered by large language models have demonstrated strong performance on software engineering tasks. Yet most agents consume repositories almost entirely as text, which differs from how human developers use visual structure such as folder hierarchies and dependency relationships to orient themselves in large codebases. With multimodal large language models (MLLMs), it is an open question whether agents can effectively benefit from visual representations of repositories. This paper presents the first systematic empirical study of visual repository representations for LLM-based agents on repository-level issue resolution. We evaluate four recent multimodal models. Our results show that a strictly vision-only setup degrades accuracy and increases token cost, because agents lack sufficient symbolic detail and compensate with repeated visual queries. In contrast, integrating visual graphs of repository structure as a supplementary modality alongside standard text interfaces helps agents understand structure more efficiently: input token consumption decreases by up to 26% while issue-resolution accuracy is maintained or improved. Visualization is most useful during fault localization and when the agent autonomously controls exploration depth. These findings point to a practical hybrid text-and-vision design for next-generation coding agents.

### 2. 论文提出的方法论

-   **核心思想**：提出一种**混合模态**的代码仓库理解框架，即同时向智能体提供代码文本和可视化的仓库结构图。文本模态用于精确的符号推理，而视觉模态用于提供空间布局、依赖方向和模块分组的全局结构感知。
-   **框架实现**：构建了一个名为 `SeeRepo` 的实验框架，其工作流程如下：
    1.  **静态图构建**：利用基于抽象语法树（AST）的静态分析，为代码仓库构建一个有向异构依赖图 `G = (V, E, A, R)`。节点 `V` 表示文件、类、函数等实体，边 `E` 捕捉四种类型的关系 `R`: `contains`（包含层级）、`imports`（模块依赖）、`inherits`（类继承）和 `invokes`（函数调用）。
    2.  **可视化查询与渲染**：智能体在推理时可主动查询该依赖图。给定一个查询节点和遍历深度，系统会构建一个以该节点为中心、上游和下游关系均有覆盖的距离感知子图。
    3.  **图像生成**：该子图通过 Graphviz 引擎渲染为PNG图像。图像中采用不同图标区分实体类型（文件、模块等），用箭头表示依赖方向，并使用高亮突出查询节点，从而将空间拓扑信息以视觉 token 的形式传递给 MLLM。
    4.  **多模态输入**：生成的视觉 token 序列与文本 token 拼接，共同输入到 MLLM 的 Transformer 架构中进行处理。

### 6. 论文的主要结论与发现

-   **发现 1：纯视觉交互效果不佳。**
    -   将所有仓库信息转为图像提供给智能体，会显著降低所有模型的准确率（最高下降 34.1 个百分点），并导致 token 成本飙升（最高增加 268%）。智能体因缺乏精确的符号信息而进行大量重复、无效的图像查询。
-   **发现 2：混合模态集成能提效降本。**
    -   将 `SeeRepo` 生成的视觉结构图作为文本界面的补充，能够在保持或略微提升准确率的同时，显著降低交互成本。输入 token 消耗最高可减少 **26%**，总成本最高可降低 **46%**。
-   **发现 3：图状布局与动态深度最具效率。**
    -   在所有视觉布局中，`图状（Graph）` 布局在 token 效率和成本节省上表现最佳。
    -   由智能体自主决定遍历深度的 `动态深度` 策略，比任何固定深度设置都能实现更优的效能与成本平衡。
-   **发现 4：可视化在故障定位阶段最有效。**
    -   在早期代码探索和故障定位阶段引入视觉结构信息，能帮助智能体缩小搜索范围并减少盲目导航。若在后期修复或验证阶段引入，反而会因无关信息干扰而降低准确率。

### 7. 优点

-   **首创系统研究**：首次对 MLLM 在软件工程仓库级任务中的视觉表示效果进行了系统、大规模的实证分析，填补了研究空白。
-   **实践指导性强**：结论清晰且具有强烈的实践指导意义，明确指出了“文本+视觉”混合路径的价值、最优的视觉图样式以及最佳的应用时机。
-   **框架设计合理**：`SeeRepo` 框架的设计符合人类直觉，它并未试图用图像取代代码，而是用图像呈现结构，用文本承载细节，充分发挥了两种模态的优势。
-   **评估全面细致**：实验不仅关注最终的有效性（Pass@1），还深入分析了 token 消耗、API 调用次数等效率指标，全面揭示了多模态交互的成本效益权衡。

### 8. 不足与局限

-   **外部有效性受限**：实验主要在 `SWE-bench Verified` 的 Python 仓库上进行，结论是否能推广到其它编程语言（如 Java、TypeScript）或具有不同架构风格的项目中尚不明确。
