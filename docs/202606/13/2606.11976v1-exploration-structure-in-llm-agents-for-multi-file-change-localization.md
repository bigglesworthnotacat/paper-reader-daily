## 摘要
软件工程工具越来越依赖基于LLM的智能体来定位需要修改的文件以解决软件问题。大多数AI智能体以线性方式探索代码仓库，即每一步访问一个目录或文件。我们假设这对于跨越多个子系统的变更是一种结构上的不匹配。我们比较了线性顺序探索与非线性、领域范围的并行智能体探索。使用SWE Bench Pro作为初始基准，我们以ansible为例。我们构建了一种方法，用于在单个基准提交上对GitHub议题进行持久会话评估。我们将非线性领域智能体文件遍历系统与没有直接仓库访问的基础LLM、具有持久Python REPL的单一智能体递归语言模型（RLM）基线以及使用Codex 5.5 High的外部CLI基线进行了比较。使用小型Haiku类模型的领域范围并行智能体派生在Haiku类模型中取得了大幅领先的最高微观F1分数。在我们自己扩展的基准上（包含2025和2026年更新的PR），领域智能体得分第二高，仅次于更大的Codex 5.5 High。在原始的、精选的2020年SWE-bench Pro基准上，更大的Sonnet普通LLM基线通过预测少量文件获得了更高的微观F1，从而提高了精确率，但所有金标准召回率显著降低。我们还提出了三个额外发现。首先，文档演变是一种所有方法都无法解决的潜在依赖。其次，天真的文件系统访问会因测试文件过度预测而降低定位性能。最后，强制多智能体协商没有明显帮助，并大幅增加了令牌成本。

## Abstract
Software engineering tools increasingly rely on LLM based agents to localize files to change to resolve a software issue. Most AI agents explore repositories linearly, that is, visiting one directory or file per step. We postulate that this is a structural mismatch for changes that span several subsystems. We compare linear sequential exploration against non-linear, domain-scoped parallel agentic exploration. Using SWE Bench Pro as initial benchmark, we focus on ansible as an exemplar. We construct an approach for persistent-session evaluation of GitHub issues anchored at a single base commit. We compare our non-linear domain-agent file traversal system against a base LLM without direct repository access, a single agent Recursive Language Model (RLM) baseline with a persistent Python REPL and an external CLI baseline using Codex 5.5 High. Domain scoped parallel agent spawning with a small Haiku-class model achieves the highest micro F1 among Haiku class models by a large margin. Domain-agents is the second highest behind only the much larger Codex 5.5 High on our own expanded benchmark including over more recent PRs from 2025 and 2026. On the original, curated, 2020 SWE-bench Pro benchmark, a larger Sonnet plain LLM baseline attains higher micro F1 by predicting few files, leading to higher precision, but at significantly lower all gold recall. We also present three additional findings. First, documentation evolution is a latent dependency unresolved by any approach. Second, naive file system access can degrade localization driven by test-file over prediction. Lastly, forced multi-agent consultation does not measurably help and raises token cost substantially.

## 2. 论文提出的方法论
*   **核心思想**：将代码仓库按**领域子系统**（如 `cli/`、`module_utils/`、`docs/docsite/` 等）进行划分，为每个领域创建一个**持久化的专属智能体**。在查询时，由一个根协调器自适应地并行派发相关领域的智能体进行探索，实现非线性、领域范围的并行定位。
*   **关键技术细节与流程**：
    *   **系统架构**：
        *   **根协调器**：初始化时探索仓库结构，识别并注册各领域智能体；查询时分析议题，自适应地派发相关领域智能体。
        *   **领域智能体**：每个智能体负责一个划定的子系统，进行预熟悉（familiarization），并维护一个可复用的知识注册表。
        *   **并行派发**：查询阶段，相关智能体并行探索各自区域，返回候选文件及理由，由协调器合并过滤。
    *   **查询策略变体**：
        *   **自适应派发**：协调器仅派发议题文本表明相关的领域智能体。
        *   **强制派发（nudged）**：作为消融对照，强制更积极地派发智能体，以测试协同策略的影响。
        *   **无派发（no-spawn）**：仅保留注册表与协调器，禁用并行派发，用于隔离架构开销的效果。
    *   **有界仓库I/O**：为避免超大文件撑爆上下文窗口，采用程序化环境（类似REPL）存储文件内容，仅将压缩预览、搜索片段或指定行范围送入LLM上下文，显著降低令牌消耗（对大型源文件降幅高达97.6%）。目录列表亦采用紧凑的相对树格式，节省约41%的输入令牌。
*   **对比基线方法**：
    *   **纯LLM**：无仓库访问权限，完全依赖模型自身知识。
    *   **单体递归语言模型**：拥有持久Python REPL进行顺序文件系统访问。
    *   **外部CLI**：使用 **Codex 5.5 High** 作为前沿模型基线。
    *   **经典信息检索**：**BM25**，对 `K`（预测文件数）进行1-30的参数扫描，形成性能包络线。

## 6. 论文的主要结论与发现
*   **探索结构的优越性**：使用小型 Haiku 模型的**领域并行智能体方案**，在Haiku级别模型中大幅领先，其微观F1分数在所有方法中位列第二，仅次于模型规模大得多的 Codex 5.5 High。这证明有效的探索结构比单纯扩大模型规模更关键。
*   **大型模型的行为差异**：大型 Sonnet 纯LLM通过预测极少数文件获得了最高的微观F1分数（高精度），但其全部金标召回率非常低，属于一种**高精度-低召回**的极端运行点，且此优势在后期时间窗口消失。
*   **文件系统访问的双刃剑**：天真的文件系统访问（如单体RLM）会**损害定位性能**。因为该方式易导致智能体预测出大量无关的测试文件（在精选金标下为误报），而领域代理通过**有界I/O**和**并行派发**实现了更好、更有针对性的探索。
*   **多智能体协同的无效性**：强制、更激进的派发（nudged变体）**并未显著提升召回率**，反而增加了来自无关子系统的噪声，并大幅推高了令牌成本。
*   **文档共演化的普遍失败**：所有方法都系统性地**无法召回需要修改的文档文件**。这是一个因议题描述缺失而导致的潜在依赖问题，属于逻辑耦合路由挑战，而非单纯的探索能力问题。
*   **跨时间窗口的泛化性**：自适应领域代理方案在2025和2026年的数据窗口上持续展现出强竞争力，证实了其结构优势的持久性。
