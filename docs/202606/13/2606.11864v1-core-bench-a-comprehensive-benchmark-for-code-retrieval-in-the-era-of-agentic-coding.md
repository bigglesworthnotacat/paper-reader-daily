## 摘要
代码检索正成为编码智能体的核心，但智能体编程需要的不仅仅是将自然语言查询与孤立代码片段匹配。给定用户请求，编码智能体需要遍历具体的仓库状态，定位相关文件和函数，收集支持性上下文，并过滤仓库内类似的干扰项。现有的代码检索基准主要评估文档字符串到函数或片段级别的匹配，因此遗漏了这一需求驱动的仓库搜索问题。为弥补这一差距，我们引入了CORE-Bench，一个面向智能体编程时代的代码检索综合基准。CORE-Bench在三个层面上评估代码检索能力：代码理解、问题到编辑的定位以及更广泛的上下文检索。基于精心整理的代码搜索任务和SWE-bench系列实例，CORE-Bench包含超过18万条查询和10.6万个更广泛上下文相关性标签。代表性嵌入模型的实验表明，从传统代码搜索到智能体编程环境下的代码检索，性能急剧下降。对现有嵌入模型进行简单的监督微调可显著提升此环境下的性能，这表明仍有巨大的进步空间。

## Abstract
Code retrieval is becoming central to coding agents, but agentic coding requires more than matching a natural-language query to an isolated snippet. Given a user request, a coding agent needs to navigate a concrete repository state, locate relevant files and functions, gather supporting context, and filter similar in-repository distractors. Existing code retrieval benchmarks mainly evaluate docstring-to-function or snippet-level matching, thereby missing this requirement-driven repository search problem. To address this gap, we introduce CORE-Bench, a comprehensive benchmark for code retrieval in the era of agentic coding. CORE-Bench evaluates code retrieval ability at three levels: code understanding, issue-to-edit localization, and broader context retrieval. Built from curated code-search tasks and SWE-bench-series instances, CORE-Bench contains over 180K queries and 106K broader-context relevance labels. Experiments with representative embedding models show a sharp drop from traditional code search to code retrieval in agentic coding settings. Simple supervised fine-tuning of existing embedding models significantly improves performance in this setting, suggesting substantial room for further progress.

*   **整体含义**：论文提出**CORE‑Bench**，一个专为智能体编程时代设计的代码检索综合基准。该基准旨在揭示传统代码理解与真实仓库级检索之间的巨大性能鸿沟，并引导模型朝着更实用、更能服务于编码智能体的方向发展。

### 2. 论文提出的方法

CORE‑Bench 构建了一个三层递进式评测体系，其核心思想是 **“需求驱动的仓库级检索”**。

*   **Level‑1：代码理解**  
    保留传统、具有挑战性的代码检索任务（如文本‑代码、代码‑文本、代码‑代码），筛选去除了饱和或与真实检索脱节的数据集，作为基础代码语义匹配能力的校准。

*   **Level‑2：问题到编辑的定位 (Issue‑to‑Edit Localization)**  
    *   **数据来源**：从 SWE‑bench 系列（Verified, Pro, Live 等）中提取真实的 PR/Issue 描述作为查询。
    *   **构建流程**：将仓库**回退到 PR 合并前的 commit**，使用 AST 和 LangChain 对源文件、文档进行分块（chunk）。然后，将**补丁 (patch)** 修改的行与分块对齐，将匹配到的分块作为**正相关文档 (qrels)**。这实现了“问题描述 → 待编辑代码块”的直接映射。
    *   **过滤与清洗**：利用 LLM 过滤掉包含修复答案或信息量不足的查询，只保留不含答案泄漏的有效请求。

*   **Level‑3：更广泛的上下文检索 (Broader Context Retrieval)**  
    超越单一的编辑位置，评估模型能否检索到实现需求所需的**辅助上下文**（测试、帮助函数、配置、相关实现路径）。
    *   **自动化标注流水线**：首先让编码智能体（mini‑SWE‑Agent）尝试解决该问题，并从其**浏览/读取文件的轨迹**中提取候选分块。然后通过**三个 LLM 裁判（多数投票）** 对候选块进行相关性分级（high / partial / irrelevant）。
    *   **质量验证**：通过限制智能体只能读取标注出的 Level‑2 和 Level‑3 文件，重新执行任务。若限制 Level‑3 文件的成功率更高，则证明提取的上下文是**功能性有用**的。
    *   **反馈迭代**：对未解决的案例重复执行，并补充新发现的上下文，直到不再产生新内容或达到预算。

*   **查询重写变体**：为贴合真实开发者向 AI 助手提问的习惯，提供了一份将原始详细 PR 描述**压缩重写**为更简洁、口语化查询的变体，保留了关键技术术语和错误信息。

*   **域内监督微调数据构建**：从**非 SWE‑bench 仓库的 Pull Request** 中收集数据，以 PR 描述为查询，补丁对齐的分块为正样本，同一仓库内的其他分块为负样本，用于训练检索模型。

    *   **评测变体**：原始查询版和查询重写版（Level‑2/3）。同时按编程语言、仓库难度（聚焦/密集、小/大）和查询意图（bug/功能/重构等）进行了细分分析。

### 6. 论文的主要结论与发现

*   **性能断层**：现有最先进的通用和代码嵌入模型在从 Level‑1（传统代码搜索）转移到 Level‑2/3（智能体仓库检索）时，性能出现**急剧恶化**。例如，Qwen3‑8B 的 NDCG@10 从 Level‑1 的 71.7 骤降至 Level‑2 的 20.3 和 Level‑3 的 34.4。
*   **域内微调有效但尚未解决难题**：使用 GitHub PR 数据进行简单的监督微调，能**显著且稳定地提升** Level‑2 和 Level‑3 的性能，甚至让小模型（如 Qwen3‑0.6B‑SFT）超越非微调的大模型。但微调后模型在 Level‑2/3 与 Level‑1 之间仍存在巨大差距，说明问题未被完全解决。
*   **查询重写未必有益**：将原始 issue 重写为简洁的开发者查询，对许多模型而言**可能导致性能不升反降**。这表明真实的仓库检索依赖于原始请求中的特定上下文线索（如具体路径、变量名），压缩可能丢失关键信号。
*   **跨语言与难度分析**：
    *   性能改进不限于 Python，在 JavaScript、TypeScript、Go 上都尤其明显，但 Java 依然是较难的任务。
    *   在不同难度仓库中，密集大型仓库是未来检索器的重要**压力测试**，即使在微调后，高召回率在此场景下依然容易丢失。
*   **Level‑3 的独特性**：Level‑3 并非 Level‑2 的宽松版本，而是评估对**广泛上下文恢复**的能力。案例显示，完成一个简单修改可能需要数十个支撑性上下文块（如迁移图、解析器状态、客户端配置等）。

### 7. 优点

*   **场景新颖且切中要害**：精准定义了“需求驱动的仓库级搜索”这一被忽略的重要问题，比现有片段匹配式基准更贴近编码智能体的实际需求。

### 8. 不足与局限

*   **数据来源偏差**：基准主要基于 SWE‑bench 系列的数据，可能**无法完全代表**新创建的、快速演进的仓库或非开源私有仓库中的检索模式。对于长尾编程语言和非常规项目结构的覆盖也相对不足。
