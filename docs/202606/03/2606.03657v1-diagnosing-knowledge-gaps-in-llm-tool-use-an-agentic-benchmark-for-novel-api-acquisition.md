## 摘要
用于代码生成的大型语言模型经常需要使用预训练数据中缺失的API。这不仅需要回忆函数名称：模型必须协调签名、模块路径、输入输出契约、语义和可执行的用法模式。现有的新API基准通常是静态的，依赖粗糙的通过/失败指标，或使用可能无法反映真实库演变的合成API。我们引入了NovelAPIBench，这是一个完全自动化的动态基准，针对任何基座模型和目标库，发现新的API，提取分解后的知识束，生成可执行的编码任务，并将失败样本归类到六个诊断类别中。在约1.9K任务、四个基座模型和五个领域上，我们比较了通过检索注入的知识与通过参数化适应内化的知识。我们发现知识组件不可互换：用法示例是最强的独立信号，而最佳的双组件设置是签名与机制或示例配对，具体视领域和骨干模型而定。添加更多上下文，尤其是源代码，可能会因增加导入路径错误而有害。一旦外部知识被移除，参数化适应也无法取代检索；相反，微调主要教会模型如何使用提供的知识束，并且这种能力可以迁移到未见的库中。这些结果表明，检索和微调发挥互补作用：检索提供易变的API内容，而微调改善程序性整合。

## Abstract
Large language models for code generation often need to use APIs that are absent from their pretraining data. This requires more than recalling a function name: models must coordinate signatures, module paths, input-output contracts, semantics, and executable usage patterns. Existing novel-API benchmarks are typically static, rely on coarse pass/fail metrics, or use synthetic APIs that may not reflect real library evolution. We introduce NovelAPIBench, a fully automated dynamic benchmark that, for any base model and target library, discovers novel APIs, extracts decomposed knowledge bundles, generates executable coding tasks, and assigns failed samples to six diagnostic categories. Across about 1.9K tasks, four base models, and five domains, we compare knowledge injected through retrieval with knowledge internalized through parametric adaptation. We find that knowledge components are not interchangeable: usage examples are the strongest standalone signal, while the best two-component setting pairs signatures with either mechanisms or examples depending on the domain and backbone. Adding more context, especially source code, can hurt by increasing import-path errors. Parametric adaptation also does not replace retrieval once external knowledge is removed; rather, fine-tuning mainly teaches models how to use provided bundles, and this ability transfers to held-out libraries. These results suggest that retrieval and tuning play complementary roles: retrieval supplies volatile API content, while tuning improves procedural integration.

### 2. 论文提出的方法论

论文提出了 **NovelAPIBench**，一个完全自动化、模型条件化的动态基准测试框架。

*   **核心思想**：
    *   **动态生成与诊断**：该框架不是静态数据集，而是针对任何给定的基座模型和目标库，动态生成编码任务，并对模型的失败进行细粒度的失败归因。
    *   **知识分解**：将API知识拆解为不同的组成部分，以孤立地研究各类知识对模型性能的影响。

*   **关键技术细节与流程**：
    *   **四阶段流水线**：
        1.  **新颖API发现（Stage 1: Novel API Discovery）**：依据模型的预训练数据截止日期，对比一个库的新旧版本，通过`inspect`和`AST`解析其公开API表面，自动找出在旧版本中不存在、在新版本中出现的新API，确保它们在模型的知识范围之外。
        2.  **知识提取（Stage 2: Knowledge Extraction）**：为每个新API提取一个**分解后的知识束**，包含以下组件：
            *   **表面信息（S）**：全限定名 (`S_name`) 和带类型、默认值、描述的参数签名 (`S_param`)。
            *   **示例（E）**：从官方文档和真实GitHub仓库中挖掘的规范用法示例。
            *   **机制（M）**：包含自然语言的算法或设计描述 (`M_prose`)，以及API的实现源代码 (`M_code`)。
        3.  **任务生成（Stage 3: Task Generation）**：使用单个LLM调用，为每个知识束生成三个难度级别（简单、中等、困难）的编码任务和对应的可执行测试套件。套件采用“执行后断言”策略，并注入目标API调用计数器作为通过测试的硬性前置条件。
        4.  **质量与新颖性过滤（Stage 4: Quality and Novelty Filtering）**：通过三个过滤器进行模型条件化筛选：`C1`保证参考答案能通过测试，`C2`保证目标基座模型在无外部知识下大概率失败，`C3`保证强LLM在获得完整知识后能解决该任务，确保任务可解。
    *   **自动化失败分类法**：对于每个失败的`pass@1`样本，自动将其归入六个互斥的诊断类别：
        1.  `WrongAPISelection`：未调用目标API，选错了函数。
        2.  `WrongImport`：选对了API但导入路径错误。
        3.  `WrongSyntax`：生成的代码有语法错误。
        4.  `WrongParam`：调用正确，但参数（命名/类型）错误。
        5.  `WrongShapeDtype`：调用正确，但在API内部的数据形状/类型上出错。
        6.  `WrongLogic`：API调用正确，但周围的程序逻辑错误。

### 6. 论文的主要结论与发现

*   **知识组件不可互换（Finding 1-3）**：
    *   用法示例（`E`）是最强的独立知识组件。表面信息（`S`，尤其是签名）是解决`WrongAPISelection`错误的最可靠锚点。
    *   签名（`S`）与机制描述（`Mprose`）的组合效果最佳，因为`Mprose`有助于解决逻辑错误。
    *   源代码（`Mcode`）是导致负面知识干扰的主要来源，它会显著增加`WrongImport`错误，因为引入了模块路径噪声。

*   **任务与模型影响知识偏好（Finding 4-6）**：
    *   对于模板化任务，示例（`S+E`）更有帮助；对于异构任务，机制描述（`S+Mprose`）更稳定。
    *   最优知识组合偏好与骨干模型的强弱项相关。签名能力强的模型从`S+Mprose`中获益，而示例使用能力强的模型则从`S+E`中获益更多。
    *   具备推理能力的模型（如R1-Distill）能更好地抵抗`Mcode`引入的导入路径噪声。

*   **参数化适应的根本性质（Finding 7-9）**：
    *   一旦在推理时移除外部知识束，没有任何参数化适应方法能真正**内化**新API知识，其表现远低于基于检索（直接提供知识束）的推理。
    *   微调主要学习的是一种可迁移的程序性**元技能**，即“**如何有效利用提供的知识束**”（如将API符号映射到正确的模块路径），而不是记忆库特定的API细节本身。这种技能可以跨库迁移，但库特有的模块路径知识不可迁移。

*   **核心洞见**：检索（或RAG）和参数化微调在使LLM适应新API方面扮演着**互补角色**：**检索**负责提供易变的、事实性的API内容；**微调**则负责改善模型操作这些知识的程序性整合能力。

*   **语言和范围限制**：
    *   **仅限Python**：整个基准构建在19个Python库之上，其发现是否适用于其他语言（如Rust, TypeScript）的程序库未知。
    *   **单API任务**：每个任务都仅针对单一API。现实中的智能体工作流可能涉及链式调用多个新API，该基准未覆盖这种组合使用场景。
*   **结论的边界条件**：
    *   **算力受限的“无内化”结论**：作者明确指出，关于参数化内化失败的结论受限于他们所采用的“事后适应”范式（如LoRA微调3轮、约1500个训练任务）。结论未排除通过更大规模的训练、全参数微调或基于执行奖励的强化学习等方法实现API知识内化的可能性。
*   **潜在的评估噪音**：
    *   **自动化分类器的混淆**：`WrongImport/WrongSyntax`和`WrongShapeDtype/WrongLogic`这两对类别在自动分类中存在一定混淆，约5%以下的精细差异应谨慎解读。
    *   **测试套件偏差**：约10%的自动化测试套件在人工审计中被标记为可能过于严格或不严格，这会引入评分的“假阴性”或“假阳性”。
