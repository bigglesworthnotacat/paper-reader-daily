## 摘要
大语言模型（LLM）代理正从请求-响应助手演进为长时运行的软件执行体：它们在模型调用之间保持状态，分叉子任务，等待外部事件，请求人工授权，生成工具，并执行需要恢复和审计的副作用。本文提出 Agent libOS，一种受库操作系统启发的 LLM 代理运行时基座。Agent libOS 运行在传统宿主操作系统之上；它不实现硬件驱动、内核模式隔离或 POSIX 兼容的操作系统。相反，它将代理视为一个 AgentProcess：一个可调度的执行主体，具有进程标识、父子谱系、生命周期状态、从 AgentImage 派生的工具表、类型化对象内存、显式能力、人工队列、检查点、事件和审计记录。其核心设计原则是工具是类似 libc 的封装器；运行时原语是权限边界。文件系统访问、对象访问、休眠、人工批准、即时工具注册和外部副作用都在显式能力和策略下在原语边界进行检查。我们描述了设计、威胁模型、Python 原型和面向安全的评估。当前原型实现了异步调度、命名空间本地对象内存、运行时集成的人工批准、一次性权限授予、每进程工作目录、shell 和镜像注册原语、通过 libOS 系统调用代理的 Deno/TypeScript 即时工具、文件系统/对象桥接工具、可注入的资源提供者基座、确定性演示、真实模型冒烟脚本，以及在撰写时 123 个回归测试。Agent libOS 并非旨在提高规划器准确性，而是展示了一种运行时基座，其中长时运行的 LLM 代理可以被调度、授权、恢复和审计，而无需将工具分发视为信任边界。

## Abstract
Large language model (LLM) agents are evolving from request-response assistants into long-running software actors: they maintain state across model calls, fork subtasks, wait for external events, request human authority, generate tools, and perform side effects that must be resumed and audited. This paper presents Agent libOS, a library-OS-inspired runtime substrate for LLM agents. Agent libOS runs above a conventional host operating system; it does not implement hardware drivers, kernel-mode isolation, or a POSIX-compatible operating system. Instead, it treats an agent as an AgentProcess: a schedulable execution subject with process identity, parent-child lineage, lifecycle state, a tool table derived from an AgentImage, typed Object Memory, explicit capabilities, human queues, checkpoints, events, and audit records. Its central design rule is tools are libc-like wrappers; runtime primitives are the authority boundary. Filesystem access, object access, sleeps, human approval, JIT tool registration, and external side effects are checked at primitive boundaries under explicit capabilities and policy.   We describe the design, threat model, Python prototype, and safety-oriented evaluation. The current prototype implements async scheduling, namespace-local Object Memory, runtime-integrated human approval, one-shot permission grants, per-process working directories, shell and image-registration primitives, Deno/TypeScript JIT tools over a libOS syscall broker, filesystem/object bridge tools, an injectable Resource Provider Substrate, deterministic demos, real-model smoke scripts, and 123 regression tests at the time of writing. Rather than improving planner accuracy, Agent libOS demonstrates a runtime substrate in which long-running LLM agents can be scheduled, authorized, resumed, and audited without treating tool dispatch as the trust boundary.

## 1. 核心问题与整体含义
- **研究动机**：LLM 代理正从单次问答助手演进为长期运行的软件角色，需要跨模型调用维护状态、派生子任务、等待外部事件、请求人工审批、生成动态工具并执行需追溯的副作用。现有框架以“聊天循环 + 工具注册表”为核心抽象，将模型可见的工具接口与宿主资源的实际权限混为一体，导致工具分发成为信任边界，在间接提示注入、工具输出注入、混淆代理等攻击下极易失效。
- **整体含义**：Agent libOS 借鉴库操作系统（library‑OS）的设计哲学，将代理建模为操作系统中的进程，通过类似 libc 的工具包装将模型可见动作与运行时原语权限解耦，从而把权限检查、策略执行、人工等待和审计下沉到运行时原语边界，为长时、安全、可审计的 LLM 代理提供基座。

## 2. 方法论
- **核心思想**：工具作为“libc 式封装”，运行时原语作为“权限边界”。将代理执行主体抽象为 `AgentProcess`，拥有进程标识、父子谱系、生命周期状态、映像派生的工具表、类型化对象内存、显式能力、人工队列、检查点、事件和审计记录。模型可见的工具表只决定“可见性”，实际资源访问需通过原语层的能力检查。
- **关键设计点**：
  - **进程模型**：支持 `spawn`、`fork`、`exec`、`wait`、信号、暂停、恢复和退出。`fork` 衰减权限和内存视图，`exec` 不自动授予新映像的所需能力，确保权限不升级。
  - **对象内存**：类型化、带能力保护的对象图，命名空间隔离。名称不是能力，需要命名空间能力和对象能力双重检查。内存视图在模型调用前由物化器转为有界文本上下文，避免模型直接接触存储。
  - **能力与策略**：能力绑定主体、资源、权限、约束、有效期等，在每次原语调用时实时校验。文件系统写入支持 `ask_each_time`、`allow_once` 等策略，一次性权限在单次成功调用后消耗。
  - **人工审批**：作为阻塞式原语集成在运行时内，由调度器统一管理等待与唤醒，无需应用层回调。审批请求包含进程ID、原语、路径、SHA‑256 等，预览内容使用 `repr` 转义以防语义欺骗。
  - **Shell 执行**：以 `argv` 数组而非命令字符串运行，策略支持白名单、黑名单、询问等模式。匹配基于 token 化的 `argv`，并检查嵌套解释器链。
  - **JIT 工具**：允许进程提出 TypeScript 工具候选，经校验后在 Deno 沙箱中执行，仅通过 `libos.syscall` 调用运行时原语，无法直接访问宿主 API 或工具注册表。Deno 运行时以 `--no-prompt` 且无文件、网络、环境等权限启动，静态导入受限，动态执行端口被禁用。
  - **镜像注册**：需要特定能力，YAML 加载受文件系统读权限和保护，防止特权升格。
  - **检查点与审计**：检查点快照可重建状态（不可回滚外部效应），审计记录所有权限变更与副作用。
- **架构分层**：模型面向的技能/工具层 → Agent libOS 运行时（进程、对象内存、能力、人工队列、系统调用代理等）→ 可注入的资源提供者基座 → 宿主运行时/提供者后端（模型 API、SQLite、Deno 等）。

## 6. 主要结论与发现
- Agent libOS 证明了可以构建这样一个运行时基座：**将工具可见性与资源权限解耦，通过进程抽象、显式能力、阻塞原语和审计，使长期 LLM 代理能被安全地调度、授权、恢复和审计**。关键发现包括：
  - 工具可见性不代表资源权限，缺失能力的工具调用会被原语层拒绝。
  - 分叉/派生不继承父进程的写权限，`exec` 不自动升级权限，命名空间隔离避免命名冲突。
  - 人工审批融入运行时原语，被拒绝后代理可获得结构化失败结果并继续执行，而非崩溃。
  - JIT 生成的 TypeScript 工具无法绕过能力检查，沙箱机制有效。
- 核心声明是：代理系统应将“模型可以请求的接口”与“运行时可以执行的权限”分离，这是比基准分数更基础的架构进步。

## 7. 优点
- **设计清晰且系统**：将成熟的 OS 概念（进程、能力、命名空间、设备阻塞）引入 LLM 代理领域，抽象合理。
- **安全边界严谨**：通过工具/libc 分层和原语检查，将可编程的代理执行与外部资源严格隔离，降低注入风险。
- **可执行的原型与测试**：123 个回归测试直接编码了安全属性，可复现、可扩展，而非仅停留在口号。
- **集成人工为第一类对象**：人工审批作为阻塞原语，由调度器统一管理恢复，避免了回调地狱和状态丢失。
- **JIT 工具隔离**：将生成式工具的执行放入 deny-by-default 的 Deno 沙箱，并通过 syscall 代理限制权限，实践了生成代码的最小权限原则。
- **模块化与可注入的提供者**：资源提供者基座允许替换文件系统、时钟等实现而不影响上层。

## 8. 不足与局限
- **缺乏端到端性能评估**：未在标准代理基准上对比任务成功率、延迟开销和吞吐量，论文的实用价值难以量化。
- **威胁模型不覆盖语义提示注入**：恶意文档仍可能诱骗模型请求危险操作，运行时仅保证这些请求会被权限拦截，但不防止请求本身（模型可能仍被欺骗）。
- **沙箱尚非生产级**：Deno 沙箱虽 deny-by-default，但未与容器、WASM 或微 VM 结合，强隔离部署仍需额外工具。
- **策略引擎简略**：目前仅覆盖基本的能力约束、文件/Shell 策略列表，缺少风险评分、配额、基于角色的人工授权等。
- **检查点不能回滚外部副作用**：仅可重建运行时状态，补偿操作需依赖审计和显式补偿，未实现事务性回滚。
- **审计仅为基础记录流**：未提供按进程、能力、原语等维度的索引查询。
- **上下文管理初步**：工具结果压缩、长文档分页、重复动作抑制等高级内存管理尚未成熟。
- **分布式与持久化限制**：当前原型为单机、非持久调度，并未涉及多工作器调度或生存期更长的代理。
