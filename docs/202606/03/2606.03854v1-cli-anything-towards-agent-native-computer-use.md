## 摘要
随着大型语言模型在推理和工具使用能力上的进步，研究人员越来越多地寻求利用它们构建能够与现有软件交互的计算机使用代理。主流方法开发的是通过视觉界面控制应用的 GUI 代理：解读屏幕截图、定位 UI 元素并执行鼠标点击以模仿人类交互。这种以 GUI 为中心的范式与代理的能力根本错位。当前的 GUI 代理在脆弱的像素级交互、时序依赖和因界面变化而失效的基于坐标的操作中挣扎。它们迫使代理模拟人类的感知局限，而不是利用其在结构化数据处理和程序化控制方面的计算优势。CLI-Anything 倡导代理原生的计算机使用设计。我们不是强迫代理浏览视觉布局，而是创建与代理自然操作方式一致的界面：通过结构化命令、显式状态表示和确定性反馈。我们将现有应用转化为命令行封装，保留功能的同时暴露针对 AI 原生交互优化的机器可读协议。这消除了困扰 GUI 代理的有损视觉到计算的转换。与其构建复杂的屏幕阅读器和点击模拟器，我们应该围绕代理的优势重新设计交互范式：精确的程序化控制和确定性执行。我们考察了这种代理原生计算机使用变革的方法论、架构、证据和未来方向。我们构建了 CLI-Hub 作为一个综合平台，将这一代理原生的计算机使用愿景付诸实践。该平台为计算机使用的根本性转变提供了方法论、架构和基础设施。

## Abstract
As large language models advance in reasoning and tool use capabilities, researchers increasingly seek to leverage them for computer use agents that can interact with existing software. The dominant approach develops GUI agents that control applications through visual interfaces: interpreting screenshots, locating UI elements, and executing mouse clicks to mimic human interaction. This GUI-centric paradigm fundamentally misaligns with agent capabilities. Current GUI agents struggle with brittle pixel-level interactions, timing dependencies, and coordinate-based actions that break with interface changes. They force agents to emulate human perceptual limitations rather than leverage their computational strengths in structured data processing and programmatic control. CLI-Anything argues for agent-native computer use design. Instead of forcing agents to navigate visual layouts, we create interfaces aligned with how agents naturally operate: through structured commands, explicit state representations, and deterministic feedback. We transform existing applications into command-line harnesses that preserve functionality while exposing machine-readable protocols optimized for AI-native interaction. This eliminates the lossy visual-to-computational translation that plagues GUI agents. Rather than building sophisticated screen readers and click simulators, we should redesign interaction paradigms around agent strengths: precise programmatic control and deterministic execution. We examine the methodology, architecture, evidence, and future directions for this agent-native transformation of computer use. We have built CLI-Hub as a comprehensive platform that operationalizes this agent-native computer use vision. The platform provides methodology, architecture, and infrastructure for this fundamental transformation of computer use.

# CLI-Anything 论文详细总结

## 1. 论文的核心问题与整体含义
- **当前困境**：主流 GUI 代理通过截图解析、像素定位和鼠标点击来操作软件，这种范式迫使代理模拟人类的视觉与操作局限，而非发挥 LLM 在结构化数据处理、程序化控制方面的优势。
- **根本错位**：GUI 交互存在像素脆弱性、时序依赖、坐标失效等问题，界面稍有变化便可能导致动作失败；同时，GUI 将软件状态压缩为像素流，代理必须进行有损的“视觉→计算”翻译。
- **整体含义**：论文提出 **代理原生计算机使用** 的理念，主张为代理重新设计软件接口，用结构化命令、显式状态和确定性反馈取代 GUI 导航，从而将“计算机使用”从视觉模仿变为可编程控制，使代理真正利用自身的计算与推理能力。

## 2. 论文提出的方法论
- **核心思想**：将现有应用转化为 **命令行封装（harness）**，对外暴露机器可读的操作协议，同时保留真实后端的完整功能。代理通过 CLI 指令操作软件，而不是模拟人类操作 GUI。
- **关键技术流程——Harness 生成 SOP（标准操作流程）**：
    1. **接口考古**：识别软件的后端引擎、原生数据模型、现有 CLI 工具、命令或 API 调用方式。
    2. **建模**：提取领域名词（project、layer、track、clip 等）、操作映射、约束条件，构建语义表面。
    3. **设计**：制定双模 CLI（REPL + 一次性子命令），命令群围绕领域模型组织，输出采用 JSON，错误可恢复。
    4. **实现**：建立持久项目状态（JSON 格式）、会话管理、撤销/重做栈、后端执行包装器（调用真实软件二进制并根据产出物验证）。
    5. **渲染/导出**：调用真实软件（如 `blender --background`、`libreoffice --headless`）完成渲染或导出，禁止玩具式替代品。
    6. **验证**：单元测试 + 端到端测试，不仅检查命令返回码，还验证最终用户面向的制品（如文件魔数、像素属性、ZIP 结构）。
    7. **发布**：生成技能文档（SKILL.md）、在 CLI-Hub 中注册元数据，提供安装与启动路径。
- **最小代理原生软件合约**：\(H = (S, C, I, R, V, D)\)
    - \(S\)：持久状态空间（项目文件、撤销历史等）
    - \(C\)：命令词汇（领域操作）
    - \(I\)：检查表面（状态、列表、历史、预览的 JSON 输出）
    - \(R\)：渲染/导出关系（委托真实后端）
    - \(V\)：验证层（单元测试、端到端检查、制品验证）
    - \(D\)：发现层（技能文件、注册元数据、安装策略）
- **状态即代理工作区**：CLI 封装不仅是一层薄命令包装，而是提供可检查、可回滚、具有回复循环的完整工作空间（包含 session、undo/redo、dry-run、自动保存）。
- **预览即显示协议**：预览协议定义了一套稳定的磁盘束结构（`manifest.json`、`summary.json`、`artifacts/`），由 harness 生产，CLI-Hub 消费，使代理和人类都能读取同一份渲染证据，既保留后端真实性又避免 GUI 截屏的有损性。
- **发现与分发——CLI-Hub**：
    - 本地注册表包含 65 个 CLI-Anything 条目（29 个类别），公共注册表包含 18 个第三方 CLI。
    - 安装器支持 pip、npm、uv、brew、捆绑命令行等多种路径。
    - 技能文档（61 个）为代理提供任务时的能力描述。

## 3. 实验设计

- **数据集/场景**：
    - **环境覆盖**：涵盖创意工具（Blender、GIMP、Inkscape、Krita、FreeCAD）、办公套件（LibreOffice、Zotero）、视频编辑（Shotcut、Kdenlive）、调试/分析工具（LLDB、RenderDoc、Nsight）、游戏（Slay the Spire II）等多类真实软件。
    - **案例演示**：展示了三个录制运行中的中段画面（Slay the Spire II 游戏控制、Blender 轨道无人机建模、FreeCAD 探测车装配），以证明命令流、状态展示与渲染输出的一致性。
- **Benchmark 形式**：
    - **目录覆盖统计**：总计 83 个 CLI 条目（65 个生成/策划 + 18 个第三方），覆盖 32 个类别。

    - **制品验证证据**：端到端测试中执行真实软件，检查 PNG 魔数字节、ZIP 结构、音频/视频时长等真实工件属性。
- **对比方法**：未直接与 GUI 代理框架（如 OSWorld、VisualWebArena、AndroidWorld）进行分值对比，而是将自身定位为“环境层”，与编码代理中的可执行环境（SWE-bench、SWE-agent）及工具使用训练（Toolformer、ToolLLM）形成概念对应。

## 6. 论文的主要结论与发现
- **接口错位是计算机使用代理的核心瓶颈**：强迫代理通过像素和点击模拟人类 UI 操作会带来诸多脆弱性，而有损的视觉-计算翻译是问题的根源。
- **代理原生接口可行且可推广**：通过对 Blender、Slay the Spire II、FreeCAD 等不同类型应用的 CLI 封装，证明将软件功能抽象为结构化命令、显式状态和后端验证是通用的实践路径。
- **制品验证回路至关重要**：必须通过真实软件后端生成、检查最终用户面对的产出物，而不是仅依赖进程返回码或脏截屏，从而为代理提供可靠反馈。
- **发现与分发基础设施是代理使用软件的前提**：CLI-Hub 的注册、安装、技能描述机制，加上代理实际调用占比高达 87.8%，表明这种基础设施已成为生产性依赖。
- **未来方向**：应构建基于可程序化制品的任务基准，将许多 GUI 计算机使用任务转变为可执行环境的任务，与编码代理的评估框架对齐。

## 7. 优点
- **范式创新**：从根本上改变了“代理如何使用软件”这一问题的定义，从“做出好人眼的 GUI 操作”转向“设计适合代理的接口”，立意高远且自洽。
- **真实软件、真实后端**：严格坚持使用真实应用的渲染器、文件格式和计算引擎，保证输出的可信性，避免玩具化模拟。
- **广泛的软件覆盖**：目录已覆盖从创意工具到办公、多媒体、调试、游戏等多个领域，展示了方法的可迁移性。
- **系统完整性**：从接口考古、CLI 生成、预览协议、技能文档到发现与分发平台（CLI-Hub），形成完整的端到端链路。
- **明确定义软件合约**：\(H = (S, C, I, R, V, D)\) 抽象给出了代理原生接口可复用的设计准则，具有理论指导价值。
- **对“硬软件”的深度处理**：Slay the Spire II 的案例分析展示了当后端无现成 API 时，如何在运行时内部注入桥接器并抽取出语义动作，技术细节扎实。

## 8. 不足与局限
- **缺少定量基准对比**：没有在标准 GUI 代理评测集（如 OSWorld、VisualWebArena）上与像素代理进行成功率和效率的对比，难以量化范式转换的增益。
- **适用范围受限**：要求软件具有可编程后端或可解析的状态格式；对于纯粹的前端界面或完全闭源的无结构应用，仍需依赖 GUI 自动化或无障碍接口。
- **预览能力尚不完善**：当前只有 5 个具备预览功能的集成，多数 harness 的视觉反馈仍不够丰富，尚难完全支撑需要密集视觉判断的任务。
- **代理调用率不代表任务成功率**：87.8% 的调用占比说明代理频繁使用 CLI-Hub，但并未报告实际任务完成的准确率或失败模式，存在“碎片化调用”的风险。
- **生态尚处早期**：大部分条目属于项目自行生成，第三方贡献和真实应用场景的深度集成仍不充分；长期的维护、版本适配和社区参与尚待观察。
- **实验缺乏消融**：未剖析例如仅用无状态命令、去除预览协议、或用简易渲染替代真实后端会怎样影响代理表现，难以确立各组件的独立贡献。
