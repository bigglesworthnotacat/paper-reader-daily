## 摘要
工具增强的大语言模型智能体越来越多地操作大型工具库，但现有评估往往关注模型能否正确调用工具，而非可见工具菜单如何影响可靠性、效率和与安全相关的风险暴露。我们引入了ToolMenuBench，这是一个用于评估多步大语言模型智能体中工具菜单构建的基准。ToolMenuBench变化工具菜单大小、干扰项类型、状态依赖的任务结构以及风险暴露，并报告过滤层面和下游智能体的指标，包括可见工具数量、风险工具暴露、任务成功率、错误工具调用、过早行动和词元用量。在使用七种模型后端、三种工具菜单大小、六种过滤方法和七种评估设置的对照评估中，CMTF将全工具暴露下的任务成功率从32.1%提升至85.7%，同时平均词元用量减少约98%。因果最小工具过滤在可见工具数、错误工具调用、过早行动和风险工具暴露方面，相对于未过滤暴露、词汇过滤、状态感知过滤和更广泛的因果路径基线，实现了最佳的整体权衡。ToolMenuBench提供了一个可重用的评估框架，用于研究智能体接口问题：哪些工具应可见、何时可见，以及在何种成本或风险约束下可见。

## Abstract
Tool-augmented large language model agents increasingly operate over large tool libraries, but existing evaluations often focus on whether a model can call a tool correctly rather than how the visible tool menu shapes reliability, efficiency, and safety-relevant risk exposure. We introduce ToolMenuBench, a benchmark for evaluating tool-menu construction in multi-step LLM agents. ToolMenuBench varies tool-menu size, distractor type, state-dependent task structure, and risk exposure, and reports both filter-level and downstream agent metrics, including visible-tool count, risky-tool exposure, task success, wrong-tool calls, premature actions, and token usage. In a controlled evaluation across seven model backends, three tool-menu sizes, six filtering methods, and seven evaluation settings, CMTF improves task success from 32.1% under all-tools exposure to 85.7%, while reducing average token usage by roughly 98%. Causal minimal tool filtering achieves the strongest overall tradeoff, reducing visible tools, wrong-tool calls, premature actions, and risky-tool exposure relative to unfiltered exposure, lexical filtering, state-aware filtering, and broader causal-path baselines. ToolMenuBench provides a reusable evaluation framework for studying the agent-interface problem: which tools should be visible, when they should be visible, and under what cost or risk constraints.

# ToolMenuBench 论文总结

## 1. 核心问题与研究背景
- **研究动机**：工具增强的大语言模型（LLM）代理面对日益庞大的工具库，但现有评测常关注“能否正确调用单个工具”，**忽略可见工具菜单（tool menu）** 对代理可靠性、效率与安全风险暴露的影响。
- **问题定义**：工具菜单的大小、干扰项类型、状态依赖的任务结构以及高风险工具的暴露程度，都会改变代理的决策空间，从而影响任务成功率、错误调用次数、过早行动和 token 消耗。因此，**工具菜单构造本身应成为代理接口设计的一级评估对象**。

## 2. 方法论
- **核心思想**：将工具菜单过滤策略抽象为 **“过滤方法→可见工具子集→下游代理行为”** 的可控评估流程，通过统一的符号状态机模拟执行，分离过滤质量与模型选择能力。
- **关键技术组件**：
  - **符号状态与合同**：每个工具均定义**前提条件（preconditions）和效果（effects）**，任务有初始状态与目标状态。执行引擎仅当工具前提满足时更新状态。
  - **过滤方法比较**：
    - **全工具暴露（All-tools）**：展示所有工具。
    - **关键词 Top-k**：基于任务描述与工具名称/描述的词汇重叠筛选 k 个工具。
    - **状态感知过滤**：仅暴露前提条件满足的工具。
    - **全因果路径暴露**：暴露从当前状态到目标状态的某条有效因果路径上的所有工具（不限于下一步）。
    - **因果最小工具过滤（CMTF）**：仅暴露**当前可执行且效果直接推进剩余目标状态**的“下一个必经工具”，严格限定可见工具边界。
  - **等式**：对于状态感知过滤，可见工具集合 = {a_i | R_i ⊆ s_t}，其中 R_i 为所需状态变量集，s_t 为当前状态。
- **评估指标**：
  - **过滤层指标**：金标准下一工具可见性、零可见工具率、额外工具暴露数、平均可见工具数、风险工具暴露频率、未授权风险工具暴露频率。
  - **下游代理指标**：任务成功率、错误工具调用次数、过早行动次数、完成步数、总 token 用量、风险工具暴露次数。
  - 支持合同质量的扩展指标（精确率、召回率、风险标签准确率）。

## 3. 实验设计
- **数据集与场景**：合成工具注册表，涵盖**日历、邮件、文件文档、联系人身份**等需要多步顺序操作的工作流领域。每个任务包含用户请求、初始符号状态、目标状态、金标准工具轨迹。
- **Benchmark 构成**：
  - **主混合干扰项基准**：包含语义干扰、近似重复工具、模式兼容错误工具、过早工具、高风险工具和跨域干扰的综合性设置。
  - **六个定向压力测试**：分别隔离上述单一干扰类型以测试过滤方法的鲁棒性。
- **对比方法**：6 种过滤策略 → 全工具暴露、关键词 Top-5、关键词 Top-10、状态感知、全因果路径、CMTF。所有方法均使用同一套任务、模型后端和执行环境，仅改变可见工具菜单。
- **评估设置**：7 个模型后端（Nova Lite/Premier/Pro, Claude Haiku/Sonnet/Opus, GPT-OSS-120B），3 个工具注册大小（25, 100, 250 工具），共 7 种评估设置（1 个混合设置 + 6 个定向设置）。

## 5. 实验数量与充分性
- **总执行次数**：共 **26,460 次**端到端代理执行（7 设置 × 7 模型 × 3 菜单大小 × 6 过滤方法 × 30 任务）。加上各消融与压力测试，实验覆盖面极广。
- **公平性**：采用严格对照设计，统一提示词格式、确定性的模拟执行环境、一致的评价脚本，确保过滤方法的效果差异可归因于菜单构造本身。
- **充分性**：通过变化工具库规模、干扰项类型和模型能力，系统性地揭示了菜单过滤对任务成功、效率和安全的影响，实验设计全面且可复现。

## 6. 主要结论与发现
- **工具菜单设计至关重要**：全工具暴露仅取得 32.1% 成功率，且平均 token 用量高达 56,062；**CMTF 将成功率提升至 85.7%，token 用量减少约 98%**（1,125 tokens）。
- **状态对齐优于语义相关**：关键词过滤虽减少菜单大小，但成功率远低于因果过滤；状态可执行性也非充分条件——可调用工具可能延迟或无关。
- **因果最小过滤的鲁棒性**：在所有类型干扰项（语义、近似重复、模式兼容、过早、风险、跨域）压力测试中，CMTF 均保持最佳成功率（84.4%~90.6%），误调用和 token 用量极低。
- **风险暴露大幅降低**：CMTF 将高风险工具暴露次数从全工具暴露的 206.87 次降至 0.71 次，印证“最小暴露”原则可缩小模型选择高风险动作的机会空间。
- **模型能力与接口的交互**：即使 CMTF 大幅提升整体表现，不同模型的成功率仍有差异，说明菜单优化不消除模型自身的工具调用错误，两者需联合优化。

## 7. 优点
- **新问题定义**：首次将“工具菜单构造”作为代理系统的独立评估维度，填补了现有基准忽略工具可见性影响的空白。
- **分层评估体系**：清晰区分过滤层与下游指标，便于故障归因。
- **丰富的干扰分类**：定义六类现实干扰项，使评测能区分不同来源的工具混淆。
- **高可扩展性**：开放支持嵌入检索、模式感知、学习型合同、风险/成本感知过滤等后续扩展。
- **实验严谨性**：大规模对照实验，统一执行环境，报告多维度指标，结果透明可复现。

## 8. 不足与局限
- **合成环境的逼真度**：使用模拟工具环境和符号状态，未涵盖真实 API 延迟、认证错误、模式漂移、外部副作用、权限作用域等问题。
- **状态表示的完整性**：依赖完美符号状态和工具前提/效果合同，现实场景中常面临部分可观测、合同不完整或变量非规范化等难题。
- **任务领域局限**：仅覆盖日历、邮件、文档等工作流助手场景，可能无法直接泛化至开放网页导航、代码执行、数据库管理或长周期企业工作流。
- **与模型的耦合性**：实验结果与所选模型后端的指令遵从、工具调用格式和提示词敏感性相关，不同模型间的表现差异可能干扰过滤效果的绝对解读。
- **资源报告缺失**：未提供推理算力成本或调用 API 的具体费用信息，限制了成本效率的完整评估。
- **安全性不替代**：降低风险工具暴露虽有益，但不能取代生产环境中的授权检查、确认流程、审计和策略执行。
