## 摘要
用于漏洞发现与验证的大语言模型智能体不仅仅是一个模型。它结合了三个组件：一个用于代码分析的大语言模型，一个如 Codex 或 OpenCode 的智能体脚手架负责导航、工具使用与执行，以及一个审计剧本，即指导大语言模型和脚手架进行漏洞发现的领域特定流程知识。先前的工作依赖人工提供的剧本，包括提示工程、手动工作流、知识库和启发式方法。这引出了两个研究问题：获取——人工策划是否必要，剧本创建能否自动化？迁移——演化后的剧本能否将审计程序传递给更弱的智能体，提升其能力？我们提出 EvoHunt，一个在开源仓库上进行安全审计的剧本演化环境。三个智能体驱动演化循环：审计智能体执行当前剧本并生成发现；评估器根据真实情况对结果进行评分；修订器基于失败分析向剧本提交更新。剧本格式不受限制：从空开始，EvoHunt 会添加或移除工作流、启发式、漏洞知识或领域特定内容。演化后的剧本只需微小调整即可在不同的 LLM 或脚手架下运行。我们在开源安全公告上评估了 EvoHunt。在获取方面，剧本演化使 Codex/GPT5.4-xhigh 的端到端利用成功率从 1.1% 提升至 6.2%，提高了 6 倍；演化后的 OpenCode/GLM5.1 剧本在所有指标上均超越了 OpenAI Codex Security，目标匹配率为 11.3% 对 9.2%，表明开源演化可以超越专用商业产品。在迁移方面，GLM 演化出的剧本给学生模型带来了最大提升：Qwen3.6-27B 从 2.4% 提升至 6.5%，Qwen3.6-35B-A3B 从 1.1% 提升至 4.6%，且 A3B 获得的匹配数量是 GPT 迁移的 2.4 倍。

## Abstract
An LLM agent for vulnerability discovery and validation is more than a model. It combines three components: an LLM for code analysis, an agent harness such as Codex or OpenCode for navigation, tool use, and execution, and an audit playbook, domain-specific procedural knowledge that guides the LLM and harness toward vulnerability discovery. Prior work relies on human-supplied playbooks, including prompt engineering, manual workflows, knowledge bases, and heuristics. This raises two research questions: Acquisition - is human curation necessary, and can playbook creation be automated? Transfer - can an evolved playbook transfer the audit procedure to weaker agents, improving their capability?   We present EvoHunt, a playbook evolution environment over open-source repositories for security auditing. Three agents drive the evolution loop: an audit agent rolls out the current playbook and produces findings; an evaluator scores outcomes against ground truth; and a reviser commits updates to the playbook based on failure analysis. The playbook format is unconstrained: starting empty, EvoHunt adds or removes workflows, heuristics, vulnerability knowledge, or domain-specific content. The evolved playbook requires only minor adaptation to run under a different LLM or harness.   We evaluate EvoHunt on open-source security advisories. For acquisition, playbook evolution raises end-to-end exploits for Codex/GPT5.4-xhigh 6x, from 1.1% to 6.2%, and the evolved OpenCode/GLM5.1 playbook surpasses OpenAI Codex Security on every metric, with 11.3% vs. 9.2% target-match rate, showing open-source evolution can outperform a dedicated commercial product. For transfer, the GLM-evolved playbook gives the strongest student lift: Qwen3.6-27B improves from 2.4% to 6.5%, Qwen3.6-35B-A3B from 1.1% to 4.6%, and A3B obtains 2.4x more matches than GPT transfer.

## 1. 研究背景与核心问题

安全审计正从人工分析向大语言模型(LLM)智能体自动化过渡。现代审计智能体通常由三个组件构成：**(i) 底层LLM负责代码分析；(ii) 通用智能体脚手架(harness)提供仓库导航与工具调用；(iii) 审计剧本(playbook)**，即外部提供的领域特定流程知识，用以指导审计行为。

### 1.1 现状与痛点
- **人工依赖性**：现有工作依赖人工精心设计的剧本，包括提示工程、角色扮演、手动审计工作流、精心策划的漏洞知识库(RAG)以及各类启发式规则。
- **能力耦合**：审计性能通常被归因于模型本身的能力（更强的模型、更大的上下文窗口），而**流程知识的贡献未经系统量化**。
- **迁移障碍**：为强模型手工编写的剧本难以迁移至弱模型，导致过程知识无法跨模型复用。

### 1.2 核心研究问题
论文围绕两个核心研究问题展开：

- **RQ1 获取(Acquisition)**：剧本创建能否完全自动化？能否从空剧本开始，通过尝试、评估和修订循环，习得可复用的审计流程？
- **RQ2 迁移(Transfer)**：一个由强教师智能体演化出的剧本，迁移给弱学生智能体后，能否提升后者的审计能力？

## 2. 方法论：EvoHunt 剧本演化框架

EvoHunt 将审计流程外部化为一个版本化的文本剧本仓库，在模型与脚手架**固定的条件下**，仅通过演化剧本本身来提升审计表现。

### 2.1 核心设计理念
- **持续流程学习**：模拟持续学习范式，以时间顺序的审计案例批次进行学习，而非对同一数据多次迭代至收敛。
- **转移流程学习**：将强教师模型演化出的剧本作为固定基座，为学生模型引入一个轻量级**执行适配器(adapter)**，仅适配目标执行环境的差异。
- **空搜索空间**：剧本格式不受任何预定义结构的约束——以空 `README` 文件为起点，智能体自主决定要学习什么、如何组织流程、积累哪些安全知识。

### 2.2 三代理演化循环

每次迭代由三个固定提示的代理顺序执行：

| 代理 | 功能 |
|------|------|
| **审计代理(Discovery)** | 在当前剧本指导下对仓库进行审计，生成发现、追踪轨迹和证据。 |
| **评估器(Evaluator)** | 基于真实标注对审计结果评分，生成结构化失败报告（学习信号）。 |
| **修订器(Reviser)** | 基于失败分析抽象出通用审计指导，向剧本仓库提交修改版本。 |

### 2.3 流程学习形式化

**训练设置**：训练数据按时序分批处理 ( \(N_1, ..., N_B\) )，每批可包含从前几批采样的重放案例( \(R_b\) )：

\[
C_b = N_b \cup R_b
\]

**运行与评估**：固定智能体 \(G\) 用剧本 \(P\) 在批次 \(C_b\) 上运行生成记录：

\[
\text{Eval}(P, C_b, X_{P,C_b}, L_{C_b}) \rightarrow (S(P,C_b), F(P,C_b))
\]

其中 \(S\) 为标量得分（发现正确性 0.7 + 复现有效性 0.3），\(F\) 为结构化失败报告。

**修订**：
\[
\text{Revise}(P, F) \rightarrow P'
\]

修订需泛化失败模式为可复用指导，**不得记忆具体仓库名、公告细节或利用字符串**。

**选择与冻结**：通过 Git 分支锦标赛比较候选剧本与当前最佳剧本，选择得分高者作为下一轮的基；训练窗口结束后冻结最优剧本 \(P^*\)，随后在留出测试集上评估。

### 2.4 迁移学习形式化

强教师环境 \(G_s\) 生成冻结源剧本 \(P^*_s\)。对目标环境 \(G_t\)，引入空适配器 \(A_t\)，组合流程为：

\[
P_{s\to t} = P^*_s \oplus A_t
\]

适配器仅在目标环境中演化，且受长度预算 \(B\) 约束，修改仅限目标执行指导，不得修改源剧本内容。

**迁移效应分解**：

\[
\Delta_{\text{direct}}^{t\leftarrow s} = S(G_t, P^*_s, C_{\text{test}}) - S(G_t, P_\emptyset, C_{\text{test}})
\]
\[
\Delta_{\text{adapted}}^{t\leftarrow s} = S(G_t, P_{s\to t}, C_{\text{test}}) - S(G_t, P_\emptyset, C_{\text{test}})
\]

残余差距 \(S(G_s, P^*_s, C_{\text{test}}) - S(G_t, P_{s\to t}, C_{\text{test}})\) 归因于不可弥补的模型 / 脚手架能力差距。

### 2.5 实现关键技术细节

- **剧本仓库**：以 Git 管理文本文件，工作流规则、漏洞类别指南、验证关卡等纳入版本控制，历史所有提交均可审查。
- **重放记忆**：基于 BM25 对历史案例排名，选取固定比例保留漏洞类别覆盖面。
- **分支锦标赛**：每轮运行两个候选剧本在同一批次上计分，胜者作为下一轮基。
- **修订门限**：检查提交是否源自正确的基、是否修改了追踪文件、报告是否符合模式等，最多允许 3 次修复尝试。
- **超参数**：每次运行最多 20 轮、2 小时超时；每批最多 10 新案例 + 0.25 比率重放；并发案例 ≤ 2。

| Qwen3.6-35B-A3B | GLM 教师 | 1.1% → **4.6%** | **4.0×** |

- **教师风格传播**：GPT 教师传递了高质量浓缩策略（T1 主导，总产出下降），GLM 教师传递了广泛扫描策略（总产出和资格率大幅上升）。
- **教师选择远比绝对值重要**：对 A3B 学生，GLM 教师给出的匹配数是 GPT 教师的 **2.4 倍**，资格率提升 24.3 pp vs 3.4 pp。
- **信号质量更易迁移**：资格率提升经常超过教师自身的进化收益，证明明确的比较、剪枝和证据阈值规则高度可移植。

### 6.3 现实世界影响
演化过程中产生了 **28 个已确认的漏洞披露**，覆盖 18 个开源项目，包括 GitHub 安全公告、CVE 编号和一笔 1,500 美元的漏洞赏金。

## 7. 优点

1.  **理论洞察**：将安全审计性能从“模型能力竞赛”中抽离，首次**系统地形式化和量化了外部流程知识 (playbook) 的独立价值**，呼应 Sutton 的“苦涩教训”——持久的进步来自可学习和可搜索的系统，而非静态专家规则。
2.  **方法论创新**：提出**持续流程学习**和**转移流程学习**范式，完全在文本空间进行“梯度”更新（通过文本修订实现），无需调整模型权重，开创了“通过文本蒸馏”的新路径。
3.  **设计严谨性**：严格的**时序分离基准**（训练/测试按时间切割）、**空 playbook 基线**（控制模型与脚手架变量）、**单 LLM 裁判 + 手工审核**的评分机制，确保了内部有效性。
4.  **实践可行性**：演化成本极低（约 $1400 订阅费用）、可断点续传（基于 Git）、产生的剧本可立即部署并能跨模型迁移，为资源受限的组织提供了一种“一次性投入、多模型复用”的高性价比方案。
5.  **开源与透明**：公开所有代码、基准、评分记录，并披露 28 个真实漏洞，兼顾科学可复现与实际安全影响。

## 8. 不足与局限

1.  **模型与脚手架耦合**：实验未能在完全相同的脚手架下比较 GPT 和 GLM——GPT 使用 Codex，而 GLM 和 Qwen 使用 OpenCode，观察到的差距可能部分由脚手架上下文窗口差异（272k vs 192k tokens）所致，无法完全归因于模型或剧本。
2.  **单一训练轨迹**：每个环境仅运行一次演化（因成本限制），结果的随机性（playbook 质量对演化轮次中随机性的敏感度）未得到评估，报告的是存在性证明，而非典型性特征。
3.  **分布偏移风险**：基准局限于 GitHub Advisory Database 内的高危害/严重漏洞、可本地复现的案例，训练/测试集生态系统分布不均（如 Maven 从训练 19.2% 下降到测试 5.5%）；Playbook 可能过拟合于特定的漏洞类别分布或公告报告风格。
4.  **评估受限**：评分依赖 LLM 裁判（GPT5.4-xhigh），尽管有手工审核机制，边缘案例仍可能存在误判。此外，对离靶真实漏洞（发现了公告之外的其他有效漏洞）一律计为“未命中”，可能低估审计的实际价值。
5.  **超参数选择**：重放比率、批次大小、评分权重等关键超参数依赖人工判断，未进行系统搜索，对基准分布存在潜在过拟合风险。
6.  **泛化边界**：目标范围明确了源可获得、可在本地 Docker 验证，排除了需要云服务、物理访问或复杂生产环境的漏洞，直接应用至更广泛的实网渗透测试仍需谨慎。
