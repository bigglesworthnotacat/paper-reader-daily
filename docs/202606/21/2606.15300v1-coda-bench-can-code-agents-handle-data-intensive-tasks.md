## 摘要
高级代理正日益展现出作为自主工程师的潜力，这催生了对能够捕捉现实开发复杂性的评估基准的需求。此类环境通常既涉及复杂代码，又涉及大规模数据（即文件系统）。然而，现有基准通常孤立地评估以代码为中心或以数据为中心的能力，与现实开发场景之间存在明显差距。在本文中，我们通过引入 CODA-BENCH 来弥合这一差距，这是首个在数据密集型环境中联合评估代码与数据智能的基准。我们基于 Kaggle 生态系统（包含数百个数据集）构建了一个数据密集型 Linux 沙盒，代理必须主动探索复杂的文件层次结构以识别相关资源，并为数据驱动的分析任务生成代码。CODA-BENCH 包含 1,009 个任务，涵盖 31 个社区，每个任务环境平均包含 980 个文件，模拟了现实的数据规模与噪声。对先进代理的评估表明，即使是表现最佳的系统也难以有效整合数据发现与代码执行，成功率仅为 61.1%。这些结果凸显了当前代理在数据密集型任务上的能力存在显著不足，并指向了未来研究的有前景方向。

## Abstract
Advanced agents are increasingly demonstrating the potential to operate as autonomous engineers, creating a growing demand for evaluation benchmarks that capture the complexity of real-world development. Such environments typically involve both complex code and large-scale data (i.e., file system). However, existing benchmarks usually evaluate code-centric or data-centric capabilities in isolation, leaving a clear gap with real development scenarios. In this paper, we bridge this gap by introducing CODA-BENCH, the first benchmark to jointly evaluate code and data intelligence in a data-intensive environment. We construct a data-intensive Linux sandbox based on the Kaggle ecosystem (containing hundreds of datasets), where agents must actively explore complex file hierarchies to identify relevant resources and generate code for data-driven analytical tasks. CODA-BENCH comprises 1,009 tasks spanning 31 communities, with each task environment containing an average of 980 files, simulating realistic data scale and noise. Evaluations of advanced agents reveal that even top-performing systems struggle to effectively integrate data discovery with code execution, achieving a success rate of only 61.1%. These results highlight a substantial gap in current agentic capabilities for data-intensive tasks and point to promising directions for future research.

### 方法论（核心思想、技术细节与算法流程）
CODA-BENCH 的构建包含三个关键步骤，旨在创造一个逼真、可验证且具有挑战性的数据密集型任务环境：

*   **数据密集型环境构建**
    *   **核心思想**：利用 Kaggle 生态中数据集在真实分析项目里的共现关系，构建具有语义相关性的“干扰文件”，而非简单填充随机噪声。

        2.  **社区划分**：使用 Leiden 算法（分辨率参数 `γ=1.0`）将图分割为 323 个语义社区。每个任务的目标数据 `D*` 所属社区内的所有其他数据集，将作为与任务主题高度相似但又无关的“分布内噪声”。
        3.  **沙盒成型**：最终每个沙盒环境平均包含约 980 个文件，涵盖 CSV、JSON、Parquet、图片等众多种格式，总大小从 20.3MB 至 45.4GB 不等。

*   **基于解答的任务反构**
    *   **核心思想**：不直接拟题，而是从 Kaggle 笔记本中已存在的精确数值结论（“解锚点”，如统计量、排名）反推出导致该结论的原始分析问题。
    *   **技术流程**：
        1.  **锚点识别**：结合 LLM 筛选与静态数据流分析，在笔记本输出中定位可验证且非平凡的数值结果，并通过动态重执行确认结果可重现。
        2.  **问题生成**：利用 LLM 依据提取出的解答路径和最终结果，反构出清晰、不泄露解决路径的自然语言问题。

*   **对抗式任务演化与验证**
    *   **核心思想**：借鉴生成对抗网络（GAN）的思想，将任务难度提升与质量验证建模为生成器 `G` 与判别器 `F` 的博弈过程，即 `min_G max_F L(G, F)`。
    *   **算法流程**：
        1.  **迭代演化**：以当前任务为起点，生成器 `G` 通过移除显式线索等方式增加任务难度；判别器 `F` 则由多个顶级 LLM（如 GPT-5.2, Gemini-3.0 等）组成的集成模型担任，计算任务完成率（`r(t)`）。
        2.  **进化策略**：当完成率过高，生成器分析成功轨迹来提升难度；当完成率降至零，生成器诊断是否因任务缺损（如歧义）导致失败，并进行修复。
        3.  **终审**：达到理想难度的任务最终交由人类专家进行可解性与无歧义校验，通过后正式纳入基准。此流程将初始的 1,395 道问题最终精炼为 1,009 个高质量任务（通过率 72.3%）。

### 论文的主要结论与发现
*   **现有能力存在显著瓶颈**：即使最先进的系统（Mini-SWE-Agent + GPT-5.5）在 CODA-BENCH 上执行准确率也仅为 61.1%，在更难的 CODA-HARD 上骤降至 49.6%。
*   **数据发现是核心挑战**：当直接向代理提供目标文件（神谕模式）时，所有代理的性能均有极大跃升，证实“在噪声中定位关键数据”是当前能力的首要短板。
*   **环境复杂度会显著削弱性能**：代理的成功率与环境的信噪比、数据总量呈强负相关关系，这表明真实世界中的高噪声、大容量数据场景对代理构成了巨大障碍。
*   **模型能力决定性能上限，框架设计影响效率**：不同后端模型在同一框架下表现存在性能天花板；而不同原生 CLI 工具实现接近准确率所需的交互回合数和 Token 成本可相差近 5 倍。
*   **错误模式存在能力分层**：更强的模型（如 GPT-5.5）更多失败在代码生成本身，而较弱模型（如 Kimi-K2.6）则更容易在最初的数据发现阶段就出错，且难以通过后续修复挽回。

### 不足与局限
