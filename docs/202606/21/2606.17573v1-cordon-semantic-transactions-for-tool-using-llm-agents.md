## 摘要
使用工具的LLM智能体正在将计算单元从人类显式发出的命令转变为具有状态化后果的模型驱动任务。然而，当今的智能体运行时仍将工具暴露为孤立的远程过程调用（RPC）。这种接口为运行时提供了方便的集成点，但缺乏跨多步智能体工作流的提交、回滚、恢复和审计的任务范围执行边界。我们认为这种不匹配需要一个运行时包含边界，而非另一种基于单次调用的防护措施。本文介绍了Cordon，一个用于在提交前暂存和验证不可逆智能体效果的事务性运行时系统。语义事务是一个任务级执行边界，它将工具意图和运行时跟踪的结果谱系与可逆本地状态、暂存的外部效果、委托权限和审计元数据绑定。Cordon通过一个事务管理器实现此抽象，该管理器跟踪派生结果对象，在影子状态中执行可逆变更，在效果发件箱中暂存外向动作，并记录恢复元数据。然后，运行时在提交状态或释放外部效果之前验证组合的执行流。我们在对抗性和良性工作流上的评估表明，Cordon暴露了现有防御措施所遗漏的跨步违规。它还减少了不可逆效果失败，同时以适度的批准和延迟开销保持良性任务完成。

## Abstract
Tool-using LLM agents are shifting the unit of computation from explicit human-issued commands to model-driven tasks with stateful consequences. Yet today's agent runtimes still expose tools as isolated RPCs. This interface gives runtimes a convenient integration point, but it lacks a task-scoped execution boundary for commit, rollback, recovery, and audit across multi-step agent workflows. We argue that this mismatch calls for a runtime containment boundary rather than another per-call guardrail. This paper introduces Cordon, a transactional runtime system for staging and validating irreversible agent effects before commit. A semantic transaction is a task-level execution boundary that binds tool intents and runtime-tracked result lineage to reversible local state, staged external effects, delegated authority, and audit metadata. Cordon implements this abstraction with a transaction manager that tracks derived result objects, executes reversible mutations in shadow state, stages outward-facing actions in an effect outbox, and records recovery metadata. The runtime then validates the composed execution flow before it commits state or releases external effects. Our evaluation across adversarial and benign workflows shows that Cordon exposes cross-step violations missed by existing defenses. It also reduces irreversible-effect failures while preserving benign task completion with modest approval and latency overhead.

### 论文核心问题与整体含义

*   **研究动机与背景**
    *   当前，LLM智能体的计算范式正从人类显式指令转向模型驱动的多步骤任务，这些任务会产生状态化的后果（如文件修改、发送消息等）。
    *   然而，现有的智能体运行时系统（如各种SDK）仍将工具视为孤立的远程过程调用（RPC）。这种设计提供了方便的集成点，但缺乏一个**任务范围（task-scoped）的执行边界**，无法对跨多个步骤的智能体工作流进行鲁棒的提交、回滚、恢复与审计。
    *   此不匹配意味着，仅仅对单个工具调用进行安全检查（如权限审批、内容过滤）是不够的，因为一个安全风险（如秘密泄露、恶意修改）可能由一系列看似无害的调用组合而成，其危险性只能在**组合的执行流**中显现。因此，论文认为需要一个**运行时容纳边界（runtime containment boundary）**，而非另一种单步防护栏。

### 论文提出的方法论

*   **核心思想：语义事务**
    *   提出**语义事务**，它是一个任务级的执行和提交边界，用于在提交前**暂存并验证**智能体行为的不可逆后果。它将一次任务中产生的工具意图、结果依赖关系（谱系）、本地状态变更、暂存的外部效果、委托权限和审计元数据都绑定到一个逻辑单元中。
*   **关键技术细节与流程**
    *   **事务模型**：语义事务`T`形式化定义为一个包含作用域、意图、读取锚点、结果对象、谱系、本地写入/删除、暂存外部效果、权限和生命周期状态的聚合体。它区分了三类对象：**结果对象**、**可逆的本地突变**和必须被延迟提交的**外部效果**。
    *   **运行时架构**：
        1.  **工具调度拦截**：在智能体运行时与具体执行工具之间插入一个调节层，将工具调用转化为对事务管理器的操作。
        2.  **在线事务构建**：随着任务执行，实时记录工具返回的**结果对象句柄**，并动态构建这些结果之间的**谱系图**，追踪“哪个后续操作衍生自哪个先前结果”。
        3.  **影子状态引擎**：所有本地文件写入、删除等变更，都先在一个事务作用域内的**影子文件系统**中执行，对真实环境不可见。这为本地状态提供了符合ACID特性的回滚能力。
        4.  **效果发件箱**：所有外部动作（如发送消息、API调用）不立即执行，而是放入一个**效果发件箱**中暂存，记录目标、载荷、谱系、权限和幂等键。
        5.  **验证与提交协议**：在任务完成时，**验证引擎**基于积累的完整事务上下文（谱系、暂存状态、待发效果、权限、策略`Ct`），检查是否满足一套**事务性容纳不变式**（例如，秘密衍生的结果流不能通往未授权的外部出口）。验证通过则原子性地**提升（commit）**影子状态并释放外部效果；失败则**丢弃/回滚**影子状态并取消待发效果，同时记录审计日志。

*   **性能表现**：事务中介带来的开销主要是具体的验证和审批工作。它在不安全的执行路径上通过提早终止任务，反而节省了时间和Token消耗。同时，它带来的延迟和审批次数增加保持在适度水平。

### 不足与局限

*   **工具覆盖度有限**：Cordon的事务和回滚保证仅适用于其运行时能够**完整中介和观察**的操作。对于不受支持的插件、黑盒工具或无法感知其副作用的外部服务，系统只能记录审计信息，无法提供完整保证，存在边界外逃逸的风险。
*   **验证规则依赖人工定义**：论文中提到的验证不变式和策略（如敏感路径、风险阈值）需要用户预先定义，其有效性高度依赖于规则制定的完备性和准确性。
*   **性能开销权衡**：虽然总体开销可控，但对于高频、简短的任务，事务管理、影子状态构建和验证步骤仍然会引入额外延迟。
*   **单运行时局限**：当前原型作为一个集中的运行时系统，未讨论在分布式、多智能体协作场景下如何进行事务协调与错误恢复，其工程实现（如gRPC、ZODB存储）仍停留在研究原型阶段，向高吞吐生产环境迁移可能存在挑战。
