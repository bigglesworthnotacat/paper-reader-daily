## 摘要
工具增强的语言模型代理通常通过是否选择正确工具、生成有效API参数以及完成请求任务来评估。然而，代理可能选择了正确的工具，但仍然对错误的外部实体进行操作。例如，请求“给Alex发邮件关于发布事宜”可能导致代理联系错误的Alex、附加错误的发布文档、在错误的线程中回复或更新错误的客户账户。我们将这些错误称为实体绑定失败。本文研究实体绑定失败作为工具增强型代理中的一个独特可靠性和安全性问题。我们形式化了工具正确性与实体正确性之间的分离，介绍了企业工作流中错误实体失败的分类法，并评估了实体感知的执行机制，包括实体解析前提条件、置信度门控绑定、歧义澄清和溯源追踪。在包含60个任务、五种模型后端和六种工具使用方法的受控诊断评估中，所有方法均实现了0.0%的工具选择错误，但面向动作的基线仍在24.0%-26.0%的运行中产生错误实体动作。实体感知方法在此设置下消除了错误实体动作和风险加权错误实体暴露，但通过延迟歧义降低了直接任务完成率。这些发现表明，安全使用工具不仅需要选择正确的工具，还需要在行动前将自然语言引用可靠地绑定到正确的现实世界实体。

## Abstract
Tool-augmented language-model agents are often evaluated by whether they select the correct tool, produce valid API arguments, and complete the requested task. However, an agent may choose the right tool and still act on the wrong external entity. For example, a request to "email Alex about the launch" may lead the agent to contact the wrong Alex, attach the wrong launch document, reply in the wrong thread, or update the wrong customer account. We call these errors entity binding failures. This paper studies entity binding failures as a distinct reliability and safety problem in tool-augmented agents. We formalize the separation between tool correctness and entity correctness, introduce a taxonomy of wrong-entity failures in enterprise workflows, and evaluate entity-aware execution mechanisms including entity-resolution preconditions, confidence-gated binding, clarification under ambiguity, and provenance tracking. In a controlled diagnostic evaluation across 60 tasks, five model backends, and six tool-use methods, all methods achieved 0.0 percent wrong-tool error, yet action-oriented baselines still produced wrong-entity actions in 24.0-26.0 percent of runs. Entity-aware methods eliminated wrong-entity actions and risk-weighted wrong-entity exposure in this setting, but reduced direct task completion by deferring under ambiguity. These findings show that safe tool use requires not only selecting the correct tool, but also reliably binding natural-language references to the correct real-world entity before action.

- **核心问题**：工具增强型语言模型代理在正确选择工具后，仍可能将自然语言引用错误地绑定到现实世界的外部实体上，例如“给 Alex 发邮件”却联系了错误的 Alex、“删除旧版发布文档”却删除了错误的版本。这类错误被论文定义为**实体绑定失败**，是一种被现有评估体系忽略的独立安全风险。
- **研究动机**：现有评测（如 API-Bank、ToolLLM、τ-bench）主要关注工具选择、API 有效性、计划和任务完成率，但无法区分“选对工具但用错目标”的情况。在企业工作流中，发错收件人、改错客户记录、取消错误会议等行为可能造成严重隐私、运营和声誉损害。
- **整体含义**：安全可靠的工具使用不仅需要正确的工具类型，还必须确保操作的目标实体正确。实体绑定应作为工具增强型代理中**一级可靠性要求**。

## 2. 论文提出的方法论：核心思想、关键技术细节、公式或算法流程

### 核心思想
在工具选择与工具执行之间插入 **实体感知门控（Entity-Aware Action Gate）**，只有当目标实体被充分解析后才允许执行，否则要求澄清或安全延迟。

### 关键技术细节
- **实体解析前提条件**：每个工具关联一组实体前提（如发送邮件需要收件人实体、日历取消需要事件实体），分为必需和可选，必需未满足则阻止执行。
- **候选实体检索**：对指令中的每个实体提及，从环境中检索候选集（如通过名称、语义、时间、所有者等），保持召回率高。
- **绑定分辨率**：对每个提及-候选对计算置信度分数 \( s(m, e, S) \)，选择最高分实体 \( \hat{e} \)，并满足：
  \[
  s(m, \hat{e}, S) \ge \tau \quad \text{且} \quad s(m, \hat{e}, S) - s(m, e_2, S) \ge \delta
  \]
  其中 \( e_2 \) 是第二高分的候选，\( \tau \) 为置信度阈值，\( \delta \) 为最小边际。
- **歧义时澄清**：当绑定未解决或存在真歧义时，模型返回具体澄清问题（如“您指的是 Alex Chen 还是 Alex Kumar？”）。
- **溯源追踪**：记录每个绑定的证据（如实体ID、名称、时间戳、最近交互等），支持审计和更安全的澄清。

### 算法流程（Algorithm 1）
1. 选择候选工具 \( t \) ；
2. 检索工具 \( t \) 的实体前提条件；
3. 提取指令中的必需实体提及集合；
4. 对每个提及：
   - 检索候选实体集合；
   - 为每个候选计算置信度分数；
   - 选择最高分候选；
   - 记录溯源证据；
   - 若未满足阈值+边际条件，则提出澄清并返回 **Deferred**；
5. 检查所有前提条件是否满足；
6. 若满足则执行工具调用并返回 **Executed**，否则请求补充信息并返回 **Deferred**。

## 3. 实验设计：数据集/场景、对比方法

### 数据集/场景
- **自建诊断测试集**：包含 5 个企业级领域：
  - 邮件（收件人、线程、附件）
  - 日历（事件、循环、参加者）
  - 文档（版本、文件夹、所有者）
  - 客户记录（账户、子账户、机会）

- 每个任务包含用户指令、环境状态（含多个候选实体）、所需工具、目标绑定标注、歧义类型和风险等级。
- 共 8 种歧义条件：无歧义、名称碰撞、文档版本歧义、时间歧义、账户碰撞、近重复记录、跨系统歧义、真歧义。

### 对比方法（6 种）
1. **Direct**：直接生成工具调用，无额外过滤；
2. **Semantic filter**：按语义相关性过滤工具后执行；
3. **CMTF only**：因果最小工具过滤，无实体门控；
4. **Entity retrieval**：在工具调用时提供候选实体列表供选择；
5. **Confidence gate**：要求绑定的实体满足置信度阈值和边际，否则延迟；
6. **Entity CMTF + provenance**：结合因果工具过滤、实体前提检查、溯源记录，未解决则延迟。

## 6. 论文的主要结论与发现

1. **工具正确≠安全执行**：所有方法在诊断设置下工具错误率为 0.0%，但动作导向基线仍有 24.0%–26.0% 的运行出现实体绑定失败，说明现有指标漏检关键风险。
2. **检索和过滤不能解决实体绑定**：Entity retrieval、CMTF only、Semantic filter 等方法实体错误率仅略有降低（24.0%–26.0%），表明候选暴露和工具过滤不足以确保正确绑定。
3. **实体感知门控消除实体错误**：Confidence gate 和 Entity CMTF+Prov 的实体错误率降至 0.0%，但直接任务完成率下降至 31.7% 和 26.0%，安全成功率为 40.0% 和 34.3%，体现了安全-完成的权衡。
4. **过度澄清率为 0.0%**：实体感知方法不在无歧义任务上多余提问，仅在实体未充分解析时延迟。
5. **实体错误集中于时间和真歧义任务**：时间歧义和真歧义条件下动作基线的实体错误率达 90%–100%，而实体感知方法 100% 检测到歧义并安全延迟。
6. **实体绑定是跨系统接地问题**：名称碰撞、跨系统引用等模式在企业环境中普遍存在，需要显式的接地策略。
