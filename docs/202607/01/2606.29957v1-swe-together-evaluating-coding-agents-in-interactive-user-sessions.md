## 摘要
大部分编码智能体基准测试是静态的：智能体会提前收到完整的任务描述，并仅根据其最终代码进行评判。然而，真实的编码辅助是交互式的，用户会在多轮对话中澄清目标、添加约束并纠正错误。我们引入了SWE-Together，这是一个从真实用户-智能体编码会话中重建的多轮基准测试。为了使得真实交互可验证，我们从11,260个录制的会话中精心挑选了109个仓库级任务，选择那些仓库状态可恢复、用户目标清晰、结果可观察的会话。为了在不同智能体上重放这些交互，我们构建了一个基于LLM的反应式用户模拟器，它保留了原始用户的意图，并在编码智能体的进度需要时提供反馈。为了评估智能体作为协作伙伴的表现，我们衡量了最终仓库的正确性以及在交互过程中所需的纠正性反馈轮数。对前沿编码智能体的实验表明，更强的智能体通常能达到更高的最终成功率，同时需要更少的干预，这表明用户体验得到了改善。

## Abstract
Most coding-agent benchmarks are static: an agent receives a complete task description up front and is judged only by its final code. Real coding assistance is interactive, with users clarifying goals, adding constraints, and correcting mistakes over multiple turns. We introduce SWE-Together, a multi-turn benchmark reconstructed from real user-agent coding sessions. To make real interactions verifiable, we curate 109 repository-level tasks from 11,260 recorded sessions, selecting sessions with recoverable repository states, clear user goals, and observable outcomes. To replay these interactions across agents, we build a reactive LLM-based user simulator that preserves the original users' intents and provides feedback when the coding agent's progress requires it. To evaluate agents as collaborators, we measure both final repository correctness and the number of corrective feedback turns required during the interaction. Experiments with frontier coding agents show that stronger agents generally achieve higher final success rates while requiring fewer interventions, suggesting an improved user experience.

## 1. 论文的核心问题与整体含义

- **研究动机**：现有编码智能体基准（如 SWE-Bench 系列）大多采用**静态评估**——提前给出完整任务描述，只检查最终代码的正确性。然而真实场景中，用户会通过多轮对话逐步阐明需求、添加约束、纠正错误，智能体需要适应这种**交互式协作**。静态基准无法反映真实用户体验，且已趋饱和，难以区分前沿模型能力。
- **整体含义**：本文提出 **SWE-Together**，一个从真实用户–智能体编码会话中重建的**多轮交互基准**，同时评估最终代码正确性和**交互过程中所需的纠正性反馈量**，从而更贴近实际使用。

## 2. 论文提出的方法论

### 核心思想
- 从约 11,260 个真实录制会话中筛选出 **109 个可验证的仓库级任务**，每个任务包含初始用户请求、可恢复的仓库状态、以及从原始会话中提取的用户意图和反馈触发条件。
- 构建一个**基于 LLM 的状态条件用户模拟器**，在评估新智能体时，模拟器仅在满足触发条件时发送反馈（如纠正、澄清、新需求），保持原始用户意图，但反馈时机根据被评估智能体的轨迹动态调整。
- 评估采用**双重指标**：最终仓库状态正确性（通过联合确定性验证器和**智能体评分器**），以及**用户纠正量（User Correction）** 和**意图覆盖率（Intent Coverage）** 来刻画交互质量。

### 关键技术细节
- **会话到任务的构建管道**分为三步：
  1. 确定性规则过滤：要求会话包含多轮用户消息、具体代码修改、可识别仓库。
  2. 可行性筛选：LLM 判断任务是否可以在本地复现并生成可验证结果。
  3. 沙盒任务生成：在隔离沙盒中自动完成仓库克隆、环境构建、确定性验证脚本编写，并生成任务包（含原始会话、初始指令、用户模拟提示）。
- **用户模拟器**：基于 LLM，动作空间包括 no-op（默认）、question、redirect、new-requirement、check-external。决策依据当前智能体轨迹、历史记忆和原始会话锚定条件，避免固定重放或泛化模拟。
- **正确性评分**：使用**阶段性评分器**：第一阶段离线为每个任务生成固定加权评分标准（基于原始参考补丁和行为目标），第二阶段用该标准评估智能体最终仓库状态，返回 [0,1] 分数。避免单个测试的偏误。
- **用户纠正量（User Correction）**：对模拟器每条消息打多标签，加权计算显式纠正（权重 1）和隐式推动（权重 0.2），越低越好。
- **意图覆盖率**：衡量模拟器是否忠实地涵盖了原始用户意图且无越界，加权组合召回和精确度。

## 3. 实验设计

- **数据集/场景**：从 4 个上游公开数据集（DataClaw、Pi-staging、Hyperswitch、SWE-chat）共 11,260 条会话中筛选出 **109 个仓库级任务**。任务覆盖不同代码库和编程场景。
- **基准方法**：使用开源 agent 框架 `opencode`，评估 7 个前沿模型：Claude Opus 4.8、GPT-5.5、Claude Opus 4.6、GLM-5.2、GLM-5.1、DeepSeek-V4-Pro、MiniMax-2.7。每个任务运行 2 次重复（k=2）。另外设置**参考补丁基线**（Oracle）作为对照。

## 7. 优点

- **基于真实交互**：任务来源于真实编码会话，更贴近实际使用场景。
- **可重复多轮评估**：通过状态条件用户模拟器实现了可复现的多轮交互，避免固定重放和泛化问题。
- **双重评估框架**：不仅评估最终代码正确性，还量化交互成本（用户纠正量），提供更全面的能力视图。
- **评分器设计稳健**：结合确定性执行和智能体评分器，生成行为性评分标准，避免单一测试偏差。
- **模拟器质量验证**：通过人类判断和意图覆盖率验证了模拟器的有效性和一致性。

## 8. 不足与局限

- **模拟器限制**：无法在 agent 回合内中断，不能直接编辑文件，仅依赖文本输出，缺乏视觉信息。适用于目标清晰的任务，对模糊、开放任务覆盖有限。
- **任务选择偏差**：从 11,260 会话中仅选出 109 个（0.97%），虽然高精度但可能损失多样性，且过滤条件偏向有明确可测量结果的补丁型任务。
- **重复次数少**：k=2 的重复次数可能不足以可靠评估方差较大的模型，pass 2 严格惩罚不一致性，但单次随机失败可能过度影响结论。
- **成本与透明度**：未报告具体算力消耗，且模型均为闭源 API，缺乏对推理细节的控制。
- **参考基线分数小于 100%**：部分原因（如过程要求、提取噪声）可能轻微影响绝对数值解释，但作者声明不影响相对比较。
- **可推广性**：当前基准主要针对“代码补丁提交”类任务，对于其他交互形式（如提问、探索、部署）的评估能力有限。
