## 摘要
大型语言模型（LLM）现已能够支持自动化的软件安全任务，包括漏洞发现和概念验证（PoC）生成。现有基准测试未能忠实评估LLM在真实漏洞挖掘场景中的表现，因为它们依赖模糊测试工具、针对特定目标的描述或漏洞复现任务。我们提出了SEC-bench Pro，这是一个用于衡量智能体在关键、高复杂性软件系统上进行漏洞挖掘能力的基准测试。这项工作通过一个三阶段流程（漏洞收集、环境重建和基于预言的验证）公开了包含具体PoC输入的报告，并将修复链接转化为可复现的任务。我们利用V8和SpiderMonkey中的183个已验证漏洞实例化了SEC-bench Pro，其中包括一个在Google漏洞奖励计划中累计奖励超过150万美元的V8子集。这些实例涵盖了浏览器级和运行时级执行条件下的内存安全、沙箱、JIT和竞态条件漏洞。评估显示，使用前沿模型的编码智能体在两个评估引擎上的成功率均低于40%。开源权重模型Kimi-K2.6基线在V8上达到11.7%，而最强的先进配置在V8上达到32.0%，在SpiderMonkey上达到38.8%。ClaudeCode和Codex解决了互补的实例集，其双智能体联合在V8上达到37.9%，在SpiderMonkey上达到48.8%。SEC-bench Pro为评估基于LLM的安全智能体提供了稳健的环境，并揭示了其在长周期漏洞挖掘任务中的局限性。

## Abstract
Large language models (LLMs) now support automated software security tasks, including vulnerability discovery and proof-of-concept (PoC) generation. Existing benchmarks do not faithfully evaluate LLMs in real-world bug hunting scenarios because they rely on fuzzing harnesses, target-specific descriptions, or vulnerability-reproduction tasks. We present SEC-bench Pro, a benchmark for measuring agent bug hunting on critical, high-complexity software systems. This work discloses reports with concrete PoC inputs and links fixes into reproducible tasks through a three-phase pipeline for vulnerability collection, environment reconstruction, and oracle-based validation. We instantiate SEC-bench Pro with 183 validated vulnerabilities across V8 and SpiderMonkey, including a V8 subset with more than $1.5 million in cumulative Google Vulnerability Reward Program awards. These instances span memory-safety, sandbox, JIT, and race-condition bugs under browser-grade and runtime-grade execution conditions. Our evaluation shows that coding agents with frontier models remain below 40% success on both evaluated engines. The open-weight Kimi-K2.6 baseline reaches 11.7% on V8, while the strongest frontier configuration reaches 32.0% on V8 and 38.8% on SpiderMonkey. ClaudeCode and Codex solve complementary instance sets, and their two-agent union reaches 37.9% on V8 and 48.8% on SpiderMonkey. SEC-bench Pro provides robust environments for assessing LLM-based security agents and exposes limitations in long-horizon bug hunting tasks.

### 2. 论文提出的方法论

SEC-bench Pro的核心思想是构建一个自演化、可复现的漏洞挖掘基准，其关键技术流程为三阶段流水线。

- **阶段一：报告收集**
    - 从项目特定的漏洞追踪器（如Chromium Issue Tracker、Mozilla Bugzilla）和公告源收集安全报告、PoC输入和关联的修复补丁。
    - 筛选标准：报告必须包含具体的PoC和对应的修复链接。
    - 对183个实例进行实例化，涵盖V8（103个）和SpiderMonkey（80个）的高价值漏洞。

- **阶段二：智能体驱动的环境重建**
    - 使用编码智能体在Docker沙箱中自动重建历史漏洞环境。
    - 智能体根据原始漏洞报告和PoC，自行驱动构建系统，配置消毒器，编译有漏洞的二进制文件，并验证PoC能否复现预期崩溃。
    - 输出标准化的工件包，包含有漏洞和已修复版本的Dockerfile、元数据、PoC和崩溃签名。

- **阶段三：自动化验证与评估**
    - **构建预言机验证**：实例需通过两个验证才能进入数据集。
        1.  **有漏洞镜像预言机**：在重建的有漏洞环境中多次运行PoC，必须产生与预期错误类型匹配的崩溃信号。
        2.  **已修复镜像预言机**：在应用补丁后的环境中运行同一PoC，崩溃必须被成功阻止。
    - **评估与评分**：将被评估智能体生成的候选PoC，在**有漏洞、已修复、最新**三个版本的Docker镜像中执行。
    - **LLM法官**：由推理型LLM分析三个镜像的执行证据（退出码、stdout、stderr），将结果分为三类：
        - `verified`：PoC在有漏洞镜像中触发了目标类型漏洞，且修复版和最新版证据不矛盾。
        - `unsure`：证据不完整，需人工裁定。
        - `illegal`：未触发目标漏洞或触发了不相关崩溃。
    - 一个实例只要有至少一个候选PoC被判定为`verified`即算成功。

### 6. 论文的主要结论与发现

- **成功率有限**：当前所有前沿智能体的单一配置均无法在任一引擎上突破40%的成功率。最强单智能体Codex在V8上为32.0%，ClaudeCode在SpiderMonkey上为38.8%。（RQ1）
- **智能体能力互补**：ClaudeCode和Codex解决的实例集高度互补，二者联合成功率在V8上达37.9%，SpiderMonkey上达48.8%。（RQ1）
- **搜索策略差异**：Codex的策略是“少而精”，仅提交经本地验证的高置信度PoC，通过率高；ClaudeCode则是“广撒网”，提交大量探索性PoC，通过率低但覆盖面广。失败的运行倾向于消耗更多Token。（RQ2 & RQ3）
- **漏洞类型难度差异**：整数溢出/截断类漏洞相对最容易解决，而沙箱逃逸（V8）和JIT问题（SpiderMonkey）仍然是最难的类别。（RQ4）
- **严格评估的必要性**：如果仅采用简单的“有漏洞镜像崩溃”标准，会导致成功率被高估高达43.6%，证明了SEC-bench Pro三镜像+LLM法官评分流程的必要性。

### 7. 优点

- **真实性与高保真度**：基准基于真实世界的JS引擎漏洞、具体PoC和补丁，不依赖模糊测试工具，任务输入不泄露漏洞位置线索，高度模拟了真实的代码审计场景。
- **自动化与可扩展性**：提出的三阶段自演化流水线可自动化处理新公开的漏洞报告，无需为每个新项目重新设计基准。
- **严格的评估方法**：三镜像（漏洞版/修复版/最新版）验证与LLM法官相结合的评分机制，能有效区分针对目标漏洞的PoC和非相关崩溃，避免了简单评分器带来的成功率虚高。
- **深入的洞察**：实验不仅给出了总分，还深入分析了不同智能体的行为模式、失败原因和各漏洞类型的相对难度，为未来改进指明了方向。

### 8. 不足与局限

- **数据集规模与领域**：当前实例化仅限于V8和SpiderMonkey两个JS引擎，总计183个实例，是否适用于其他类型复杂软件仍有待验证。
- **依赖性**：基准的构建依赖于漏洞报告中公开提供的高质量PoC和修复链接，这可能排除了只有描述而无PoC的安全问题。
- **LLM法官的可靠性**：虽然LLM法官弥补了简单评分的不足，但其本身可能存在误判，论文虽有人工裁定等保障措施，但其客观性并非绝对。
- **评估范围**：评估仅限于“从发现到生成PoC”，并未涵盖完整的漏洞利用（Exploit）开发过程。
- **算力成本**：实验揭示了成功运行智能体需要高昂的API调用费用，开源模型的性能远落后于闭源模型，这可能限制未来研究的可复现性。
